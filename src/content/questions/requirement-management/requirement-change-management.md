# 需求变更管理：如何在开发中途应对变更请求

## 面试场景

> 面试官说："你的产品已经进入开发阶段，预计两周后上线。这时销售团队带来一个大客户的需求，要求增加一个关键功能，否则客户就要流失。开发团队已经满负荷，你会怎么处理？"

这是 PM 面试中的经典题，考察的不是"加不加需求"，而是**变更管理的系统性思维**。

---

## 一、核心概念：需求变更 vs 范围蔓延

### 需求变更（Requirement Change）
有明确来源、有明确请求人、有明确业务理由的需求修改。是**正常的、可管理的**。

### 范围蔓延（Scope Creep）
无明确来源、无控制流程、逐渐累积的小改动。是**危险的、应避免的**。

**PM 的核心职责不是拒绝所有变更，而是建立一个让变更可控的机制。**

---

## 二、变更管理框架：CCB 四步法

### Step 1: Capture — 记录变更请求

任何变更必须有**书面记录**，不能是口头说一下。

变更请求模板：
```
【变更请求单】
- 请求人：xxx
- 请求日期：2024-xx-xx
- 变更类型：新增功能 / 修改功能 / 删除功能 / 技术架构变更
- 变更描述：xxx
- 业务理由：xxx（为什么必须改？不改会怎样？）
- 期望上线时间：xxx
- 优先级判断：P0/P1/P2
```

### Step 2: Constrain — 评估变更影响

变更评估需要覆盖五个维度：

| 维度 | 评估要点 | 负责人 |
|------|---------|--------|
| **进度影响** | 是否影响原定上线日期？延迟多少？ | PM + Tech Lead |
| **资源影响** | 需要多少额外人天？是否需要抽调其他人？ | Tech Lead |
| **质量影响** | 是否增加测试复杂度？是否有回归风险？ | QA Lead |
| **范围影响** | 是否需要砍掉其他需求来腾空间？ | PM |
| **业务影响** | 不做的损失 vs 做了收益 | PM + 业务方 |

### Step 3: Balanced Decision — 权衡决策

决策矩阵：

```
                    业务价值高           业务价值低
              ┌──────────────────┬──────────────────┐
  实现成本低  │  ✅ 纳入本轮      │  ⏸ 放入需求池     │
              ├──────────────────┼──────────────────┤
  实现成本高  │  ⚖ 需高层决策    │  ❌ 婉拒并说明    │
              └──────────────────┴──────────────────┘
```

对于面试题中的"大客户需求"场景：
- **业务价值**：客户流失成本 vs 此功能对其他客户的价值
- **实现成本**：是否真的需要两周？能否简化方案？
- **替代方案**：能否用临时方案（人工处理、配置化）先撑住？

### Step 4: Communication — 透明沟通

决策结果需要同步给所有干系人：

- **对请求方**：说明决策理由，如果是婉拒要给出替代方案
- **对开发团队**：说明变更原因，调整后的排期
- **对管理层**：重大变更需要升级决策

## 三、实战案例：游戏内购支付方式变更

### 背景
某游戏计划 6 月上线，5 月初渠道团队提出：某新兴支付方式用户增长快，必须接入。

### 变更评估
- **进度影响**：接入 + 联调需要 5 人天，不影响上线日期（还有缓冲）
- **资源影响**：需要支付后端 1 人支持 3 天
- **质量影响**：增加一种支付方式的测试路径
- **业务影响**：该支付方式用户占比 8%，且增长趋势明显

### 决策
纳入本轮，但条件：
1. 只接入基础支付流程，分期付款等高级功能放二期
2. 原定的"皮肤分享"功能延后到版本 1.1

### 结果
上线顺利，该支付方式首月贡献 6% 付费用户。

---

## 四、变更管理的三条红线

### 红线 1：上线前一周冻结需求
- 除非是 P0 级 Blocker，否则一律放入下一版本
- 冻结期内的变更请求需要产品总监审批

### 红线 2：不做无记录的变更
- 所有变更必须走变更请求单
- 口头答应 = 没答应

### 红线 3：变更必须有 trade-off
- 加了什么就必须减什么
- "都做"不是变更管理，是讨好

## 五、面试常见追问

### 追问 1："如果老板直接拍板要加呢？"

回答框架：
1. 先理解老板的业务考量（不要上来就反对）
2. 呈现影响评估数据（延迟 X 天 / 砍掉 Y 功能）
3. 提出替代方案（A：加但延期 / B：加但砍功能 / C：下版本加）
4. 让老板在知情的前提下做选择

> "好的老板，我理解这个需求的重要性。我评估了一下，如果加入这个功能，上线需要延期 3 天，或者我们可以把 XXX 功能移到下个版本。您看哪个方案更合适？"

### 追问 2："开发说来不及怎么办？"

回答要点：
- 先确认是"真的来不及"还是"不想加班"
- 如果是排期问题：砍低优先级需求腾空间
- 如果是技术难点：寻求技术方案简化或降级
- 如果是态度问题：拉 Tech Lead 一起评估，用数据说话

### 追问 3："频繁变更导致团队士气低落怎么办？"

回答要点：
- 承认问题，推动建立变更控制流程
- 和团队同步变更的业务背景，让开发理解"为什么"
- 保护核心开发时间（如设定每周二、四不接受新变更）
- 适时 push back 不合理的需求方

---

## 六、标准回答框架

```
1. 接到变更请求 → 不急着答应或拒绝
2. 记录请求 → 填写变更请求单
3. 评估影响 → 五维度（进度/资源/质量/范围/业务）
4. 权衡决策 → 用决策矩阵 + trade-off 思维
5. 透明沟通 → 所有干系人同步决策和理由
6. 更新计划 → 调整排期、需求文档、测试范围
```

---

## 七、关键收获

1. **变更管理的本质是决策管理**——不是不让变，而是让每一次变都有据可查、有人负责、有 trade-off
2. **流程的价值在于保护所有人**——PM 不被需求方裹挟，开发不被 PM 随意改需求，需求方理解变更的成本
3. **"好的 PM 不是什么需求都接，也不是什么需求都拒，而是让每个决策经得起追溯"**
4. **对于技术转 PM 的人**：你有天然优势理解"变更的技术成本"，但要补的能力是"变更的业务价值判断"——不能只因为实现成本低就接，也不能只因为实现成本高就拒
