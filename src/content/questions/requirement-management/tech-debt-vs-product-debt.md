# 如何在版本规划中平衡技术债与产品需求

## 面试场景

> 面试官问："你负责一款产品已经迭代了两年，技术团队反映代码质量很差、架构扛不住了，要求花两个版本专门还技术债。但业务方要求继续做新功能，因为竞品在快速追赶。你怎么处理？"

这道题是高级PM的必考题，考察的是你能否在"长期技术健康"和"短期业务价值"之间做出理性权衡。这不是一个能取悦所有人的问题，优秀的PM需要有清晰的决策框架和推进行动的能力。

---

## 一、核心概念：理解技术债的本质

### 定义

技术债（Technical Debt）：为了快速交付短期价值，在代码质量、架构设计上做出的妥协。就像金融债务一样，它有"本金"（欠下的技术工作）和"利息"（每次迭代因为代码差而额外消耗的时间）。

```
实际开发速度 = 理论速度 - 技术债利息
当技术债积累到一定程度，实际开发速度 → 0
```

### 技术债的分类

| 类型 | 产生原因 | 紧急度 | 典型表现 |
|------|---------|--------|---------|
| 鲁莽型（Reckless） | 不在乎质量，赶工堆代码 | 高 | 没有测试、复制粘贴遍地 |
| 谨慎型（Prudent） | "我们知道有更好的方案但当时没时间" | 中 | 架构选择在当前规模下不再合适 |
| 无意的（Accidental） | 当时不知道更好的做法 | 低 | 代码可以work但不够优雅 |
| 不可避免的 | 业务变化快于架构演进 | 中 | 原本单体架构需要拆微服务 |

**关键区分**：不是所有技术债都需要立即还。就像金融债务一样，只要"利息"可控，维持一定的技术债是合理的工程决策。

### 技术债与产品债的关系

很多PM只关注技术债，忽视了一个对等概念——**产品债（Product Debt）**：

```
技术债：代码和架构层面的妥协
  → 后果：开发变慢、bug增多、性能下降

产品债：产品设计和用户体验层面的妥协
  → 后果：用户体验差、留存下降、品牌损伤
  → 例子：临时拼凑的功能、不一致的交互模式、缺失的边缘case处理
```

**优秀PM的视角**：技术债和产品债是一体两面。技术债导致开发变慢，最终体现为产品债无法及时修复；产品债的积累又催生更多临时方案，加剧技术债。

## 二、技术债评估框架

### 框架一：利息矩阵

将技术债按"利息大小"和"本金大小"放入矩阵：

```
          利息大（影响大）        利息小（影响小）
         ┌─────────────────┬─────────────────┐
本金大   │  ⚡ 紧急且重要    │   📋 有计划地还   │
(重构大) │  → 分期偿还       │   → 放入版本池    │
         ├─────────────────┼─────────────────┤
本金小   │  🔥 优先偿还      │   😌 可以接受     │
(改动小) │  → 立即安排       │   → 暂时忽略      │
         └─────────────────┴─────────────────┘
```

### 框架二：量化技术债的影响

不要凭感觉说"代码很差"，要用数据：

| 指标 | 健康范围 | 技术债信号 | 数据来源 |
|------|---------|-----------|---------|
| 开发效率（故事点/人周） | 稳定或上升 | 连续3个迭代下降 | JIRA |
| Bug率（每千行代码） | 行业平均 | 持续上升 | Bug追踪系统 |
| 部署频率 | 稳定 | 越来越低/越来越怕发布 | CI/CD |
| 平均修复时间(MTTR) | <4小时 | 持续上升 | 运维数据 |
| 代码覆盖率 | >70% | <40%且下降 | 测试报告 |
| 测试环境搭建时间 | <30分钟 | >半天 | 工程效能 |

### 框架三：技术债的业务影响翻译

**PM的核心能力之一：把技术语言翻译成业务语言。**

| 技术说法 | 业务翻译 | 对决策的影响 |
|---------|---------|------------|
| "数据库需要分库分表" | "再不分库，大促时系统会挂，预计损失XXX万" | 决策者能理解 |
| "代码耦合度太高" | "改一个功能平均会引发2.3个新bug" | 直观 |
| "需要升级框架版本" | "旧版本明年停止安全更新，有合规风险" | 有紧迫感 |
| "测试覆盖率太低" | "每轮回归测试需要3天手动验证，拖慢发版" | 量化影响 |

## 三、平衡决策的四个步骤

### Step 1：盘点技术债清单

联合技术负责人，列出所有技术债项：

```markdown
## 技术债清单示例

| 编号 | 描述 | 类型 | 本金(人天) | 利息(每迭代多耗人天) | 业务风险 |
|------|------|------|-----------|-------------------|---------|
| TD-01 | 数据库单点未做主从 | 基础设施 | 5 | 0（但宕机风险极高） | 🔴 高 |
| TD-02 | 支付模块无单元测试 | 质量 | 8 | 2 | 🟡 中 |
| TD-03 | 前端组件未组件化 | 架构 | 15 | 3 | 🟡 中 |
| TD-04 | 日志系统杂乱 | 运维 | 3 | 1 | 🟢 低 |
| TD-05 | 老版本API未清理 | 维护 | 2 | 0.5 | 🟢 低 |
```

### Step 2：计算ROI排序

```
技术债ROI = (减少的利息/迭代 × 预计收益迭代数) / 本金

示例：
TD-02 支付模块加测试：
  ROI = (2人天/迭代 × 12迭代) / 8人天 = 3.0  → 高ROI

TD-03 前端组件化：
  ROI = (3人天/迭代 × 12迭代) / 15人天 = 2.4  → 较高ROI

TD-04 日志系统：
  ROI = (1人天/迭代 × 12迭代) / 3人天 = 4.0  → 最高ROI
```

**注意**：ROI不是唯一标准。基础设施类技术债（如TD-01）虽然ROI难量化，但风险极高，必须优先处理。

### Step 3：资源分配比例规则

建立固定的版本资源分配比例：

```
推荐分配（成熟产品）：
├── 60% 新功能开发
├── 20% 技术债偿还
├── 10% Bug修复和优化
└── 10% 探索性工作

高技术债期分配（临时调整）：
├── 40% 新功能开发
├── 40% 技术债偿还
├── 15% Bug修复和优化
└── 5% 探索性工作
```

**关键原则**：
- 比例是约定，不是铁律。但一旦约定，不要每次都挤占技术债的份额
- 可以"借债"（这个版本少还技术债，下个版本多还），但要"有借有还"
- 由PM和技术负责人共同商定，不是单方面决定

### Step 4：将技术债嵌入版本规划

**不要单独开"技术债版本"**——这会让业务方很难接受。

**推荐做法：每个版本混合包含：**

```
版本 v3.2 规划示例：

🎯 业务目标：提升支付转化率（北极星：支付成功率+5%）

需求列表：
├── [功能] 新增支付方式（8人天）          ← 业务需求
├── [功能] 支付失败重试优化（3人天）       ← 业务需求  
├── [技术债] 支付模块单元测试补全（5人天）  ← 技术债（与本版本功能相关）
├── [技术债] 数据库主从配置（5人天）        ← 技术债（高优先级）
├── [优化] 支付页面加载速度优化（3人天）    ← 体验优化
└── [Bug] iOS支付回调失败修复（2人天）      ← Bug修复

总计：26人天，技术债占比约 38%
```

**技巧**：将技术债与同版本的业务功能做关联。"你想要支付转化率提升，我们必须先补测试，否则新功能的bug会吃掉转化率提升。"

## 四、Stakeholder沟通策略

### 对业务方（老板/运营/销售）

**不要说**："技术团队需要两个版本还技术债。"
**应该说**："根据数据，开发效率过去6个月下降了35%。如果不处理，下季度的3个核心功能有2个会延期。我的方案是每版本投入30%资源做基础优化，确保下季度功能按时交付。"

### 对技术团队

**不要说**："业务方不同意还技术债，你们忍忍。"
**应该说**："我理解技术债的严重性。我的方案是每版本固定分配30%给技术债，优先处理ROI最高的3项。虽然不能一次性解决，但确保每个版本都有进步。同时我需要你们把技术债的影响量化，帮我去跟业务方争取更多资源。"

### 对高管

**用投资回报语言**：
```
"这笔技术债偿还的投入，相当于给开发引擎做保养。
 不保养：每年开发效率衰减30%，相当于损失X个工程师的产能。
 保养后：恢复到健康效率，新功能交付速度提升40%。"
```

## 五、面试常见追问

### 追问1："如果技术团队说'不还债就不开发了'，你怎么办？"

**分三步处理：**
1. **紧急调解**：私下和技术leader深谈，理解最痛的点是什么（往往1-2个核心问题，不是全部）
2. **快速见效**：从技术债清单中挑出"本金小、利息大"的项，在当前版本立即安排，让技术团队看到诚意
3. **长期机制**：建立技术债常态化机制（每版本固定比例），避免积攒到崩溃

### 追问2："怎么判断技术团队是不是在夸大技术债？"

**信号判断：**
- 能量化影响（效率下降数据、bug率上升）→ 通常不是夸大
- 只会说"代码很烂"但说不清具体影响 → 可能是主观感受
- 有明确的技术债清单和优先级 → 专业且可信
- 每次需求评审都说"这个做不了因为架构问题" → 需要深入了解

**验证方法**：要求技术团队提供量化数据（见框架二），用数据而非感觉做决策。

### 追问3："技术转PM的人在这道题上有什么优势和劣势？"

**优势**：能理解技术债的真实严重性，不容易被技术团队"忽悠"，能判断哪些技术债是紧急的哪些可以等
**劣势**：容易过度共情技术团队，把技术债优先级排得过高，忽视业务紧迫性。记住——你的角色是PM，不是技术架构师

### 追问4："如果业务方坚持100%资源做新功能，你怎么推技术债？"

- **数据说服**：用效率衰减数据证明，不做技术债会导致未来功能延期
- **风险警示**：列出可能的系统性风险（如宕机、数据丢失），量化业务影响
- **小步嵌入**：不单独要资源，而是把技术债嵌入功能需求中（"做这个功能必须先重构那块代码"）
- **升级机制**：如果以上都无效，将问题升级到技术VP和业务VP共同决策

## 六、标准回答框架

1. **重新定义问题**（30秒）："这不是技术vs业务的对立，而是短期速度和长期速度的投资分配问题。"

2. **量化现状**（1分钟）："我会先用数据评估技术债的严重程度——开发效率趋势、bug率、部署频率等。"

3. **给出方案**（2分钟）："我的方案是：不设独立的技术债版本，而是每版本固定30%资源给技术债，按ROI排序偿还。同时把技术债的业务影响翻译成业务方能理解的语言。"

4. **处理冲突**（1分钟）："如果业务方完全不同意，我会用'小步嵌入'策略——把必要的技术优化和新功能绑定，让技术债偿还成为功能交付的前提。"

5. **建立机制**（1分钟）："长期来看，我会建立技术债可视化看板，每版本review技术债指标，确保不再积累新的高息债。"

## 七、关键收获

1. **技术债不是技术问题，是资源分配问题**——PM的核心职责是在多方需求中做理性的tradeoff
2. **量化是一切的基础**——没有数据，技术债讨论就变成了"技术说很严重vs业务说没那么急"的主观争论
3. **不要一次性还完，要持续还**——每版本固定比例比"集中还债版本"更可持续
4. **把技术债翻译成业务影响**——这是PM不可替代的价值，纯技术讨论中PM插不上话，纯业务讨论中技术债会被忽视
5. **产品债同样重要**——在关注代码质量的同时，不要忽视产品体验的一致性和完整性
