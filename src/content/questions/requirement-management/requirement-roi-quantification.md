# 需求ROI评估与量化决策

## 面试场景

> 面试官：你手上有20个来自不同部门的需求，但本季度研发资源只够做8个。除了用优先级框架排序之外，你会如何量化评估每个需求的ROI，来支撑你的资源分配决策？

## 核心概念解析

### 为什么需求ROI评估很难

需求ROI（Return on Investment）评估是PM的核心能力之一，但在实践中面临三大挑战：

1. **收益难以量化**：用户体验优化、技术债清理等需求的直接收益不易衡量
2. **时间维度复杂**：短期收益vs长期价值之间的权衡缺乏统一标准
3. **跨需求比较困难**：一个增长类需求和一个留存类需求，如何放在同一把尺子上量

### ROI的本质公式

```
需求ROI = (预期收益 - 实施成本) / 实施成本 × 100%

其中：
- 预期收益 = 业务价值 × 实现概率 × 时间系数
- 实施成本 = 开发成本 + 机会成本 + 维护成本
```

## 评估框架：RICE进阶版

### 基础RICE模型回顾

RICE = (Reach × Impact × Confidence) / Effort

这是最常用的需求评分模型，但它有明显的局限性。

### 进阶版：加权多维ROI评估

#### 维度一：收益量化（Benefit）

| 收益类型 | 量化方式 | 数据来源 |
|---------|---------|---------|
| 直接收入 | 预计带来的营收增长 | 市场调研、历史数据 |
| 转化提升 | 转化率提升×流量×客单价 | 漏斗分析、A/B测试预估 |
| 留存改善 | 留存率提升×LTV | 群组分析 |
| 效率提升 | 节省人力成本/时间 | 流程审计 |
| 风险规避 | 合规风险、 churn风险量化 | 风险评估模型 |
| 战略价值 | 对北极星指标的贡献度 | 战略对齐评估 |

#### 维度二：成本评估（Cost）

| 成本类型 | 评估方式 | 说明 |
|---------|---------|------|
| 开发成本 | 人天估算×人均日成本 | 需要研发参与评估 |
| 设计成本 | 设计师人天 | 常被忽略 |
| 测试成本 | QA人天+测试环境 | 含自动化测试投入 |
| 机会成本 | 同期无法做的其他需求 | 最容易被忽视 |
| 维护成本 | 上线后的持续投入 | 按月估算 |
| 风险成本 | 上线失败的概率×损失 | 需要风险量化 |

#### 维度三：不确定性折扣（Confidence）

- **高信心（80-100%）**：有历史数据支撑、有A/B测试验证、有竞品对标
- **中信心（50-80%）**：有用户调研支持、有逻辑推导、有部分数据
- **低信心（20-50%）**：纯直觉判断、无数据支撑、市场变化大

```
实际ROI = 理论ROI × Confidence系数
```

## 实战案例：电商平台需求排序

### 场景

某电商平台Q3需求池，资源仅够做8个需求：

| 编号 | 需求 | 提出方 | 预估人天 |
|-----|------|-------|---------|
| A | 商品详情页视频展示 | 运营 | 15 |
| B | 搜索结果个性化排序 | 增长 | 25 |
| C | 会员积分体系重构 | 用户 | 40 |
| D | 评论区图片上传 | 社区 | 8 |
| E | 跨境支付通道接入 | 国际化 | 30 |
| F | 后台商品批量导入 | 商家运营 | 12 |
| G | 推荐算法升级 | 增长 | 35 |
| H | 客服工单系统优化 | 客服 | 10 |

### ROI评估过程

以需求B「搜索结果个性化排序」为例：

**收益量化：**
- Reach：日均搜索用户50万 × 搜索转化率提升预估3% = 1.5万额外转化
- Impact：每转化订单价值45元 → 日增收67.5万元
- 月度收益：67.5万 × 30 = 2025万
- 季度收益（考虑爬坡期）：约4000万

**成本评估：**
- 开发成本：25人天 × 2000元/人天 = 5万元
- 设计成本：3人天 × 1500元/人天 = 0.45万元
- 机会成本：这25人天无法做需求D+F（预估收益500万）
- 维护成本：每月2人天

**Confidence：**
- 有竞品数据（某东已上线类似功能，转化提升2.8%）
- 有A/B测试灰度数据验证方向
- Confidence = 75%

**最终ROI计算：**
```
理论季度ROI = (4000万 - 5.45万) / 5.45万 ≈ 733倍
实际ROI = 733 × 75% ≈ 550倍
考虑机会成本后 = (4000万 - 500万) / (5.45万 + 500万分摊) ≈ 6.9倍
```

### 排序结果

评估完所有需求后，按"考虑机会成本后的实际ROI"排序：

1. B - 搜索个性化排序（ROI ~6.9x）
2. G - 推荐算法升级（ROI ~4.2x）
3. D - 评论区图片上传（ROI ~3.8x，低成本高收益）
4. E - 跨境支付接入（ROI ~3.1x，战略价值加分）
5. F - 后台批量导入（ROI ~2.9x）
6. A - 商品视频展示（ROI ~2.1x）
7. H - 客服系统优化（ROI ~1.5x，效率型）
8. C - 会员积分重构（ROI ~0.8x，成本太高）

## 面试常见追问

### 追问1：对于无法量化的需求怎么办？

**回答框架：**
- 第一步：找到代理指标（Proxy Metric）。如品牌认知度无法直接量化，可用搜索指数、社媒提及量代替
- 第二步：使用类比法。参考同类功能上线后的实际效果
- 第三步：设定最低门槛。即使无法精确量化，也要给出区间估计（如"预期提升5%-15%"）
- 第四步：如果实在无法量化，用"战略必须性"单独标记，走特殊审批通道

### 追问2：需求ROI评估的常见陷阱有哪些？

**五大陷阱：**

1. **沉没成本谬误**：已经投入研发的需求不愿砍掉，即使ROI已经变差
2. **乐观偏差**：PM倾向于高估收益、低估成本。解法：引入"现实系数"，历史平均高估比例的倒数
3. **短期偏好**：优先选择短期见效的需求，忽视长期战略投入
4. **可量化偏好**：只做能量化的需求，忽略难以量化但重要的需求
5. **忽略二阶效应**：一个需求可能触发连锁反应（如新功能上线导致老功能使用下降）

### 追问3：如何让各 stakeholder 认可你的ROI评估？

**策略：**
- 让提出方参与收益预估："你认为这个需求能带来多少提升？"——让他们自己给数字
- 用历史数据校准："上次类似的需求，实际效果是预估的60%"
- 建立需求复盘机制：上线后追踪实际ROI vs 预估ROI，持续校准
- 对于争议需求，提出MVP验证方案：用最小成本验证假设

## 标准回答框架

```
1. 开场表态（30秒）
   → "需求ROI评估是我做需求排序的核心工具，但不是唯一工具"

2. 量化框架展示（2分钟）
   → 多维收益评估 + 全成本核算 + 不确定性折扣
   → 强调机会成本的重要性

3. 案例说明（2分钟）
   → 准备一个自己做过的真实案例
   → 展示从需求到ROI数字的完整推导过程

4. 局限性认知（1分钟）
   → 承认ROI不是万能的
   → 说明如何处理不可量化但重要的需求

5. 闭环机制（1分钟）
   → 上线后的实际ROI追踪
   → 用复盘数据持续校准评估模型
```

## 关键收获

1. **ROI量化不是精确科学，而是决策辅助**：目的是让讨论从"我觉得这个重要"变成"让我们看看数字"
2. **全成本核算是PM的必修课**：尤其是机会成本，它区分了"做了一件好事"和"做了最好的事"
3. **Confidence折扣防止过度乐观**：每个ROI数字都应该有一个信心系数
4. **复盘是校准的唯一方式**：不追踪实际效果的ROI评估永远只是纸上谈兵
5. **ROI评估本身也有ROI**：花2天评估20个需求，避免3个月浪费在错误需求上，这个投入完全值得
