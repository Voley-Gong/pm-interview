# 需求来源管理与权重分配

## 面试场景

> 面试官：你负责一款B端SaaS产品，每周收到来自销售、客服、老板、竞品分析、用户反馈等十几个渠道的需求。销售说"不签这个单就丢了"，老板说"这个必须有"，客服说"用户天天投诉"。你的需求池有80条需求，开发只能做10条。你如何管理这些需求来源，如何分配权重？

这是一道高频PM面试题，考察的不是某个具体框架，而是**需求治理的系统思维**。

---

## 一、为什么需求来源管理是个真问题

很多PM的需求池像一个"垃圾桶"——什么都往里扔，最后靠感觉捞。常见痛点：

| 痛点 | 表现 | 后果 |
|------|------|------|
| 来源单一依赖 | 80%需求来自老板或销售 | 产品变成"定制项目" |
| 声大优先 | 谁嗓门大谁的需求先做 | 真正重要的需求被淹没 |
| 无来源标记 | 需求池里只有标题，没有来源 | 无法追溯、无法评估价值 |
| 权重模糊 | 所有需求"都很重要" | 排优先级变成政治博弈 |
| 渠道偏见 | 只听用户反馈，忽略数据 | 需求方向系统性偏差 |

**核心洞察：需求的可信度和优先级，在很大程度上取决于它的来源和质量。**

## 二、需求来源的六大典型渠道

### 2.1 渠道全景图

```
┌─────────────────────────────────────────────────┐
│               需求来源生态                         │
├──────────┬──────────┬──────────┬────────────────┤
│  用户侧   │  业务侧   │  竞品侧   │   内部侧        │
├──────────┼──────────┼──────────┼────────────────┤
│ 用户反馈  │ 销售线索  │ 竞品监控  │  老板/高管     │
│ 客服工单  │ 客户成功  │ 行业报告  │  团队创意      │
│ NPS调研   │ 市场需求  │ 分析师   │  技术驱动      │
│ 可用性测试 │ 合规要求  │ 标杆对标  │  数据洞察      │
└──────────┴──────────┴──────────┴────────────────┘
```

### 2.2 各渠道特征对比

| 渠道 | 典型可信度 | 数量 | 转化率 | 常见偏差 |
|------|-----------|------|--------|---------|
| 用户直接反馈 | 高 | 中 | 高 | 幸存者偏差（只有活跃用户才反馈） |
| 客服工单 | 中高 | 高 | 中 | 只反映"出了什么问题"，不反映"想要什么" |
| 销售线索 | 中 | 高 | 低 | 单客户导向，容易变成定制需求 |
| 竞品分析 | 中 | 低 | 中 | "别人有我也要有"的陷阱 |
| 数据洞察 | 高 | 低 | 高 | 需要强的分析能力才能产出需求 |
| 老板/高管 | 视情况 | 低 | 变异大 | 可能是战略洞察，也可能是一时兴起 |
| 团队内部 | 中 | 中 | 中 | 容易"自嗨"，离用户远 |

## 三、需求来源权重分配框架（SRWM模型）

### Source-Weight Matrix（来源-权重矩阵）

核心思路：给每个来源打分，分数由三个维度构成：

### 维度1：战略对齐度（Strategic Alignment）

这个来源的需求，和你当前的产品战略目标有多匹配？

- **高（3分）**：直接服务于本季度OKR/北极星指标
- **中（2分）**：和产品方向相关，但不是本季度核心
- **低（1分）**：游离于战略之外

### 维度2：用户覆盖面（User Coverage）

这个需求代表了多少用户？

- **高（3分）**：影响全部或大部分核心用户
- **中（2分）**：影响某个重要用户群体
- **低（1分）**：只影响个别用户/客户

### 维度3：验证强度（Evidence Strength）

这个需求的证据链有多强？

- **高（3分）**：有数据+定性+竞品三重验证
- **中（2分）**：有1-2个来源验证
- **低（1分）**：单一来源，无交叉验证

### 综合权重计算

```
来源权重 = (战略对齐 × W1) + (用户覆盖 × W2) + (验证强度 × W3)

推荐权重：W1=0.4, W2=0.3, W3=0.3
```

**实战示例：**

| 需求来源 | 战略对齐 | 用户覆盖 | 验证强度 | 综合权重 |
|---------|---------|---------|---------|---------|
| 数据洞察：注册转化漏斗在第三步流失60% | 3 | 3 | 3 | **3.0** |
| 客服工单：100+用户反馈某按钮找不到 | 2 | 3 | 2 | **2.3** |
| 销售线索：某大客户要求自定义报表 | 1 | 1 | 1 | **1.0** |
| 老板：觉得UI不够时尚 | 1 | 1 | 1 | **1.0** |
| 竞品分析：竞品新增AI推荐功能 | 2 | 2 | 1 | **1.7** |

## 四、需求来源治理的5个实操步骤

### 步骤1：建立需求录入规范

每一条进入需求池的需求，必须包含：

```yaml
来源渠道: [用户反馈/客服/销售/竞品/数据/老板/内部]
来源详情: "2024年Q3客户大会，3家KA客户共同提及"
提出人: 张三（销售总监）
关联用户数: 预估影响500+用户
验证状态: [未验证/部分验证/充分验证]
战略关联: 对应OKR中的"提升企业客户留存"
```

### 步骤2：定期来源审计

每月做一次需求来源分布审计：

- 如果销售来源占比 >50%：警惕产品变成定制工厂
- 如果老板来源占比 >30%：警惕产品变成执行工具
- 如果数据来源占比 <10%：你的产品决策缺乏数据支撑

### 步骤3：交叉验证机制

单个来源的需求不能直接进入开发排期，至少需要两个独立来源交叉验证：

| 场景 | 来源A | 来源B | 结论 |
|------|-------|-------|------|
| 用户反馈"搜索不好用" + 数据显示搜索跳出率高 | 定性 | 定量 | ✅ 高置信度 |
| 销售说"客户要" + 仅此一家 | 单一来源 | — | ❌ 待验证 |
| 竞品有 + 老板觉得好 | 竞品 | 直觉 | ⚠️ 需进一步验证 |

### 步骤4：来源去偏见

常见偏见及应对：

- **大客户偏见**：KA客户声音大，但不代表大多数。设"中小客户需求配额"
- **近期偏见**：最近收到的需求感觉更紧迫。用时间衰减函数校正
- **权威偏见**：老板的需求自动排前面。所有需求走同一评分流程

### 步骤5：需求来源反馈闭环

告诉提出人需求的处理结果，建立长期信任：

- **已排期**：告知预期上线时间
- **已拒绝**：说明原因，给出替代方案
- **需更多信息**：明确需要补充什么

## 五、实战案例

### 背景

某B端项目管理工具，需求池120条，每两周迭代可做8-10条。

### 问题

PM小李的需求分布：销售来源45%，老板25%，客服15%，数据5%，内部10%。

产品逐渐变成"大客户定制工具"，标准功能停滞。

### 解决方案

1. **设立来源配额**：每迭代周期，销售来源需求不超过30%
2. **建立数据驱动需求通道**：每周从数据看板提取1-2个洞察转化为需求
3. **引入用户咨询委员会**：从不同规模客户中选取代表，定期收集需求
4. **需求预审会**：所有需求在进入排期前，先过SRWM评分

### 结果（3个月后）

- 需求来源分布趋于均衡：销售30%，数据25%，客服20%，老板15%，内部10%
- 标准功能使用率提升40%
- 大客户满意度没有下降（因为需求被合理排期而非被拒绝）

## 六、面试常见追问

### 追问1："如果老板坚持要做一个你认为不该做的需求怎么办？"

**回答框架：**
1. 先理解背后的战略意图（老板看到的可能比你大）
2. 用数据说话：展示SRWM评分和用户覆盖面
3. 提供"最小可行验证"方案：用最低成本验证需求价值
4. 如果老板仍然坚持：执行，但设定明确的成功标准和退出条件

### 追问2："如何处理销售提出的'不加这个功能客户就不续费'的需求？"

**回答框架：**
1. 评估真实流失风险：是单个客户还是群体趋势？
2. 区分"必须有的底线功能"和"锦上添花的定制需求"
3. 如果是底线功能：紧急排期
4. 如果是定制需求：评估该客户的LTV，考虑替代方案（如插件、API开放）

### 追问3："你如何减少来自客服的重复低质量需求？"

**回答框架：**
1. 对客服工单做根因分析：很多"需求"其实是bug或交互问题
2. 建立工单→需求的转化标准：只有经过聚类的、有明确场景的才转需求
3. 给客服团队培训产品边界：什么是产品能做的，什么不能

## 七、标准回答框架

面试时按以下结构回答：

```
1. 框架引入（30秒）
   "我会用来源-权重矩阵来管理需求池，核心是三个维度：
   战略对齐、用户覆盖、验证强度。"

2. 来源分类（30秒）
   "首先，我把需求来源分为四类：用户侧、业务侧、竞品侧、内部侧，
   每个来源有不同的可信度和偏差模式。"

3. 权重分配（60秒）
   "每个来源的需求按SRWM模型打分，得分高的优先排期。
   特别注意避免三种偏见：大客户偏见、近期偏见、权威偏见。"

4. 治理机制（60秒）
   "除了评分，我还建立了三个治理机制：
   - 来源配额（防止某个渠道垄断）
   - 交叉验证（至少两个独立来源）
   - 反馈闭环（让提出人知道结果）"

5. 案例佐证（30秒）
   用一个简洁的真实案例说明你怎么做的。
```

## 八、关键收获

| 洞察 | 说明 |
|------|------|
| 需求来源 ≠ 需求优先级 | 来源是输入变量，优先级是输出结果 |
| 单一来源是风险信号 | 任何需求如果只有一个来源，都需要交叉验证 |
| 来源分布是产品健康指标 | 来源单一 = 产品方向脆弱 |
| 治理优于判断 | 系统化的来源管理比逐条判断更可持续 |
| 说"不"的能力取决于说"为什么" | 有理有据的拒绝比含糊接受更专业 |
