## 什么是需求生命周期管理

大多数 PM 面试题只考"如何写 PRD"或"如何排优先级"，但真正区分初级 PM 和高级 PM 的，是**你是否能管理一个需求从诞生到下线的完整旅程**。

面试官问这道题，想考察的是：
- 你是否有**全生命周期意识**（而不是只管写 PRD 丢给开发）
- 你能否在**需求演进过程中做动态决策**（而不是一成不变执行计划）
- 你是否理解**版本规划的节奏感**（而不是想到什么做什么）
- 你能否在**多方利益中推动需求落地**（而不是被 stakeholder 拖垮）

## 需求生命周期的六个阶段

```
Stage 1        Stage 2        Stage 3        Stage 4        Stage 5        Stage 6
 需求发现  →    需求定义  →    需求评审  →    开发交付  →    上线验证  →    迭代/下线
   🔍            📝             🤝             ⚙️             🚀             📊
 Discover       Define         Review         Deliver         Validate       Iterate/Retire
```

### Stage 1：需求发现（Discover）

**目标：** 找到真正值得做的事。

**输入来源：**
| 来源 | 信号强度 | 说明 |
|------|----------|------|
| 用户反馈/访谈 | ⭐⭐⭐⭐⭐ | 最直接的真实需求信号 |
| 数据分析 | ⭐⭐⭐⭐ | 漏斗断裂、功能使用率低 |
| 业务目标 | ⭐⭐⭐⭐ | 公司战略向下拆解 |
| 竞品分析 | ⭐⭐⭐ | 参考但不盲从 |
| 销售/客服反馈 | ⭐⭐⭐ | 一线声音，但可能有偏差 |
| 老板灵感 | ⭐⭐ | 需要验证，不能直接执行 |

**关键产出：** 需求池（Backlog），每条需求包含：
- 问题描述（不是解决方案）
- 用户场景
- 数据支撑
- 初步优先级评估

### Stage 2：需求定义（Define）

**目标：** 把模糊的想法变成清晰的方案。

**PRD 的核心要素（不是模板，是思考框架）：**

1. **背景与价值**：为什么要做？预期收益是什么？
2. **用户故事**：作为[谁]，我想要[做什么]，以便[达到什么目的]
3. **功能描述**：详细到开发能动手写代码
4. **边界条件**：什么情况不处理？异常流程怎么走？
5. **非功能需求**：性能、安全、兼容性、可访问性
6. **埋点方案**：上线后怎么衡量效果？
7. **风险评估**：可能出什么问题？应对方案是什么？

**初级 PM 和高级 PM 的区别：**

| 初级 PM 的 PRD | 高级 PM 的 PRD |
|----------------|----------------|
| 只写 Happy Path | 写全所有异常分支 |
| 不考虑非功能需求 | 性能/安全/兼容性都有考虑 |
| 埋点方案缺失或后补 | 埋点和功能同步设计 |
| 不评估风险 | 有风险矩阵和降级方案 |

### Stage 3：需求评审（Review）

**目标：** 让所有相关方达成共识，可以进入开发。

**参与者与关注点：**
| 角色 | 关注什么 | 常见阻力 |
|------|----------|----------|
| 开发 | 工作量评估、技术可行性 | "这个做不了"、"工期不够" |
| 设计 | 交互方案、视觉规范 | "需求太模糊，没法设计" |
| 测试 | 测试用例覆盖、验收标准 | "验收标准不清晰" |
| 运营 | 上线节奏、运营准备 | "功能不够完整，没法推" |
| 法务/合规 | 合规风险 | "这个数据收集不合规" |

**评审推进策略：**
1. **评审前**：提前 1-2 天发 PRD 给各方阅读，收集初步意见
2. **评审中**：聚焦分歧点，不在细节上纠缠（可以会后单独沟通）
3. **评审后**：输出会议纪要，明确：结论、待解决项、责任人、时间节点

**经典冲突场景：**

> 开发说："这个功能要 3 周。"
> 你说："但是下周就要上线。"

**高 PM 的处理方式：**
1. 先理解工作量评估的依据（不是讨价还价）
2. 一起拆解——哪些是必须的，哪些可以简化
3. 提出取舍方案："如果先不做 B 模块，A 能在 1 周内完成吗？"
4. 如果确实做不完，和业务方沟通调整预期

### Stage 4：开发交付（Deliver）

**目标：** 确保开发过程中需求不走样、不延期。

**PM 在开发阶段的三件事：**

1. **每日跟进**：站会/异步同步，了解进度和阻碍
2. **需求答疑**：开发过程中一定会有细节问题，及时回答
3. **变更管理**：如果需要改需求，走变更流程（不能偷偷改）

**变更管理的黄金法则：**
> 需求一旦进入开发，任何变更都必须：
> ① 说明原因
> ② 评估影响（工期、测试、其他依赖）
> ③ 由 PM + Tech Lead + 相关方共同确认
> ④ 更新 PRD 版本号

### Stage 5：上线验证（Validate）

**目标：** 确认上线后效果符合预期。

**验证清单：**
- [ ] 线上功能冒烟测试通过
- [ ] 核心埋点数据正常上报
- [ ] 灰度阶段数据监控无异常（崩溃率、性能指标）
- [ ] 用户反馈渠道畅通（客服/社区/社媒）
- [ ] 业务指标对照预期基线

**上线后的数据观察节奏：**
| 时间段 | 关注什么 |
|--------|----------|
| 上线后 1 小时 | 系统稳定性、是否有 P0 bug |
| 上线后 24 小时 | 核心指标初步趋势、用户反馈 |
| 上线后 3 天 | 短期指标对照预期 |
| 上线后 14 天 | 中期指标（留存、转化） |
| 上线后 30 天 | 长期效果评估、ROI |

### Stage 6：迭代或下线（Iterate / Retire）

**目标：** 基于数据决定下一步。

| 数据表现 | 决策 |
|----------|------|
| 超预期 | 加大投入、扩展场景 |
| 符合预期 | 持续优化、进入常规迭代 |
| 低于预期 | 分析原因：是需求问题还是实现问题？ |
| 完全无效 | 考虑下线或重构 |

**功能下线也是 PM 的工作：**

很多 PM 只管上线，不管下线。但**一个产品的健康度，不仅取决于做了什么，也取决于砍掉了什么。**

下线决策框架：
1. 使用率是否低于阈值（如 < 5% MAU）
2. 维护成本是否高于收益
3. 是否有更好的替代方案
4. 下线影响范围评估（哪些用户会受影响）
5. 渐进式下线计划（通知 → 灰度关 → 全量关）

## 版本规划：需求组合的艺术

### 单个需求 vs 版本规划

单个需求管理是"做对的事"，版本规划是"在正确的时间做正确的事"。

**版本规划三原则：**

1. **每个版本一个主题**（Theme）：不要一个大版本里塞 5 个不相关的功能
2. **大中小搭配**：每个版本包含 1-2 个大功能 + 3-5 个小优化 + 若干 bugfix
3. **留 buffer**：计划永远留 20% 的 buffer 给临时需求和 hotfix

### 版本类型与节奏

| 版本类型 | 频率 | 内容 | 用户感知 |
|----------|------|------|----------|
| 大版本 (Major) | 季度 | 核心功能、重大改版 | 有发布会/公告 |
| 小版本 (Minor) | 月度 | 功能迭代、体验优化 | 有更新日志 |
| 补丁 (Patch) | 周度/按需 | Bugfix、紧急修复 | 静默更新 |

### 需求池管理矩阵

```
          紧急
           ↑
   快速胜利 │  战略投入
  (Quick    │  (Strategic
   Wins)    │   Bets)
  ─────────┼───────── → 重要
   填充项   │  重大投入
  (Fill-ins)│  (Big
           │   Bets)
           ↓
          不紧急
```

- **快速胜利**：重要且紧急——优先做
- **战略投入**：重要但不紧急——排入近期版本规划
- **填充项**：紧急但不重要——授权给团队处理
- **重大投入**：不重要不紧急——说"不"

## 面试常见追问

### Q1：如果开发做到一半，发现做不完，怎么办？

1. **立即评估**：哪些已完成、哪些没做、风险有多大
2. **分情况处理**：
   - 核心路径已完成、边缘功能没做 → 可以上线，后续迭代补
   - 核心路径没完成 → 评估是否可以简化方案按时上线，或者延期
3. **决策原则**：宁可功能少一点按时上线，也不要功能全但延期——**信誉损失 > 功能缺失**

### Q2：需求池有 50 条需求，你如何做版本规划？

1. **先分类**：按主题归类（性能优化、新功能、体验改进...）
2. **再排序**：每个分类内按价值/成本排序
3. **组合**：每个版本选一个主题，搭配 1 大 + 数小 + 若干 fix
4. **检查依赖**：有些需求必须先做（如基础设施类）
5. **对齐 OKR**：确保版本内容能支撑季度目标

### Q3：如何管理来自不同 stakeholder 的冲突需求？

1. **量化优先级**：用统一框架（RICE / ICE）给所有需求打分
2. **对齐业务目标**：哪个需求更支撑当前季度 OKR
3. **透明化决策**：让 stakeholder 看到排期逻辑，理解为什么他们的需求排在后面
4. **提供替代方案**：完整版做不了，是否有轻量版可以先解决核心问题

## 标准回答框架

面试时被问"你如何管理一个需求"时，按六阶段回答：

1. **发现**（30 秒）：需求来源、如何验证真实性
2. **定义**（30 秒）：PRD 核心要素、如何澄清边界
3. **评审**（30 秒）：如何推进多方达成共识
4. **交付**（30 秒）：变更管理、进度跟进
5. **验证**（30 秒）：上线后看什么数据、如何判断成功
6. **迭代/下线**（30 秒）：基于数据决定下一步

## 关键收获

- 需求管理不是"写完 PRD 就结束了"——它是一个**六阶段闭环**
- 初级 PM 关注"怎么写 PRD"，高级 PM 关注"如何让需求在组织中**顺利流转并产生价值**"
- 版本规划是需求管理的**高阶能力**：不是做最重要的事，而是在正确的时间做正确组合的事
- **变更管理**是最考验 PM 成熟度的环节——混乱的变更流程是团队信任崩塌的开始
- 技术转 PM 的优势：你更理解开发评估的合理性，能更精准地**做技术-业务权衡**
