## 面试场景

> 面试官：「你加入一家快速增长的创业公司，发现团队的 PRD 质量参差不齐——有的详尽专业，有的缺失关键信息，导致开发频繁返工。老板让你设计一套 PRD 质量评估体系。你会怎么做？」

---

## 核心概念解析

### 为什么需要 PRD 质量评估体系？

PRD（Product Requirements Document）是产品经理最核心的工作产物之一。一份低质量的 PRD 会引发连锁反应：

```
PRD 质量低
    │
    ├─→ 开发理解偏差 → 返工 → 延期
    ├─→ 测试覆盖不全 → 漏测 → 线上事故
    ├─→ 设计反复修改 → 效率下降 → 团队摩擦
    └─→ 需求评审拖沓 → 决策延迟 → 错过窗口期
```

**关键洞察**：PRD 质量不是「写得好不好」的主观判断，而是可以被系统化定义、度量、改进的工程问题。

---

## 框架：PRD 质量成熟度模型（PQMM）

### Level 0：无规范阶段

- 没有统一模板
- 口头传达为主
- 开发靠猜，测试靠问

### Level 1：基础规范阶段

- 有 PRD 模板
- 包含基本要素：背景、目标、功能描述
- 但格式不统一，深度不一致

### Level 2：标准化阶段

- 团队统一的 PRD 模板和规范
- 有明确的必填项和评审流程
- PRD 评审是开发前置条件

### Level 3：量化管理阶段

- PRD 质量有可度量的指标
- 定期回顾 PRD 质量趋势
- 数据驱动改进

### Level 4：持续优化阶段

- PRD 质量数据与项目结果（延期率、Bug率）关联分析
- 自动化辅助检查（模板合规、完整性校验）
- 团队有 PRD 写作的最佳实践库

## PRD 质量评估维度：6C 框架

### 1. Completeness（完整性）

PRD 是否包含了所有必要的信息模块？

**检查清单**：
| 模块 | 权重 | 说明 |
|------|------|------|
| 背景与动机 | 10% | 为什么要做？解决了什么问题？ |
| 目标与指标 | 15% | 成功如何衡量？北极星指标是什么？ |
| 用户故事 | 10% | 谁在什么场景下需要什么？ |
| 功能详述 | 20% | 详细的交互逻辑、边界条件、异常处理 |
| 非功能需求 | 10% | 性能、安全、兼容性、可访问性 |
| 数据埋点 | 5% | 需要追踪哪些事件和指标？ |
| 排期与依赖 | 5% | 里程碑、前置依赖、风险 |
| 验收标准 | 15% | 什么样的结果算「完成」？ |
| 变更记录 | 5% | 版本历史、变更原因 |
| 附录 | 5% | 竞品参考、术语表、相关链接 |

**评分方法**：每个模块按 0-2 分打分
- 0 = 缺失
- 1 = 存在但不充分
- 2 = 存在且充分

### 2. Clarity（清晰度）

PRD 是否容易被正确理解？

**评估指标**：
- **歧义指数**：评审中「这里是什么意思？」的提问次数
- **往返次数**：需求评审后修改的轮数
- **阅读时间比**：理解 PRD 所需时间 / PRD 页数（越低越好）

**清晰度反模式**：
```
❌ "系统应该快速响应用户操作"     → 多快？什么操作？
✅ "用户点击搜索按钮后，结果列表
   应在 500ms 内展示，P95 标准"  → 可验证
```

### 3. Consistency（一致性）

PRD 内部及与其他文档之间是否一致？

**检查点**：
- 术语使用统一（不要混用「用户」和「客户」如果含义不同）
- 数据指标定义与公司指标字典一致
- 交互描述与原型一致
- 版本号和变更记录对应

### 4. Correctness（正确性）

PRD 中的信息是否准确？

**验证方法**：
- 数据引用可溯源
- 技术可行性已与开发确认
- 合规要求已与法务确认
- 用户假设已通过用研验证或标注为假设

### 5. Coverage（覆盖度）

PRD 是否覆盖了所有重要场景？

**场景完整性检查**：
```
┌──────────────────────────────────┐
│        场景覆盖度检查             │
├──────────────────────────────────┤
│                                  │
│  Happy Path ✓  ── 主流程         │
│                                  │
│  Edge Cases  ?  ── 边界条件      │
│    · 空状态                      │
│    · 极端数据                    │
│    · 并发操作                    │
│    · 网络异常                    │
│                                  │
│  Error Path  ?  ── 异常流程      │
│    · 权限不足                    │
│    · 数据冲突                    │
│    · 服务不可用                  │
│                                  │
│  Negative    ?  ── 反向操作      │
│    · 撤销                       │
│    · 超时取消                    │
│    · 重复提交                    │
│                                  │
└──────────────────────────────────┘
```

### 6. Concision（精练度）

PRD 是否简洁高效，没有冗余？

**信号**：
- 是否有大量复制粘贴的模板内容未实际使用？
- 是否有与当前版本无关的「未来规划」描述？
- 是否可以用地形图或流程图替代大段文字？

## 实操方案：建立 PRD 质量评估体系

### Phase 1：基线测量（第 1-2 周）

1. **抽取过去 3 个月的 10-15 份 PRD**
2. **用 6C 框架进行回溯评分**
3. **统计问题分布**：哪个维度得分最低？
4. **关联项目结果**：低分 PRD 是否对应更多返工/Bug？

**常见发现**：
```
完整性：7.2/10  ← 多数 PRD 缺少非功能需求和埋点
清晰度：5.8/10  ← 歧义问题最严重
一致性：7.5/10  ← 术语不统一是主要问题
正确性：8.0/10  ← 基本准确
覆盖度：5.5/10  ← 异常流程和边界条件覆盖最差
精练度：6.5/10  ← 模板冗余较多
```

### Phase 2：建立规范（第 3-4 周）

1. **制定 PRD 模板 v1.0**：基于 6C 检查清单
2. **建立术语库**：统一团队用语
3. **创建场景检查清单**：确保覆盖度
4. **定义评审流程**：
   - PRD 自检（作者用清单自评）
   - 同行评审（另一位 PM 审阅）
   - 技术评审（Tech Lead 评估可行性）
   - 正式需求评审（全团队）

### Phase 3：量化跟踪（第 5-8 周）

1. **PRD 评审记分卡**：

```
┌────────────────────────────────────────┐
│  PRD 质量评审记分卡                     │
│  PRD: [需求名称]  版本: v1.0           │
│  评审人: [姓名]    日期: YYYY-MM-DD    │
├────────────────────────────────────────┤
│                                        │
│  完整性    [0][1][2] 各模块检查        │
│  清晰度    [0][1][2] 无歧义描述        │
│  一致性    [0][1][2] 术语/数据统一     │
│  正确性    [0][1][2] 信息准确          │
│  覆盖度    [0][1][2] 场景完整          │
│  精练度    [0][1][2] 简洁高效          │
│                                        │
│  总分: ___/12                          │
│                                        │
│  关键问题:                             │
│  1. ________________________           │
│  2. ________________________           │
│  3. ________________________           │
│                                        │
└────────────────────────────────────────┘
```

2. **建立 PRD 质量看板**：
   - 每周/每迭代统计平均质量分
   - 按维度展示得分趋势
   - 标记低分 PRD 需要重点跟进

### Phase 4：持续改进（持续进行）

1. **月度 PRD 质量回顾**
2. **收集开发/测试对 PRD 的反馈**
3. **更新模板和检查清单**
4. **建立 PRD 最佳实践库**（优秀 PRD 示例 + 点评）

## 实战案例

### 某电商团队 PRD 质量提升

**背景**：某电商团队 15 人 PM 团队，迭代周期 2 周，平均每迭代产出 8 份 PRD。

**基线数据**：
- 平均评审轮数：3.5 轮
- 开发返工率：23%（因需求不清晰）
- PRD 平均页数：28 页（过多冗余）

**实施 3 个月后**：
- 平均评审轮数：1.8 轮 ↓ 49%
- 开发返工率：9% ↓ 61%
- PRD 平均页数：16 页 ↓ 43%
- 团队满意度：4.2/5

**关键动作**：
1. 建立了 6C 评分体系
2. 将 PRD 自检纳入提交流程
3. 每周分享一份「本周最佳 PRD」
4. 将异常流程覆盖度作为重点改进项

## 面试常见追问

### 追问1：「团队中有人觉得 PRD 质量评估太形式化，怎么推动？」

**回答策略**：
1. **用数据说话**：展示低质量 PRD 导致的返工成本和延期数据
2. **从痛点切入**：不要推「质量评估体系」，而是说「减少返工和评审次数」
3. **渐进式引入**：先从自愿参与开始，让效果说话
4. **降低门槛**：自检清单只需要 2 分钟，不是增加负担

### 追问2：「如何平衡 PRD 详尽程度和敏捷开发的效率？」

**回答框架**：
- **按需求复杂度分级**：
  - S 级需求：完整 6C 评估 + 多轮评审
  - M 级需求：精简 PRD + 同行评审
  - S/M 级需求：用户故事卡 + 口头对齐
- **关键原则**：PRD 的详细程度应与需求的不确定性成正比

### 追问3：「PRD 质量和 PM 个人能力的关系？」

**回答策略**：
- PRD 质量评估**不是考核工具**，而是改进工具
- 不要用 PRD 分数排名 PM
- 应该关注**团队整体趋势**和**共同改进**
- 个人能力差异通过导师制和最佳实践分享来弥合

### 追问4：「AI 时代 PRD 质量评估会被自动化吗？」

**回答框架**：
- AI 可以自动检查：模板合规性、术语一致性、基础覆盖度
- AI 难以替代的：业务正确性、逻辑深度、用户同理心
- 未来方向：AI 做 6C 中的 Consistency + Concision，人做 Correctness + Coverage
- **趋势**：PRD 评审从「找格式问题」转向「深度业务讨论」

## 标准回答框架

```
1. 现状诊断（30秒）
   "我会先抽样历史 PRD，用 6C 框架做基线评估"

2. 体系设计（1分钟）
   "从完整性、清晰度、一致性、正确性、覆盖度、精练度
    六个维度建立可量化的评分标准"

3. 推进策略（30秒）
   "分四阶段：基线测量 → 建立规范 → 量化跟踪 → 持续改进"

4. 预期效果（30秒）
   "核心目标：降低返工率和评审轮数，提升团队协作效率"

5. 风险应对（30秒）
   "最大风险是团队认为太形式化——用数据证明价值，
    渐进式引入，降低门槛"
```

## 关键收获

1. **PRD 质量是可度量的工程问题**：6C 框架提供了系统化的评估维度
2. **完整性是基础，覆盖度是分水岭**：多数团队在异常场景覆盖上得分最低
3. **质量评估的目的不是惩罚，而是改进**：用数据驱动团队写作水平提升
4. **渐进式推行是关键**：从自检清单开始，用效果推动采纳
5. **PRD 质量直接影响项目效率**：一份好的 PRD 可以减少 40-60% 的返工
6. **AI 会改变质量评估的方式，但不会消除对人的判断的需求**：深度业务逻辑的正确性仍需要人来把控
