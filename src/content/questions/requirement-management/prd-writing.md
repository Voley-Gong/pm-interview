# 如何写一份高质量的产品需求文档（PRD）？

## 面试场景

> 面试官："你负责一个新功能，需要和前端、后端、设计、测试四个团队协作。请描述你会如何撰写和组织你的需求文档，确保各方都能高效理解并执行？"

这道题考察的不是"你知道 PRD 的模板格式"，而是你是否理解 **PRD 的本质是一份沟通契约**——它替代你站在每个团队成员身边回答问题。

---

## 核心认知：PRD 到底解决什么问题？

很多新人 PM 的误区是：PRD = 功能描述文档。

**实际上，PRD 至少承担四重角色：**

| 角色 | 服务对象 | 核心诉求 |
|------|---------|---------|
| 决策记录 | 产品/业务方 | 为什么要做这个？目标是什么？ |
| 设计指引 | UI/UX 设计师 | 需要设计什么？约束条件是什么？ |
| 开发依据 | 前端/后端工程师 | 逻辑是什么？边界条件？异常处理？ |
| 验收标准 | 测试工程师 | 怎么算做完了？验收用例是什么？ |

**判断 PRD 质量的终极标准：一个不熟悉这个项目的人，只看文档能不能独立完成开发。**

---

## PRD 核心结构框架

### 1. 文档头信息

```
项目名称：XXX功能
版本：v1.0
作者：张三
日期：2026-06-17
状态：草稿 / 评审中 / 已确认 / 开发中
变更记录：v1.1 新增 XX 异常处理逻辑
```

> 看似简单，但"变更记录"是最常被忽视的——每次改了什么、为什么改、谁确认了，必须留痕。

### 2. 背景与目标（Why）

- **业务背景**：当前遇到了什么问题？数据表现如何？
- **用户价值**：解决了用户什么痛点？预期收益是什么？
- **目标设定**：用 SMART 原则定义可衡量的目标

```
❌ "提升用户体验"
✅ "降低购物车放弃率，从 65% 降至 55% 以内，3 个月内达成"
```

### 3. 用户故事与场景（Who & When）

```
作为 <用户角色>，
我想要 <做什么>，
以便于 <达到什么目的>。
```

**关键：写场景，不写功能。**

```
❌ "做一个搜索筛选功能"
✅ "作为求职者，我想按照薪资范围筛选职位，
    以便快速找到符合我预期的岗位，不用逐个翻看。"
```

### 4. 功能需求详述（What & How）

这是 PRD 的核心，需要覆盖：

**a) 功能流程图**
- 主流程（Happy Path）
- 异常流程（网络失败、权限不足、数据为空等）
- 分支流程（不同条件的不同路径）

**b) 界面说明**
- 页面布局草图 / 交互原型链接
- 每个元素的展示规则（什么时候显示/隐藏/置灰）
- 不同状态下的展示（空状态、加载中、错误、成功）

**c) 业务规则**
- 数据校验规则（输入限制、格式要求）
- 权限规则（谁能看、谁能操作）
- 边界条件（极值、并发、离线情况）

**d) 数据定义**
- 接口需要返回什么数据
- 数据来源
- 数据更新频率

### 5. 非功能需求

- **性能要求**：页面加载时间 < 2s，支持 1 万 QPS
- **兼容性**：iOS 14+ / Android 10+，各屏幕尺寸适配
- **安全性**：敏感信息脱敏、接口鉴权
- **埋点需求**：需要追踪哪些用户行为

### 6. 验收标准

```
Given（前置条件）< 已登录且在订单列表页 >，
When（操作）< 点击"取消订单"按钮 >，
Then（结果）< 弹出确认弹窗，确认后订单状态变为"已取消"，退款在3个工作日内原路退回 >
```

### 7. 排期与依赖

- 各方工作量预估
- 前置依赖（需要等 XX 接口先完成）
- 里程碑节点

## 高质量 PRD 的五个判断标准

### 标准 1：完整性——"不留暗角"

- 所有异常路径都有处理方案
- 所有空状态都有设计稿
- 所有数字都有来源和计算逻辑
- 所有"大概""可能"都被量化

### 标准 2：清晰度——"一句话一个意思"

```
❌ "用户可以在设置里调整消息通知的频率和方式"
   → 开发会问：频率有哪些选项？方式指什么？默认值？

✅ "设置页新增'消息通知'入口：
    - 频率选项：实时 / 每日汇总 / 每周汇总，默认每日汇总
    - 通知方式：App推送 / 短信 / 邮件，可多选，默认仅App推送"
```

### 标准 3：可追溯性——"每个决策都有依据"

- 功能设计基于哪个用户洞察或数据结论？
- 砍掉了哪些方案？为什么？
- 技术方案的选型讨论记录

### 标准 4：可测试性——"QA 能直接写用例"

- 每条需求都应该可以被验证或证伪
- 避免主观性描述（"好看的""流畅的""方便的"）

### 标准 5：可维护性——"改了不乱"

- 模块化组织，功能之间边界清晰
- 变更记录及时更新
- 过时内容及时标记或删除

## 实战案例：直播打赏功能 PRD 片段

**背景：** 平台主播希望增加收入来源，用户打赏意愿存在但缺乏便捷入口。

**目标：** 3 个月内，直播间打赏渗透率从 0 提升至 5%，ARPPU 达到 8 元。

**核心用户故事：**
> 作为直播观众，我想在观看时方便地给喜欢的主播打赏虚拟礼物，以表达喜爱和支持。

**关键业务规则：**
1. 用户余额 ≥ 礼物价格时，可直接打赏
2. 余额不足时，弹出充值引导，充值后自动完成打赏
3. 单次打赏金额上限 500 元（防沉迷）
4. 每日打赏上限 2000 元，达到上限后禁用打赏入口并提示
5. 主播端需实时收到打赏通知，延迟不超过 1 秒
6. 打赏记录在双方"钱包"中可见，支持按日期查询

**异常处理：**
- 网络超时：打赏请求重试 1 次，仍失败则退还扣款
- 主播已下播：打赏入口置灰，提示"主播已下播"
- 账号异常（被封禁）：禁止打赏

## 面试常见追问

### Q1：开发和设计觉得你的 PRD 不清晰，怎么办？

1. **先听具体哪里不清晰**——不要急于辩解，逐条记录问题点
2. **当场能澄清的立即补充**，不能的承诺时间节点
3. **反思根因**：是表达不够精确？还是自己没想清楚？
4. **改进机制**：建立 PRD 模板和自查清单，避免同类问题

### Q2：需求写到一半被老板叫去加急改方向，怎么处理？

1. **确认优先级和影响范围**——是真的紧急还是只是"老板觉得紧急"？
2. **评估对当前 PRD 的影响**——哪些部分需要改？影响多少人的工作量？
3. **及时同步所有相关方**——不要默默改了让开发踩坑
4. **版本管理**——PRD 必须有版本号，每次大改都留痕

### Q3：PRD 写多详细算够？

**原则：详细到"读完不需要追问"为止。**

- 团队默契高 → 可以精简，但逻辑和规则不能省
- 团队新人多 / 远程协作 → 越详细越好
- 涉及金钱、安全、隐私 → 每个细节都要写清楚
- 内部工具 / 快速验证 → 可以用 User Story + 验收标准替代完整 PRD

## 标准回答框架

```
1. 理解PRD的本质（30秒）
   → PRD 是沟通契约，不是功能清单

2. 说明PRD的核心结构（1分钟）
   → 背景→目标→用户故事→功能详述→非功能需求→验收标准

3. 展示自己的PRD方法论（1分钟）
   → 五个质量标准：完整、清晰、可追溯、可测试、可维护

4. 举例说明（1分钟）
   → 用一个真实项目展示你如何写PRD、遇到什么问题、如何改进

5. 强调协作（30秒）
   → PRD不是写完就结束，评审→跟进→变更管理是持续过程
```

---

## 关键收获

1. **PRD 的本质是沟通工具**——降低信息传递中的损耗和误解
2. **写清楚异常和边界比写主流程更重要**——80% 的开发返工来自这些遗漏
3. **好的 PRD 是迭代出来的**——不要追求一次完美，但每次评审都要有价值增量
4. **PRD 的深度反映 PM 的思考深度**——模糊的 PRD 通常意味着 PM 自己还没想清楚
5. **对于技术转PM的人来说，这是你的核心展示机会**——你对技术边界的理解，能让你写出更扎实的 PRD
