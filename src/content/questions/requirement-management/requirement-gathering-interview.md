## 面试场景

> 面试官：「你刚加入一个新产品团队，负责一款面向中小企业的SaaS工具。研发团队等着你给需求，但你对这个市场完全陌生。你会如何开展需求调研？具体说说你的调研方法和用户访谈策略。」

这道题考察的是 PM 最核心的基本功之一：**在没有明确需求输入的情况下，如何主动发现和收集真实需求**。很多 PM 习惯于"老板给需求""销售传需求""竞品抄需求"，但真正的需求管理始于系统化的调研。

---

## 核心概念解析

### 需求调研的三个层次

| 层次 | 目标 | 方法 | 产出 |
|------|------|------|------|
| **探索层** | 理解市场和用户全貌 | 桌面研究、行业报告、竞品分析 | 市场地图、竞争格局 |
| **验证层** | 验证具体痛点和场景 | 用户访谈、问卷调查、现场观察 | 用户画像、痛点清单 |
| **量化层** | 量化需求优先级和规模 | 数据分析、A/B测试、原型测试 | 需求优先级矩阵 |

### 用户访谈的本质

用户访谈不是"问用户要什么"——用户往往说不清自己想要什么。访谈的本质是：

1. **理解用户的真实工作流程**（而不是用户认为的流程）
2. **发现用户自己都没意识到的痛点**（隐性需求）
3. **验证你对问题的假设**（而不是寻找认同）

> "如果你问用户想要什么，他们会说要一匹更快的马。" —— 亨利·福特（ attributed）

## 框架：需求调研五步法（RADIANT）

### Step 1: Ready — 调研准备

**明确调研目标：**
- 我要回答什么问题？（例：中小企业为什么不用专业财务软件？）
- 我的假设是什么？（例：因为太贵/太复杂/不需要）
- 调研结果如何影响决策？

**确定调研对象：**
- 决策者（老板/CFO）
- 使用者（财务人员/出纳）
- 影响者（会计师/税务顾问）
- 流失用户（曾经用过但放弃了的人）
- 非用户（有需求但没用过此类产品的人）

**设计调研工具：**
- 半结构化访谈提纲（核心问题5-7个，备用问题10个）
- 访谈同意书/录音授权
- 调研记录模板

### Step 2: Ask — 访谈执行

**黄金五问法（5W1H进阶版）：**

```
1. Walk me through... — "请带我走一遍你上次做这件事的完整过程"
2. Tell me about the last time... — "说说你上一次遇到这个问题是什么情况"
3. What was the hardest part? — "在这个过程中最困难的是什么？"
4. Why did you...? — "你当时为什么选择这样做？"
5. What would ideal look like? — "如果有一个魔法工具，你希望它帮你做什么？"
```

**访谈技巧要点：**

| 技巧 | 说明 | 示例 |
|------|------|------|
| **开放式提问** | 避免是非题 | ❌"你觉得这个功能好吗？" → ✅"你平时怎么处理这个问题的？" |
| **追问细节** | 挖掘具体场景 | "你刚才说很麻烦，能具体说说是哪一步吗？" |
| **沉默等待** | 让用户主动补充 | 用户回答后停顿3秒，往往会补充更深层的信息 |
| **复述确认** | 确保理解正确 | "你的意思是，每次月底对账都要手动核对三张表？" |
| **避免引导** | 不暗示答案 | ❌"你是不是觉得这个功能太复杂了？" |

### Step 3: Digest — 信息消化与综合

**用户故事映射（User Story Mapping）：**
将访谈中收集的碎片信息，按照用户实际工作流程组织成一条完整的故事线。

```
发现问题 → 寻找方案 → 试用产品 → 日常使用 → 遇到问题 → 决定续费/流失
  │           │          │          │          │           │
  痛点1      痛点2      痛点3      痛点4     痛点5      痛点6
```

**痛点频率-严重度矩阵：**
- 高频率 + 高严重度 = 核心需求（P0）
- 高频率 + 低严重度 = 体验优化（P1）
- 低频率 + 高严重度 = 风险防护（P2）
- 低频率 + 低严重度 = 暂不处理（P3）

### Step 4: Interpret — 需求解读

将用户反馈翻译为产品需求的三个步骤：

1. **原始记录**："我每个月要花整整两天对账，眼睛都花了"
2. **痛点描述**：月结对账耗时过长（2天/月），手工操作易出错
3. **需求表述**：作为财务人员，我希望系统能自动匹配银行流水和内部账目，这样我可以将对账时间从2天缩短到2小时

### Step 5: Navigate — 调研结论与行动

输出一份调研报告，包含：
- **核心发现**（3-5个关键洞察）
- **用户画像**（基于真实数据而非猜测）
- **需求清单**（按优先级排序）
- **竞品差距**（基于用户认知而非功能对比）
- **建议的下一步行动**

## 实战案例：中小企业SaaS财务工具需求调研

### 调研背景
团队计划开发一款面向中小企业的财务SaaS，但不确定应该切入哪个具体场景。

### 调研执行

**第一周：桌面研究**
- 研究中小企业财务管理的行业报告
- 分析3款主要竞品的用户评价（App Store + 应用市场）
- 梳理中小企业财务流程的法规要求

**第二周：用户访谈**
- 拜访5家不同行业的中小企业
- 每家访谈2-3人（老板 + 财务人员 + 外部会计师）
- 每场访谈60-90分钟，全程录音

**第三周：数据补充**
- 发放线上问卷（300份）
- 收集竞品用户的NPS和流失原因

### 关键发现

1. **真正的痛点不是"功能不够"，而是"对账太痛苦"**——80%的受访者提到月结对账是最大痛点
2. **决策者和使用者的需求完全不同**——老板关心"能不能少请一个人"，财务人员关心"能不能少加点班"
3. **现有产品功能臃肿**——大多数产品试图服务所有规模的企业，中小企业只需要30%的功能

### 调研对产品的直接影响
- MVP聚焦"智能对账"单一场景，不做大而全的财务系统
- 定价按"节省的人力成本"来锚定，而不是按功能数量
- 营销信息分两条线：对老板讲ROI，对财务人员讲体验

## 面试常见追问

### Q1: "如果用户说的和做的不一样，你怎么处理？"
**标准回答框架：**
- 以行为数据为准，而不是口头表述
- 使用现场观察法（shadowing）验证用户说法
- 分析产品使用日志比对用户描述
- 原因分析：用户可能出于社会期望偏差（social desirability bias）或记忆偏差

### Q2: "调研样本量多少才够？你怎么判断？"
**参考框架：**
- 定性访谈：5-8人即可发现80%的主要问题（尼尔森可用性法则）
- 如果到第8个人时已经听不到新信息（信息饱和），说明样本足够
- 量化验证需要更大样本（至少100+），取决于总体规模和置信度要求

### Q3: "老板说'不用调研了，竞品做什么我们做什么就行'，你怎么回应？"
**应对策略：**
- 不直接反对，先理解老板的出发点（可能是赶时间/省预算）
- 用最小成本调研方案说服：「我只需要2天时间做5个用户访谈，就能确认竞品的功能是否真的适合我们"
- 用竞品失败案例说明不调研的风险
- 提出折中方案：先做轻量级调研，用结果争取更深入的调研预算

### Q4: "如何区分用户说的'伪需求'和'真需求'？"
**判断方法：**
- 看用户是否已经为解决这个问题付费或花时间（revealed preference）
- 看痛点出现的频率和严重程度
- 用"5个为什么"追问到根本原因
- 对比用户的行为和态度是否一致

### Q5: "如果调研结论和你最初的假设矛盾怎么办？"
**正确态度：**
- 这正是调研的价值所在——用最小成本验证假设
- 不要为了维护假设而选择性解读数据
- 分析矛盾的原因：假设本身有问题？调研对象不合适？调研方法有偏差？
- 如果有足够数据支撑，勇敢推翻假设并调整方向

## 标准回答框架（面试时使用）

```
1. 框架展示（30秒）
   "我会用RADIANT五步法来系统化需求调研：
   准备→访谈→消化→解读→行动"

2. 差异化亮点（60秒）
   - 强调调研对象的多样性（决策者 vs 使用者 vs 非用户）
   - 强调定性+定量的结合
   - 提到信息饱和原则控制样本量

3. 真实案例（90秒）
   用一个自己做过的或了解的调研案例来说明
   重点讲"调研前假设 vs 调研后发现"的反差

4. 落地行动（30秒）
   调研不是目的，要转化为具体的PRD和迭代计划
```

## 关键收获

1. **需求调研不是"问用户想要什么"，而是"理解用户在做什么、痛在哪里"**
2. **访谈的核心技巧是开放式提问+追问+沉默——让用户自己讲故事**
3. **调研对象要多元化：决策者、使用者、流失用户、非用户缺一不可**
4. **痛点频率-严重度矩阵是需求优先级排序的利器**
5. **调研结论要与最初假设对比，矛盾才是最有价值的发现**
6. **调研不是一次性的，而是贯穿产品全生命周期的持续活动**
