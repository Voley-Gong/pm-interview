## 面试场景

> 面试官：「你负责一个支付系统改造项目，其中涉及风控、收银台、账务清算三个团队的需求。风控的新规则引擎依赖收银台传新的字段，账务清算的改造又依赖风控的决策结果。你如何梳理和管理这些需求之间的依赖关系，确保按时交付？」

## 核心概念解析

### 什么是需求依赖？

需求依赖是指一个需求的实现前提是另一个需求先完成。在多团队、多模块的复杂项目中，需求之间的依赖关系会形成一张**有向图**，而产品经理的核心职责之一就是管理这张图。

### 依赖的四种类型

| 类型 | 定义 | 例子 |
|------|------|------|
| **强依赖（必须先做）** | A不做完，B无法启动 | 收银台不传新字段，风控无法用新规则 |
| **弱依赖（最好先做）** | A先做会让B更顺利，但B也能独立做 | 风控先做好降级方案，收银台改造可以并行 |
| **双向依赖（循环）** | A依赖B，B也依赖A——这是危险信号 | 风控需要账务的清算结果做判断，账务需要风控的决策做清算 |
| **隐式依赖（跨层）** | 表面无直接关系，但共享底层资源 | 风控和收银台都改了用户服务的接口，产生隐性冲突 |

## 需求依赖管理框架

### 框架一：DAG 可视化——把依赖画出来

**DAG（Directed Acyclic Graph，有向无环图）** 是需求依赖管理的数学基础。

```
     ┌──────────────┐
     │  需求A: 新字段  │
     │  （收银台）      │
     └──────┬───────┘
            │
     ┌──────▼───────┐     ┌──────────────┐
     │  需求B: 新规则  ├────►│  需求D: 联调   │
     │  （风控）       │     │  （全链路）    │
     └──────┬───────┘     └──────┬───────┘
            │                    │
     ┌──────▼───────┐            │
     │  需求C: 清算   ├────────────┘
     │  （账务）       │
     └──────────────┘
```

**关键原则：DAG 中不能有环。** 如果你画出了环（A→B→C→A），说明需求定义有问题，必须通过拆分或定义接口来打破循环。

### 框架二：关键路径法（CPM）

关键路径是 DAG 中**耗时最长的路径**，决定了项目的最短完成时间。

以上面的例子：
- 路径1: A → B → D（收银台→风控→联调）= 2 + 3 + 1 = **6周**
- 路径2: A → C → D（收银台→账务→联调）= 2 + 4 + 1 = **7周** ← 关键路径
- 路径3: A → B → C → D（收银台→风控→账务→联调）= 2 + 3 + 4 + 1 = **10周** ← 实际关键路径（如果C依赖B）

**PM 的关键洞察：关键路径上的任何延期 = 整体项目延期。** 非关键路径上的需求有缓冲时间（Float/Slack）。

### 框架三：依赖管理矩阵

用一张矩阵表来管理跨团队依赖：

| 依赖ID | 上游需求 | 上游团队 | 下游需求 | 下游团队 | 依赖类型 | 接口定义 | 预计完成 | 风险等级 |
|--------|---------|---------|---------|---------|---------|---------|---------|---------|
| D01 | 新增risk_level字段 | 收银台 | 风控规则v2 | 风控 | 强依赖 | API文档v1.2 | W4 | 高 |
| D02 | 风控决策透传 | 风控 | 清算规则更新 | 账务 | 强依赖 | 消息队列协议 | W7 | 高 |
| D03 | 用户服务改造 | 收银台 | 用户服务改造 | 风控 | 隐式依赖 | 共享Service层 | W3 | 中 |

## 实战案例：支付系统改造

### 项目背景

你的支付平台要做"基于风险的动态路由"——根据风控评分实时决定走哪个支付通道。涉及三个团队：

- **风控团队**：开发新的评分模型（预计5周）
- **收银台团队**：改造支付路由逻辑（预计3周，但依赖风控的评分接口）
- **账务团队**：更新清算规则支持多通道（预计4周，依赖收银台的路由结果）

### 第一步：画依赖图，找关键路径

```
风控评分模型(W5) → 收银台路由(W3) → 账务清算(W4)
         总工期: 5 + 3 + 4 = 12周
```

这就是关键路径——12周。

### 第二步：寻找并行优化机会

能否打破串行？可以！

**优化策略：接口先行（Contract First）**
1. **W1**: 三个团队一起定义接口契约（评分API格式、路由决策格式、清算数据格式）
2. **W2-W5**: 风控开发评分模型，收银台用Mock接口并行开发路由逻辑
3. **W5-W6**: 收银台切换到真实接口，联调（1周）
4. **W4-W7**: 账务在路由逻辑确定后即可并行开发清算规则

**优化后关键路径：**
```
接口定义(W1) → 风控模型(W4) → 路由联调(W1) → 清算联调(W1) = 7周
```

从12周压缩到7周，省了5周。

### 第三步：建立风险对冲

| 风险 | 概率 | 影响 | 对冲方案 |
|------|------|------|---------|
| 风控模型准确率不达标 | 中 | 延期2周 | 准备规则引擎降级方案，不阻塞收银台 |
| 接口定义频繁变更 | 高 | 返工 | 接口变更需三方签字，每周对齐一次 |
| 账务团队同时有其他项目 | 高 | 资源冲突 | 提前跟账务TL锁定人力，或在W4开始前确认排期 |

## 面试常见追问

### 追问1：「如果关键路径上的团队延期了怎么办？」

四步应急：
1. **确认影响范围**：延期多久？是否影响下游所有团队？
2. **评估缓冲**：下游有没有Float时间可以吸收？
3. **压缩策略**：能否并行化（加人）或缩小范围（先上MVP版本）？
4. **升级与沟通**：影响里程碑的延期必须在24小时内通知所有stakeholder

### 追问2：「如何避免隐式依赖？」

隐式依赖是最危险的——你以为没有依赖，直到上线时才发现冲突。

三个预防手段：
- **共享资源审查**：每次需求评审时，列出"这个需求改了哪些共享Service/表/配置"
- **架构对齐会**：每两周一次，各团队技术负责人同步技术变更
- **灰度发布设计**：任何新功能上线都通过feature flag控制，确保可以独立开关

### 追问3：「需求依赖和任务依赖有什么区别？」

- **需求依赖**是产品层面的（功能A需要功能B的数据才能运行）
- **任务依赖**是工程层面的（前端需要后端的API才能联调）

PM管需求依赖，Tech Lead管任务依赖。但两者必须对齐——需求依赖往往决定了任务依赖的方向。

## 标准回答框架

面试中回答需求依赖管理问题的五步法：

### Step 1: 识别依赖（30秒）
"首先我会把所有需求列出来，标注每个需求的前置条件和输出物，从而识别出需求间的依赖关系类型——强依赖、弱依赖还是隐式依赖。"

### Step 2: 可视化与关键路径（1分钟）
"然后我会画出DAG依赖图，标注每条路径的预计工期，找出关键路径。关键路径决定了项目的最短交付周期。"

### Step 3: 优化排期（1分钟）
"针对关键路径，我会用三种策略压缩工期：接口先行让团队并行、调整需求范围缩短单路径工期、将弱依赖从串行改为并行。"

### Step 4: 风险对冲（30秒）
"我会为关键路径上的高依赖需求准备Plan B——降级方案、备用资源或可裁剪的范围。"

### Step 5: 持续监控（30秒）
"建立依赖看板，每周更新依赖矩阵，追踪上游交付状态。任何上游变化第一时间通知下游团队。"

## 常见误区

| 误区 | 正确做法 |
|------|---------|
| 把所有需求画成一条串行链 | 找出可以并行的分支，优化关键路径 |
| 只关注强依赖，忽视隐式依赖 | 定期做架构对齐，发现共享资源冲突 |
| 接口在开发中逐步"有机"形成 | 接口先行，各方法成契约后再并行开发 |
| 依赖管理只在排期时做一次 | 每周持续更新依赖状态，动态调整 |

## 关键收获

1. **DAG 是需求依赖管理的核心工具**——有向无环图帮你看清全局，环意味着需求定义有问题
2. **关键路径决定项目工期**——非关键路径有缓冲，关键路径零容错
3. **接口先行（Contract First）是打破串行依赖的最有效手段**——让团队从"排队等接口"变成"并行用Mock"
4. **隐式依赖是最大的定时炸弹**——通过架构对齐会和共享资源审查来提前发现
5. **依赖管理是动态过程**——不是排期时画一次图就结束了，而是需要每周持续追踪和调整
