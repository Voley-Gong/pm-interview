# 版本开发到一半需求优先级变了，你如何动态调整？

## 面试场景

> 面试官：你的版本计划已经确定，开发进行到50%。突然销售团队接到一个大客户的定制需求，老板要求插入版本。同时，竞品刚上线了一个重要功能，你需要跟进。你怎么处理？

这道题考察的是PM在**动态环境下的优先级管理能力**。现实中，需求优先级几乎不可能一成不变，面试官想看你是否有系统化的调整机制，而不是手忙脚乱或一味拒绝。

---

## 核心概念解析

### 静态优先级 vs 动态优先级

| 特征 | 静态优先级 | 动态优先级 |
|------|-----------|-----------|
| 适用场景 | 长期路线图、季度规划 | Sprint级、版本开发中 |
| 调整频率 | 季度/月度review | 随时响应 |
| 决策依据 | 战略对齐、用户价值 | 紧急程度、信息更新、机会成本 |
| 主要风险 | 响应慢、脱离市场 | 团队疲劳、scope creep |

成熟的PM不是"不调整"，而是**有原则地调整**。

### 优先级动态调整的本质

优先级调整本质上是一个**持续的资源重新分配决策**。每次调整都会产生三个影响：

```
插入新需求 → 
├── 原有需求被挤出（交付延迟）
├── 团队上下文切换成本（效率损失）
└── stakeholder预期管理（信任影响）
```

好的PM不是消除这些影响，而是**让影响可见、可控、可解释**。

## 框架：动态优先级调整的 DART 模型

### D - Detect（识别变化信号）

什么信号应该触发优先级重新评估？

| 信号类型 | 具体表现 | 紧急程度 |
|----------|---------|----------|
| **市场变化** | 竞品上线关键功能、行业政策变化 | 高 |
| **客户变化** | 大客户需求变更、客户流失风险 | 中-高 |
| **数据变化** | 某功能使用率骤降/骤升、新增负面反馈趋势 | 中 |
| **技术变化** | 技术方案发现不可行、依赖的第三方服务变更 | 高 |
| **资源变化** | 团队成员变动、新增/削减人力 | 中-高 |
| **战略变化** | 公司方向调整、新OKR下达 | 高 |

**关键原则**：不是每个信号都需要调整。建立信号评估机制，避免"噪音驱动"的频繁调整。

### A - Assess（评估影响）

收到调整信号后，用以下矩阵评估：

```
           高价值
              │
     Ⅱ 快速插入  │  Ⅰ 重新规划
    ─────────┼─────────
     Ⅲ 延后排期  │  Ⅳ 坚守原计划
              │
           低价值
     低紧迫 ←────────→ 高紧迫

 价值 = 对产品目标/用户价值的影响
 紧迫 = 不做会怎样（时间窗口、竞争压力）
```

**评估清单**：
- [ ] 如果不插入，最坏结果是什么？
- [ ] 如果插入，哪个需求被挤出？对那个需求的stakeholder影响？
- [ ] 上下文切换成本是多少？（通常 = 2-3天的效率损失）
- [ ] 这个变化是"必须做"还是"做了更好"？
- [ ] 信息是否充分？还是需要先验证假设？

### R - Respond（执行调整）

根据评估结果，选择响应策略：

**策略1 - 交换式调整**（推荐）
```
"我们可以插入这个需求，但需要把X功能移到下个版本。
X功能的stakeholder是Y，我需要先和他确认。"
```
核心思想：**总量不变，内部交换**。每次插入必须明确挤出什么。

**策略2 - 延迟式调整**
```
"这个需求确实重要，但当前版本已经进入测试阶段。
我把它放入下个版本的P0，预计3周后启动。"
```
核心思想：**承认重要性，但用版本节奏保护当前交付**。

**策略3 - 分解式调整**
```
"我们不做完整版本，但可以从大客户需求中提取核心功能，
用5天做一个简化版先上线。完整版放下个版本。"
```
核心思想：**MVP思维解决紧迫性问题**。

**策略4 - 防御式坚守**
```
"我理解这个需求的重要性。但当前版本如果再插入，
会导致测试时间被压缩、质量风险大幅上升。
建议保持当前版本scope，优先级已排入下个版本P0。"
```
核心思想：**有时候最好的决策是说"不"**。需要数据和逻辑支撑。

### T - Transparency（透明化沟通）

动态调整最大的风险不是调错了，而是**stakeholder不知道为什么调**。

**透明化工具**：

1. **优先级变更日志**
```
| 日期 | 变更内容 | 变更原因 | 影响范围 | 决策人 |
|------|---------|---------|---------|--------|
| 6/15 | 插入客户A定制需求 | 成单依赖，Q2收入目标 | 功能C延期到v2.1 | PM+老板 |
| 6/18 | 竞品功能跟进降级 | 竞品功能数据表现一般 | 无 | PM |
```

2. **版本看板更新**
- 每周更新版本scope和进度
- 变更项高亮标注，附原因
- 所有stakeholder可见

3. **变更通知机制**
- 影响交付日期的变更 → 正式邮件通知 + 理由说明
- 内部scope调整 → 版本看板备注 + 站会同步
- 紧急插入 → 即时群通知 + 走变更审批流程

## 实战案例

### 场景：大客户需求插入 + 竞品跟进

回到面试题的场景。你的处理流程：

**Step 1 - 紧急评估（当天）**

| 新增需求 | 价值评估 | 紧迫度 | 初步判断 |
|----------|---------|--------|---------|
| 大客户定制需求 | 高（成单100万+） | 高（客户给deadline：月底） | 需要插入 |
| 竞品功能跟进 | 中（不是核心差异化） | 中（观望2周看数据） | 可延迟 |

**Step 2 - 评估挤出影响**

大客户需求预估工作量：8人天。当前版本剩余人力：6人天。缺口2人天。

方案对比：
```
方案A：插入大客户需求，功能C延期到下版本
  → pro: 拿下大单，客户满意
  → con: 功能C的stakeholder（运营团队）会有意见

方案B：大客户需求做MVP版（5人天），完整版下版本
  → pro: 平衡了客户和内部需求
  → con: 需要客户接受简化版

方案C：加人补齐2人天缺口
  → pro: 都不延期
  → con: 新人上手成本，可能影响质量
```

**Step 3 - 决策与沟通**

选择方案B（分解式调整）：

```
对老板：
"大客户需求我拆成两期。核心功能（3个场景）在当前版本交付，
扩展功能放入v2.1。这样既能赶上客户deadline，
又不影响运营团队的计划。已和客户确认可接受。"

对运营团队（功能C的stakeholder）：
"功能C按计划在本版本交付，不受影响。
大客户需求我们用MVP方式处理了，不会挤占。"

对开发团队：
"本周plan不变。下周一启动大客户MVP需求，
预计3天开发+2天测试。具体PRD周五前给到。"

对客户：
"核心场景我们月底前交付，扩展功能在下个版本。
想确认下：三个核心场景能覆盖你们80%的使用吗？"
```

**Step 4 - 竞品跟进的决策**

```
"竞品功能先不跟进。我设置了一个2周后的review：
如果竞品这个功能的数据表现好（用户使用率>15%），
我们在下个版本快速跟进。如果表现一般，就不投入。"
```

这个决策体现了：**不是所有竞品动向都需要立即响应**。用数据驱动决策，而非焦虑驱动。

## 面试常见追问

### 追问1："老板直接拍板要插入，你没得选怎么办？"

**回答思路**：
1. 承认现实：老板有权做这个决定，PM的角色不是对抗
2. 但PM的职责是让决策更优：
   - "好的老板，我来安排。同时想同步下影响：插入后功能C会延期1周，运营的活动计划可能受影响，我需要和他们协调一下。"
   - 把影响做visible，即使不能改变决定，也能让决策方知道全貌
3. 事后复盘：如果这种情况频繁发生，说明需要向上管理，建立变更审批流程

### 追问2："团队对频繁调整很不满，你怎么处理？"

**回答思路**：
1. 承认情绪合理性：开发团队最讨厌的是不确定性，不是变更本身
2. 区分"必要的变更"和"管理混乱"：
   - 必要的变更（市场驱动）：做好解释和context
   - 管理混乱（stakeholder随意提需求）：建立门槛和流程
3. 具体措施：
   - 设定版本冻结期：开发到70%后不再接受新需求插入
   - 每次变更说清楚"为什么变"，建立信任
   - 保护团队的专注时间，变更通过PM统一入口，不让stakeholder直接找开发

### 追问3："你怎么判断一个变更是否值得调整整个版本计划？"

**回答思路**：
用"三因素法"判断：
1. **不可逆性**：如果不做，是否造成不可逆的损失？（大客户流失 = 高不可逆）
2. **时间窗口**：这个需求是否有时间窗口限制？（月底deadline = 有窗口）
3. **机会成本**：调整后损失的代价 vs 不调整损失的代价，哪个更大？

如果三个因素中有两个指向"必须调整"，就调整。否则用延迟策略。

## 标准回答框架

```
1. 承认复杂性（20秒）
   "版本开发中的优先级调整是PM日常工作的常态，
   关键不是回避调整，而是建立机制让调整有序、透明、可解释。"

2. 展示框架（60秒）
   "我会用DART模型处理：
   - Detect：识别变化信号，区分信号vs噪音
   - Assess：用价值-紧迫度矩阵评估
   - Respond：选择交换/延迟/分解/坚守策略
   - Transparency：变更日志+看板+通知机制"

3. 场景实战（90秒）
   回到面试题场景，逐一处理：
   - 大客户需求 → MVP分解策略
   - 竞品跟进 → 数据驱动延迟决策
   - 各方stakeholder → 差异化沟通

4. 原则总结（30秒）
   "动态调整的核心原则：每次调整都要回答三个问题——
   为什么变？影响什么？谁需要知道？"
```

## 常见错误模式

| 错误模式 | 表现 | 后果 |
|----------|------|------|
| **好好先生** | 所有需求都答应插入 | 版本永远延期，团队精疲力竭 |
| **铁板一块** | 拒绝所有变更 | 错过市场机会，stakeholder信任崩塌 |
| **暗箱操作** | 调整了但不告诉所有人 | 信息不对称导致协作断裂 |
| **无原则调整** | 谁声音大听谁的 | 失去PM的主动权，变成传话筒 |
| **过度调整** | 每周都大改优先级 | 团队陷入上下文切换地狱 |

## 关键收获

1. **动态调整是常态，不是异常**：好的PM接受变化，但用机制管理变化
2. **DART模型**：Detect → Assess → Respond → Transparency，是处理任何变更的结构化方法
3. **四种响应策略**：交换、延迟、分解、坚守——不是只有"插入"或"拒绝"两个选项
4. **透明度是信任的基础**：每次调整的可解释性比调整本身更重要
5. **向上管理是关键**：频繁变更往往不是PM的问题，而是缺少组织级的变更管理流程——优秀的PM会推动建立这个流程
6. **面试核心信号**：面试官想看到的是"我不怕变化，但我有能力让变化有序"——而不是"我会保护团队不受干扰"或"我会满足所有需求"
