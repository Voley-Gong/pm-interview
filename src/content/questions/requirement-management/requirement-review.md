# 如何主持一场高效的需求评审？

## 面试场景

> 面试官：「你组织过需求评审吗？如果开发和设计对需求理解不一致，甚至在评审会上产生冲突，你作为PM怎么处理？请完整描述你主持需求评审的方法论。」

这是一道高频出现的PM面试题，尤其在招中高级PM时几乎必问。它考察的不是「你知道需求评审是什么」，而是你能否**在有阻力的情况下推动共识**。

---

## 一、核心概念：需求评审的本质

### 常见误区

| 误区 | 真相 |
|------|------|
| 评审 = 宣读PRD | 评审 = 对齐理解 + 暴露风险 + 达成共识 |
| 所有人都该关心每个细节 | 不同角色关注点完全不同 |
| 评审会越长越好 | 超过90分钟的评审效率断崖式下降 |
| 评审通过 = 万事大吉 | 评审通过只是起点，执行中还会不断变化 |

### 需求评审的三个层次

1. **信息层**：让所有人知道「做什么、为什么做、做成什么样」
2. **逻辑层**：验证「需求是否自洽、边界是否完整、异常是否覆盖」
3. **共识层**：让团队「认同优先级、接受时间线、承诺交付」

很多PM只做到第一层，所以评审会开完，开发还是各写各的。

---

## 二、框架：PREP需求评审法

我总结了一套 **PREP框架**（Prepare → Review → Engage → Pin），帮你系统化地主持评审。

### P - Prepare（会前准备）

**做什么：**
- 提前2天发送PRD草案和评审议程给参会者
- 标注PRD中的「关键决策点」和「开放讨论点」
- 对高风险模块，提前与核心技术骨干1on1沟通

**为什么重要：** 评审会上的很多争论，本质是「信息不对称导致的防御反应」。提前给信息，让人有时间消化，争论质量会大幅提升。

**实操模板：**
```
评审邀请邮件/消息模板：

主题：【需求评审】XXX功能 V1.0 PRD评审
时间：YYYY-MM-DD HH:MM（预计60分钟）
参会人：开发@XX、设计@XX、测试@XX、运营@XX

会前准备：
1. 请阅读附件PRD（重点看第3节「功能详述」和第5节「边界条件」）
2. 标注了3个关键决策点（PRD中标黄部分），请提前思考
3. 如有疑问，可在会前直接在文档中评论

议程：
- 0-10min：背景与目标同步
- 10-35min：核心功能走查（按用户路径）
- 35-50min：技术可行性与边界讨论
- 50-60min：确认action items与next steps
```

### R - Review（评审执行）

**核心原则：按用户路径走查，而不是按功能模块罗列**

错误方式：「这个页面有5个按钮，按钮A做XX，按钮B做XX……」
正确方式：「用户小明想完成XX任务，他进入页面后第一步看到XX，点击XX后触发XX……」

**按角色引导关注点：**

| 角色 | 关注重点 | PM应引导的问题 |
|------|---------|---------------|
| 开发 | 技术可行性、性能、边界条件 | 「这个方案在XX约束下可行吗？有什么技术风险？」 |
| 设计 | 交互流程、视觉一致性、异常态 | 「这个流程是否有断点？空状态设计了吗？」 |
| 测试 | 验收标准、异常场景、兼容性 | 「这个功能的验收标准大家看看是否完整？」 |
| 运营/业务 | 上线节奏、配置项、数据追踪 | 「上线需要运营配合什么？埋点是否覆盖？」 |

### E - Engage（处理冲突与分歧）

这是评审中最考验PM能力的环节。

**冲突处理三级响应：**

**Level 1 - 理解分歧（最常见）**
- 表现：「我理解的不一样……」
- 处理：回到PRD原文，用具体场景重新对齐。画图、举例子。
- 话术：「让我确认一下，你理解的是XX，而我的设计意图是YY。差异在于ZZ。大家觉得哪种更合理？」

**Level 2 - 方案分歧（常见）**
- 表现：「我觉得这样做不太好/有更好的方案」
- 处理：引导对方说出完整方案，评估trade-off，现场能定就定，不能定就记为offline讨论项。
- 话术：「你的方案有道理。我们对比一下：方案A优势是XX但代价是YY，方案B优势是YY但代价是XX。我们5分钟内决定，定不了我会后会找XX再确认。」

**Level 3 - 优先级/资源冲突（较严重）**
- 表现：「这个做不了/时间不够/人手不够」
- 处理：不要在评审会上争论资源问题。确认约束条件，会后升级处理。
- 话术：「我理解当前资源有约束。我们先对齐需求本身是否合理，资源和排期的问题我会后跟XX对齐，明天给出答复。」

### P - Pin（确认与收口）

**评审结束前必须完成的5件事：**
1. 复述所有action items（谁、做什么、什么时间前）
2. 确认所有「待定项」的负责人和截止时间
3. 确认下一步的关键里程碑（如技术方案完成时间、设计稿交付时间）
4. 发送会议纪要（24小时内）
5. 更新PRD版本号和changelog

## 三、实战案例

### 场景：电商App购物车优化需求评审

**背景：** 需要将购物车从「数量修改」升级为「规格+数量修改」，涉及前端、后端、设计三端。

**会前准备：**
- PRD提前3天发出，附带交互原型链接
- 提前与后端技术leader确认了接口改动的可行性
- 标注了2个关键决策点：①是否支持混批优惠 ②历史订单的兼容方案

**评审过程：**
- 前10分钟同步业务背景（为什么改、预期收益）
- 中间30分钟按用户路径走查（加入购物车→修改规格→结算）
- 后端提出：「历史订单的旧规格数据怎么展示？」——这是PRD遗漏点
- PM现场决定：「旧订单展示旧规格但标记'历史规格'，不支持修改，只支持重新下单」
- 设计提出空状态和异常态补充方案
- 测试确认验收标准已覆盖主流程和3个异常分支

**评审结果：**
- 3个action items明确到人
- 1个待定项（混批优惠排期）标记为P2
- PRD更新至V1.1，记录了评审中的4处修改

## 四、面试常见追问

### Q1：「评审中开发说做不了，你怎么办？」
回答框架：
1. 先理解「做不了」的具体原因（技术约束？时间不够？理解偏差？）
2. 如果是技术约束：请技术提供替代方案，评估trade-off
3. 如果是时间不够：拆分需求，确认MVP范围，剩余排入后续迭代
4. 如果是理解偏差：重新对齐需求目标，回到用户价值层面讨论

### Q2：「评审会没人发言怎么办？」
回答要点：
- 这通常是因为准备不充分或氛围过于权威
- 提前指定「每个角色至少提1个问题」的规则
- 用具体问题代替开放问题（不说「大家有什么想法？」，而是「@开发，这个接口改动你评估大概几个人日？」）
- 会前与关键人1on1预热

### Q3：「评审结论执行不下去怎么办？」
回答要点：
- 确认是「认知问题」还是「态度问题」
- 认知问题：重新同步背景和目标，用数据说话
- 态度问题：升级到双方leader，用OKR/目标对齐来推动
- 建立需求变更流程，所有偏离都需要正式提变更而非口头沟通

## 五、标准回答框架

面试时用这个结构回答，逻辑清晰且有深度：

```
1. 定义（15秒）：需求评审的本质是共识构建，不是信息宣读
2. 方法论（60秒）：描述你的评审流程框架（如PREP）
3. 关键挑战（30秒）：冲突处理、注意力管理、收口确认
4. 真实案例（45秒）：用一个具体项目展示你如何处理评审中的困难
5. 反思（15秒）：你从某次失败的评审中学到了什么
```

---

## 六、关键收获

1. **需求评审的80%工作在会前**。会前准备充分，评审效率提升一倍。
2. **按用户路径走查**，不要按功能模块罗列。前者让所有人理解「为什么」，后者只让人记住「做什么」。
3. **冲突不是坏事**。评审中暴露的分歧，总比上线后暴露的bug成本低100倍。
4. **每次评审必须有action items和待定项的收口**。没有收口的评审等于没开。
5. **技术转PM的人在评审中要克制「自己写代码时的思维」**，不要越位做技术决策，而是引导技术团队自己做决策。
