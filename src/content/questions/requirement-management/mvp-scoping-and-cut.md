## 面试场景

> 面试官：「你是某B2B SaaS产品的PM，销售团队反馈客户急需一个『审批工作流』功能，否则大单签不下来。研发团队评估要3个月。CEO说必须在4周内出可用版本。你怎么界定这个MVP的范围？」

这道题考察的是候选人在真实业务约束下，能否用结构化方法做出合理的范围取舍——既不是简单地说「砍功能」，也不是盲目承诺。

---

## 一、核心概念澄清

### 1.1 MVP 的本质不是「最小功能集」，而是「最便宜的假设验证」

| 常见误解 | 正确理解 |
|---------|---------|
| MVP = 把完整产品砍到最简 | MVP = 用最低成本验证最关键的假设 |
| MVP 是缩小版的最终产品 | MVP 可能根本不是软件（可以是人工流程） |
| 所有功能都做一点 | 核心路径做透，非核心先不做 |
| 做完MVP就直接做完整版 | MVP的产出是「学习」，不是代码 |

### 1.2 MVP 裁剪的三层思考

```
第一层：这个产品要验证的核心假设是什么？
第二层：验证这个假设需要的最小路径是什么？
第三层：路径上的哪些环节可以用非产品化方式替代？
```

## 二、需求裁剪框架：MoSCoW + 假设映射 + 代价矩阵

### 2.1 第一步：假设映射（Assumption Mapping）

列出产品背后的所有假设，按「信心度」和「重要性」分类：

```
         高重要性                    低重要性
高信心  [无需验证，直接做]        [暂时忽略]
       例：审批需要登录           例：用户喜欢蓝色界面

低信心  [MVP必须验证！]           [有时间再做]
       例：用户会用移动端审批     例：审批模板可自定义
```

**MVP只聚焦「低信心 + 高重要性」象限的功能。**

### 2.2 第二步：MoSCoW 分级

| 级别 | 含义 | MVP中的角色 | 审批工作流举例 |
|-----|------|-----------|--------------|
| **Must Have** | 没有就不能用 | 必须在MVP中实现 | 创建审批单、提交审批、审批通过/驳回 |
| **Should Have** | 很重要但可变通 | 尽量做，可用降级方案 | 审批催办通知（可先用人工提醒） |
| **Could Have** | 锦上添花 | MVP不做 | 自定义审批条件、审批数据看板 |
| **Won't Have** | 明确不做 | 明确排除 | 审批流程模板市场、跨系统审批 |

### 2.3 第三步：代价-价值矩阵

对每个 Must Have 功能做成本估算：

| 功能 | 用户价值(1-5) | 开发成本(人天) | 价值/成本比 | 决策 |
|-----|-------------|--------------|-----------|------|
| 审批单创建 | 5 | 5 | 1.0 | ✅ 做 |
| 多级审批流转 | 4 | 8 | 0.5 | ⚠️ 降级为单级 |
| 审批通知推送 | 4 | 3 | 1.3 | ✅ 做 |
| 审批历史记录 | 3 | 4 | 0.75 | 🔄 用日志替代 |
| 审批批量处理 | 2 | 5 | 0.4 | ❌ 不做 |
| 自定义审批规则 | 3 | 12 | 0.25 | ❌ 不做 |

## 三、实战案例：4周交付审批工作流MVP

### 3.1 挖掘真实需求

**第一步：问题深挖**

不要直接接受「客户需要审批工作流」这个解决方案级需求。追问：
- 客户现在的审批是怎么做的？（邮件？纸质？微信群？）
- 最痛的环节是什么？（找不到审批人？不知道进度？审批太慢？）
- 为什么现在签不了单？（没有审批就不能用？还是客户内部合规要求？）

**假设深挖后发现**：客户真正需要的不是一个完整的工作流引擎，而是**「在系统内留痕审批结果，满足合规审计要求」**。

### 3.2 MVP方案设计

**方案A：产品化MVP（4周，~30人天）**
- 审批单创建（表单 + 附件）
- 单级审批（提交 → 审批人通过/驳回）
- 审批结果记录与导出
- 邮件通知（接入现有邮件服务）

**方案B：Wizard of Oz MVP（1周，~5人天）**
- 前端：简单的审批表单页面
- 后端：人工处理审批流转（客服/运营操作后台）
- 用户感知是一个产品功能，实际是人工驱动

**选择方案A**——因为B2B客户对「系统化」有硬性要求，人工方案无法满足合规审计需求。

### 3.3 范围界定结果

```
MVP 范围（v0.1，4周）：
├── ✅ 审批单创建（表单 + 文件附件）
├── ✅ 指定审批人（单级，手动选择）
├── ✅ 审批操作（通过 / 驳回 + 原因）
├── ✅ 审批结果通知（邮件）
├── ✅ 审批记录列表与导出
├── ⏭️ 多级审批（v0.2，6周后）
├── ⏭️ 审批模板（v0.3）
├── ⏭️ 审批催办与超时升级（v0.4）
└── ❌ 审批条件引擎（不在本季度规划）
```

### 3.4 与销售团队对齐

关键沟通话术：
> 「MVP版本的审批功能可以满足XX客户的合规留痕需求。多级流转和自定义规则会在后续版本迭代。建议在售前沟通中明确区分『当前可用』和『路线图中的能力』，避免客户预期偏差。」

## 四、需求裁剪的六大技巧

### 技巧1：用「假装已经做了」替代「真的做」
- 不需要做完整的权限系统 → 用固定角色替代
- 不需要做搜索功能 → 用Ctrl+F替代（早期Notion就这么干过）
- 不需要做数据分析 → 导出Excel让用户自己分析

### 技巧2：人工替代自动化
- 不需要做智能推荐 → 编辑手动推荐
- 不需要做审批流引擎 → 运营手动操作
- 不需要做支付系统 → 销售手动收款后后台开通

### 技巧3：限制维度而非砍功能
- 不做「所有页面的暗色模式」→ 只做主页面
- 不做「所有文件格式预览」→ 只支持PDF和图片
- 不做「所有审批类型」→ 只支持费用审批

### 技巧4：借用现有能力
- 不需要做通知系统 → 接入企业微信/钉钉
- 不需要做文件存储 → 用OSS/七牛云
- 不需要做用户认证 → 用Auth0或微信登录

### 技巧5：约定优于配置
- 不需要做灵活配置 → 用最佳实践默认值
- 用户不需要选择 → 替他们选好
- 「给用户他们需要的，而不是他们想要的」

### 技巧6：时间盒约束
- 设定固定时间盒（如2周），时间到就发布
- 没做完的功能不延期，直接砍到下个版本
- 倒逼团队做优先级决策

## 五、面试常见追问

### 追问1：「如果销售说客户必须要完整功能才签单呢？」

回答框架：
1. **验证紧迫性**：是真的签单前置条件，还是nice-to-have？
2. **评估替代方案**：能否用临时方案（如人工+Excel）过渡？
3. **量化机会成本**：为这一个客户做3个月，其他客户的需求怎么办？
4. **分级处理**：
   - 如果是头部客户（收入占比>15%）→ 可考虑定制化交付
   - 如果是标准客户 → 用MVP + 承诺路线图推进签约

### 追问2：「MVP做完后发现假设不成立怎么办？」

这不是失败，这是MVP最大的成功——用最低成本避免了更大的浪费。

应对步骤：
1. 复盘假设为什么不成立
2. 判断是否需要调整方向（Pivot）还是调整方案
3. 将学习成果转化为新的假设和新的MVP

### 追问3：「如何说服工程师接受裁剪后的范围？」

工程师最怕的是「裁剪=偷懒=产品质量差」。

沟通策略：
1. **用假设驱动解释**：不是偷懒，是先验证需求真伪
2. **承诺技术方案前瞻设计**：MVP架构要能扩展，不是技术债
3. **定义明确的成功标准**：MVP发布后看什么指标决定是否继续投入
4. **分享行业案例**：Dropbox用视频验证需求、 Airbnb用人工验证房源

## 六、标准回答框架

```
1. 需求澄清（1分钟）
   - 追问真实问题（不是表面需求）
   - 明确约束条件（时间、资源、质量底线）

2. 假设识别（1分钟）
   - 列出核心假设
   - 区分「必须验证」和「可以假设成立」

3. 范围界定（2分钟）
   - MoSCoW分级
   - 代价-价值排序
   - 明确Won't Have（这条最重要）

4. 替代方案（1分钟）
   - 哪些用非产品方式替代？
   - 哪些用现有能力拼装？

5. 预期管理（1分钟）
   - 与销售/客户对齐预期
   - 定义MVP→完整版的迭代路线
   - 设定MVP成功的衡量标准
```

## 七、关键收获

1. **MVP裁剪的核心是「想清楚要验证什么」**——不是砍功能，是聚焦假设验证
2. **Won't Have比Must Have更重要**——明确不做什么，才能保证关键的事情被做完
3. **替代方案体现PM的创造力**——最好裁剪不是砍掉，而是找到更低成本的实现方式
4. **MVP的范围管理是持续过程**——不是一次性决策，而是每个迭代点都要重新评估
5. **沟通预期比裁剪本身更难**——技术上界定范围容易，让销售、老板、客户接受这个范围才是真正的挑战
