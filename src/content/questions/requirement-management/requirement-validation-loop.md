# 需求验证循环：从假设到证伪

## 面试场景

> 面试官：业务方提了一个需求，要在App首页加一个"智能推荐"模块，预计开发周期6周。你作为PM，在投入开发之前，会怎么验证这个需求是否值得做？

---

## 核心概念解析

### 什么是需求验证

需求验证不是"确认需求描述是否清楚"，而是在资源投入之前，**验证需求背后的假设是否成立**。

很多PM的工作流程是：

```
收到需求 → 写PRD → 评审 → 开发 → 上线 → 发现没人用
```

正确的流程应该是：

```
收到需求 → 拆解假设 → 设计验证 → 证伪/证实 → 决定是否继续
```

### 假设驱动的思维方式

**每一个需求背后都隐藏着假设。** PM的核心能力之一就是把隐性假设变成显性假设，然后逐一验证。

举例——"首页加智能推荐模块"背后至少隐藏着这些假设：

| 假设类型 | 具体假设 | 验证方式 |
|----------|----------|----------|
| **用户假设** | 用户在首页有"不知道看什么"的困扰 | 用户访谈、行为数据 |
| **价值假设** | 推荐内容能解决用户的发现困难 | A/B测试、原型测试 |
| **行为假设** | 用户会点击推荐内容 | 点击率数据、眼动测试 |
| **商业假设** | 推荐带来的点击能转化为业务收益 | 漏斗分析、收入预测 |
| **技术假设** | 推荐算法的效果足够好 | 离线评估、灰度测试 |

## 验证框架：Build-Measure-Learn 循环

Eric Ries在《精益创业》中提出的核心循环，同样适用于需求验证：

```
    Build（构建最小验证物）
        ↓
    Measure（测量用户行为）
        ↓
    Learn（学习并决策）
        ↓
    Pivot or Persevere（转向或坚持）
```

### 按验证成本从低到高的五层验证

**第一层：问题验证（成本：≈0）**

在做任何方案之前，先验证问题是否真实存在。

方法：
- **用户访谈**：直接问5-8个目标用户，是否有这个困扰
- **数据回溯**：查看现有数据，是否能看到行为迹象
- **竞品分析**：竞品是否有类似功能？如果没有，为什么？

**判断标准**：至少3/5的访谈用户主动提到这个问题，或数据中有明显的行为信号。

**第二层：方案验证——纸面原型（成本：1天）**

用纸面或低保真原型测试方案方向。

方法：
- 画出手绘原型或Figma简单线框
- 找5个用户做"你会怎么用"测试
- 观察用户是否理解功能目的

**判断标准**：用户能在无引导下说出这个功能的用途。

**第三层：需求验证——Fake Door测试（成本：2-3天）**

也叫"伪装门测试"或"冒烟测试"。在前端放一个入口，但后端不实现功能。

方法：
- 在首页加一个"智能推荐"入口（按钮/卡片）
- 用户点击后显示"功能开发中"或弹出简单的兴趣收集表单
- 统计点击率

**判断标准**：点击率超过预设阈值（如5%），说明用户有兴趣。

**关键提醒**：Fake Door测试要控制时间（<1周），避免长期欺骗用户。

**第四层：价值验证——MVP（成本：1-2周）**

构建最小可用版本，用真实用户验证价值。

方法：
- 实现功能但用最简单的逻辑（如热门内容排序而非AI推荐）
- 灰度发布给5-10%用户
- 对比实验组和对照组的核心指标

**判断标准**：核心指标（点击率、停留时长等）有统计显著的提升。

**第五层：规模化验证——A/B测试（成本：完整开发周期）**

在功能完整开发后，正式进行A/B测试。

方法：
- 随机分50/50实验组/对照组
- 设定主要指标和次要指标
- 确定实验时长（至少覆盖一个完整周期）

**判断标准**：主要指标提升达到预期且统计显著。

## 实战案例：某阅读App的"书摘分享"功能验证

### 背景
运营团队提出："我们应该加一个书摘分享功能，用户可以选中文字生成精美图片分享到社交媒体，带来新用户。"

### 验证过程

**第1周：问题验证**
- 用户访谈8人，5人表示"有分享欲望但截屏不方便"
- 数据显示：用户平均每阅读3本书会截屏1次
- **结论：问题成立，用户确实有分享需求**

**第2周：方案验证**
- 制作用户调研问卷：你更倾向哪种分享方式？（图片/链接/文字）
- 用Figma做3个方案原型，找5个用户测试
- **结论：精美图片方案最受欢迎，但用户更在意"操作便捷"而非"图片精美"**

**第3周：Fake Door测试**
- 在阅读页选中文字后，增加"生成分享卡片"入口
- 点击后弹出"功能即将上线，提前体验请留邮箱"
- **结果：点击率12%，远超5%的阈值，有230人留了邮箱**

**第4-5周：MVP开发**
- 最简实现：选中文字 → 生成简单卡片（纯色背景+文字） → 保存到相册
- 灰度发布10%
- **结果：分享卡片生成率8%，其中32%的用户实际保存了图片。分享卡片功能的次日留存提升了3%**

**最终决策**：投入完整开发资源，优化卡片设计，增加社交媒体直发功能。

### 关键决策点
- 如果第1周问题验证失败 → **终止，不做**
- 如果第3周Fake Door点击率<3% → **重新评估方案**
- 如果第5周MVP数据不显著 → **分析原因，可能pivot方向**

## 面试常见追问

### Q1：如果业务方催着上线，没时间做验证怎么办？

**回答思路：**

1. **理解紧迫性的来源**：是市场竞争压力？领导KPI？还是估计不足？
2. **用最小成本验证替代"不验证"**：即使只有2天，也能做一个Fake Door测试
3. **提出分阶段上线方案**：先上基础版（1周），验证后再迭代高级版（4周）
4. **明确风险**：如果跳过验证直接开发6周，风险是"6周后可能发现做的方向不对"

**话术参考**："我理解时间紧迫，我建议我们用1天做一个最简单的验证——在首页加个入口看看点击率。如果数据好，我们可以更有信心地投入开发；如果数据不好，我们省下了6周的开发资源去做更有价值的事。这1天的投入是值得的。"

### Q2：验证结果和业务方的直觉矛盾怎么办？

**处理原则：数据 > 直觉，但尊重业务方的经验。**

1. **不带偏见地呈现数据**：只展示事实，不急于下结论
2. **理解直觉的来源**：业务方的直觉可能基于你没有的行业经验
3. **检查验证方法的局限性**：Fake Door点击率低，可能是因为入口位置不对，而不是需求不成立
4. **设计更精准的验证**：如果对结果有争议，设计一个更有说服力的验证实验
5. **小步快跑，让事实说话**：先做小版本上线，让真实数据说服

### Q3：哪些需求不需要验证？

- **合规性需求**：法规要求的功能（如隐私协议、实名认证）不需要验证，必须做
- **技术债偿还**：影响系统稳定性的关键修复
- **已有数据的直接延伸**：如已验证的核心功能的优化迭代
- **竞品已充分验证的标准做法**：如电商的购物车功能（但你的实现方式仍可验证）

## 标准回答框架

当被问到"如何验证需求"时：

```
1. 拆解假设
   → 这个需求背后有哪些用户假设、价值假设、商业假设？
   → 哪个假设风险最高？（先验证风险最高的）

2. 选择验证方法（按成本从低到高）
   → 问题验证（访谈/数据）→ 成本最低，先做
   → 方案验证（原型测试）→ 验证方向对不对
   → Fake Door测试 → 验证用户兴趣
   → MVP → 验证真实价值
   → A/B测试 → 验证最终效果

3. 设定判断标准
   → 每一层验证前，先定好"通过标准"
   → 不要先看数据再定标准（确认偏差）

4. 决策
   → 通过 → 进入下一层验证或正式开发
   → 不通过 → 分析原因，pivot或终止
   → 模糊 → 设计更精准的验证实验

5. 沉淀知识
   → 无论结果如何，记录验证过程和发现
   → 这些知识对未来决策有参考价值
```

---

## 关键收获

1. **每个需求都是假设，不是事实**——PM的第一反应应该是"有什么证据支持这个假设"，而不是"PRD怎么写"。
2. **验证成本应与风险成正比**——小需求简单验证，大需求多层验证。6周开发的需求至少需要1周的验证。
3. **Fake Door是最高性价比的需求验证方法**——2-3天的投入可能省下数周的开发资源。
4. **验证不是为了否定需求，而是为了做对的需求**——验证结果支持就加速，不支持就转向，两种结果都有价值。
5. **"先上线再迭代"不等于"不验证"**——分阶段上线本身就是一种验证策略，关键是每阶段都有明确的判断标准。
