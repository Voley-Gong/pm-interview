# PLG产品驱动增长 vs SLG销售驱动增长

## 面试场景

> 面试官：你负责一款企业协作SaaS产品，目前主要靠销售团队获客（SLG），CAC（获客成本）持续上升，转化周期长达3个月。CEO要求你研究PLG（Product-Led Growth）转型。你怎么评估你们产品是否适合PLG？如果适合，转型路径是什么？如果不适合，为什么？

这是一道高级产品策略题，考察PM对增长模型底层逻辑的理解，以及从战略到执行的分析能力。

---

## 一、PLG与SLG的本质区别

### 1.1 定义辨析

**PLG（Product-Led Growth）**：产品本身是主要的获客、转化、扩展工具。用户通过自服务（self-serve）体验产品价值，无需销售人员介入。

典型代表：Slack、Notion、Figma、Zoom、Calendly

**SLG（Sales-Led Growth）**：销售团队是主要的获客和转化引擎。产品需要销售人员演示、讲解、谈判才能完成签约。

典型代表：Salesforce、Oracle、SAP、Workday

### 1.2 核心维度对比

| 维度 | PLG | SLG |
|------|-----|-----|
| 获客方式 | 自服务注册、口碑传播、病毒循环 | 销售外呼、展会、ABM（客户营销） |
| 首次体验 | 免费试用/Freemium，自助上手 | 销售演示、POC（概念验证） |
| 决策路径 | 用户→团队→采购（自下而上） | 采购→IT→业务（自上而下） |
| 转化周期 | 天/周 | 月/季度 |
| CAC | 低（$50-$500） | 高（$5,000-$50,000+） |
| 扩展方式 | 用量自然增长、交叉销售 | 客户成功团队驱动增购 |
| LTV/CAC | >3（因为CAC低） | >3（因为LTV高，但需要更长时间回本） |
| 关键指标 | 激活率、病毒系数、自然增长占比 | 销售管道、赢单率、合同金额 |
| 产品要求 | 极低的认知门槛、快速达到Aha Moment | 功能完整、集成能力强、安全合规 |

### 1.3 不是二选一，而是光谱

```
纯PLG ◄──────────────────────────────────────► 纯SLG
  │              │              │              │
 Calendly      Slack       HubSpot      Salesforce    SAP
 (自助即用)    (自下而上)   (混合模式)    (自上而下)   (完全定制)
```

大多数成功的SaaS公司最终都走向**混合模式**——以PLG获取用户，以SLG扩展大客户。

## 二、PLG适合性的评估框架

### 2.1 PLG可行性评估矩阵

从五个维度评估产品是否适合PLG：

#### 维度1：产品复杂度

```
低复杂度 ──────────────────────────── 高复杂度
  │                                          │
  极适合PLG                              极不适合PLG
  （用户3分钟内能理解价值）        （需要专家配置才能使用）
```

**关键问题：**
- 用户在没有人指导的情况下，能否在10分钟内完成注册并体验到核心价值？
- 产品是否需要复杂的数据迁移/系统集成才能开始使用？
- 用户是否需要培训才能使用基础功能？

#### 维度2：终端用户属性

| 用户类型 | PLG适合度 | 原因 |
|---------|----------|------|
| 个人贡献者（IC） | ⭐⭐⭐⭐⭐ | 可以自主决策试用新工具 |
| 团队Leader | ⭐⭐⭐⭐ | 可以为团队引入工具 |
| 部门总监 | ⭐⭐⭐ | 需要ROI论证 |
| C-level/IT | ⭐ | 需要正式采购流程 |

**PLG的最佳起点是个人贡献者或小团队——他们有痛感、有试用权、有传播力。**

#### 维度3：价值传递速度（Time to Value）

```
T < 5分钟    → 极适合PLG（如Calendly发一个链接就能用）
T < 1小时    → 适合PLG（如Notion创建第一个页面）
T < 1天      → 可以PLG，但需要好的引导（如Slack建频道拉人）
T > 1周      → PLG困难，可能需要SLG辅助
T > 1月      → 必须SLG（需要POC、集成、数据迁移）
```

#### 维度4：病毒性（Virality）

PLG的增长引擎之一是病毒循环——用户使用产品时自然带来新用户：

- **天然病毒性**：产品本身需要协作（如Slack、Figma、Google Docs）
- **人为病毒性**：通过邀请机制设计（如Dropbox的推荐奖励）
- **内容病毒性**：用户产出内容可被外部消费（如Notion公开页面）

**关键指标：K因子 = 邀请数 × 转化率。K>1意味着自然增长。**

#### 维度5：单位经济学

```
PLG经济模型：
   CAC: $50-500（低）
   LTV: $500-5,000/年
   回本周期: <6个月
   → 需要大量用户来覆盖固定成本

SLG经济模型：
   CAC: $5,000-50,000（高）
   LTV: $50,000-500,000/年
   回本周期: 12-18个月
   → 需要高客单价来覆盖销售成本
```

### 2.2 评估总结表

| 你的产品特征 | PLG适合度 |
|-------------|----------|
| 简单、自服务、快速上手 | ✅ 强烈推荐 |
| 需要协作、天然病毒性 | ✅ 强烈推荐 |
| 高客单价、长决策链 | ❌ 不适合纯PLG |
| 需要深度集成/数据迁移 | ❌ 不适合 |
| 中等复杂度、可分阶段上手 | ⚠️ 可以混合模式 |

## 三、PLG转型的实战路径

### 3.1 从SLG向PLG转型的四个阶段

#### Phase 0：可行性评估（1-2月）

**关键动作：**
1. 挑选一个"最容易自服务化"的场景/功能模块
2. 设计Free Trial体验流程
3. 分析现有客户中，是否有"自服务成功"的先例

**决策门：** 如果没有任何客户在无销售介入的情况下成功使用产品，先做产品简化，不要急于做PLG。

#### Phase 1：自服务基建（2-3月）

**核心交付物：**

| 交付物 | 说明 |
|--------|------|
| Freemium/Free Trial | 降低尝试门槛 |
| Onboarding流程 | 引导用户在10分钟内达到Aha Moment |
| 自服务文档 | 帮助中心、视频教程、模板库 |
 |产品内分析 | 追踪用户行为，识别激活点和流失点 |
| 自动化转化 | In-app引导、邮件 nurture、触发式升级提示 |

**关键指标：**
- 注册→激活率（目标 >40%）
- 激活→付费转化率（目标 >5%）
- Time to Value（目标 <1天）

#### Phase 2：增长循环构建（3-6月）

**两种核心增长循环：**

**循环A：用户→邀请→用户（病毒循环）**
```
用户使用产品 → 邀请协作者 → 协作者注册 → 成为用户 → 继续邀请
```
优化点：降低邀请摩擦（一键邀请）、设计协作场景（共享文档/项目）。

**循环B：内容→发现→注册（内容循环）**
```
用户创建内容 → 内容被公开/分享 → 新用户发现 → 注册使用 → 创建更多内容
```
优化点：公开页面SEO、可分享的内容模板、内容发现机制。

#### Phase 3：PLG→SLG衔接（持续）

PLG不是完全取代销售，而是**改变销售的角色**：

| 传统SLG销售 | PLG时代的销售 |
|------------|-------------|
| 主动外呼寻找线索 | 基于产品用量识别高价值线索 |
| 从零演示产品功能 | 用户已经在用，讨论升级需求 |
| 长周期POC和谈判 | 产品数据驱动的精准销售 |
| CAC高 | CAC降低60-80% |

**关键：从"猎人"变成"农夫"——不再追逐猎物，而是培育已经冒头的用户。**

### 3.2 常见转型陷阱

| 陷阱 | 表现 | 解决方案 |
|------|------|---------|
| 只加了Free Trial就以为做了PLG | 注册后无人引导，激活率<5% | PLG的核心是Onboarding和激活，不是免费 |
| 产品复杂度没有降低 | 免费版用户注册后3分钟流失 | 先做减法，找最小可用场景 |
| 销售团队抵触 | 销售担心被替代 | 重新定义销售角色，设计新的激励机制 |
| 没有数据基建 | 不知道用户在产品里做了什么 | 先建设用户行为追踪和分析能力 |
| 同时服务两种模式力不从心 | PLG和SLG互相干扰 | 分设团队/品牌，或分阶段推进 |

## 四、实战案例

### 案例A：Slack的PLG经典路径

```
1. 找到最小场景：团队内部沟通（替代邮件/IM）
2. 降低门槛：免费版功能够用、注册即用
3. 天然病毒性：拉同事进频道 = 自然邀请
4. 用量扩展：从一个小团队扩展到整个公司
5. 销售介入：当企业规模/需求超出免费版时，自然升级
```

**Slack的洞见：不要试图"卖"产品，让产品自己"证明"自己。**

### 案例B：某B端HR SaaS的PLG转型失败

```
背景：传统SLG模式，客单价$20,000+，销售周期4个月
转型决策：CEO要求做PLG转型
执行：推出免费版，开放注册
结果：
- 注册量高但激活率<3%（产品太复杂）
- 免费用户无法自己完成配置
- 销售团队线索质量下降（免费用户不是决策人）
- 6个月后取消免费版，回归SLG
失败原因：产品复杂度和用户类型根本不适合PLG
```

**教训：PLG不是"加个免费版"这么简单，它需要产品本身的自服务基因。**

## 五、面试常见追问

### 追问1："PLG模式下，如何防止免费用户'白嫖'？"

**回答框架：**
1. 设计好Freemium的功能边界：免费版解决基础问题，付费版解决规模化/高阶问题
2. 用用量限制（而非功能限制）驱动升级：当用户用量到达天花板时自然付费
3. 设计PQL（Product Qualified Lead）机制：当用户行为表明有高价值需求时，销售精准介入

### 追问2："如果竞品用PLG模式抢你的用户怎么办？"

**回答框架：**
1. 评估竞品PLG的实际效果（注册量≠激活量≠付费量）
2. 如果竞品PLG真的有效：分析他们做对了什么，我们能否借鉴
3. 如果我们的产品不适合PLG：加强SLG优势——深度服务、定制能力、行业理解
4. 考虑"中间路线"：提供免费体验版但不完全自服务，用混合模式应对

### 追问3："PLG和MLG（Marketing-Led Growth）有什么区别？"

**回答框架：**
> PLG的核心是产品本身驱动获客和转化——用户通过使用产品来"发现"价值。MLG的核心是市场活动驱动获客——用户通过内容、SEO、广告来"了解"价值，然后进入产品。PLG的转化发生在产品内，MLG的转化发生在漏斗中。最理想的状态是MLG负责"引流"，PLG负责"激活和扩展"。

## 六、标准回答框架

```
1. 定义澄清（30秒）
   "PLG是产品驱动增长，核心是自服务和病毒循环。
   SLG是销售驱动增长，核心是专业销售和高客单价。
   大多数成功SaaS最终走混合模式。"

2. 可行性评估（60秒）
   "我会从5个维度评估我们产品的PLG适合性：
   产品复杂度、终端用户、价值传递速度、病毒性、单位经济学。
   关键问题是：用户能否在无人指导下10分钟内体验到价值？"

3. 转型路径（60秒）
   "如果评估通过，我会分四步走：
   Phase 0可行性评估 → Phase 1自服务基建 →
   Phase 2增长循环 → Phase 3 PLG到SLG衔接。
   核心是先做产品简化，再加免费版。"

4. 风险提示（30秒）
   "最大的风险是把PLG等同于'加个免费版'。
   PLG的本质是产品本身的自服务能力。
   如果产品太复杂，先做减法而不是先做免费。"

5. 案例佐证（30秒）
   用Slack的成功路径或某个PLG失败案例说明。
```

## 七、关键收获

| 洞察 | 说明 |
|------|------|
| PLG不是策略选择，是产品属性 | 产品复杂度、协作性、自服务能力决定了PLG可行性 |
| Time to Value是核心指标 | TTV越短，PLG越可行 |
| PLG不排斥销售 | PLG改变的是销售的角色和时机，不是消灭销售 |
| 混合模式是终局 | PLG获客 + SLG扩展大客户 = 最优组合 |
| 单位经济学是底线 | 如果PLG的LTV/CAC < 3，模型不成立 |
| 转型是产品工程 | 不是加个免费版，而是重新设计产品的上手体验 |
