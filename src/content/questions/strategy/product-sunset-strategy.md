# 产品下线与日落策略

## 面试场景

> 面试官：「你负责的一款产品线，连续两个季度营收下滑30%，市场份额被竞品蚕食，技术债高筑。CEO要求你在6个月内完成产品下线。你会怎么做？」

产品下线（Product Sunset / End of Life）是产品策略中极少被讨论、但极其考验综合能力的高难度场景。它需要同时管理用户情绪、团队士气、商业利益和法律风险，是高级PM面试中的经典"压力测试"题。

---

## 核心概念：产品下线的战略意义

### 为什么下线比创建更难

| 维度 | 创建产品 | 下线产品 |
|------|---------|---------|
| 心态 | 激情、创造、增长 | 割舍、收缩、承认失败 |
| 用户关系 | 从零建立信任 | 打碎已有信任 |
| 团队 | 招兵买马士气高涨 | 人心浮动、不确定性 |
| 决策信号 | 正向指标（增长、营收） | 多维度综合判断 |
| 风险 | 做错了最多没做成 | 做错了面临法律诉讼、品牌危机 |

### 产品下线的三种类型

**类型一：主动日落（Proactive Sunset）**
- 产品仍盈利但战略方向调整
- 预见到市场趋势变化，提前退出
- 例：Google关停Hangouts以聚焦Meet

**类型二：被动淘汰（Reactive EOL）**
- 产品持续亏损，无法维持
- 市场需求消失或技术被颠覆
- 例：Flash Player被HTML5全面替代后下线

**类型三：整合重组（Consolidation）**
- 产品被合并到更大的产品线中
- 功能被其他产品吸收
- 例：Microsoft将To-Do整合进Microsoft 365

## 框架：产品下线的六阶段模型

### Phase 1：决策评估——该不该下线？

在决定下线之前，必须进行系统性的评估：

**评估维度**：
| 维度 | 关键问题 | 量化指标 |
|------|---------|---------|
| 商业价值 | 营收趋势如何？维护成本占比？ | 营收、毛利率、维护成本/营收比 |
| 用户价值 | 还有多少活跃用户？用户满意度趋势？ | DAU/MAU、NPS、用户投诉率 |
| 战略对齐 | 产品是否仍服务于公司核心战略？ | 战略对齐评分（1-5） |
| 替代方案 | 是否有更好的替代品？迁移成本？ | 替代品成熟度、迁移工时 |
| 机会成本 | 团队投入在这里的机会成本？ | 人力成本 × 可投入其他项目的预期收益 |
| 法律风险 | 是否有合同义务、数据保留要求？ | SLA条款、GDPR/数据合规要求 |

**决策矩阵**：

```
         高用户价值
            │
    迁移    │   维持
    转型    │   优化
  ─────────┼─────────
    逐步    │   立即
    下线    │   下线
         低用户价值
```

**关键原则**：不要因为"维护烦"就下线。要区分"产品不行了"和"我们不想维护了"。

### Phase 2：利益相关者分析——谁会受影响？

产品下线涉及多方利益，需要提前梳理并制定沟通策略：

| 利益相关者 | 核心关切 | 沟通策略 |
|-----------|---------|---------|
| 现有付费用户 | 我的投资怎么办？数据怎么办？ | 提前6个月通知、提供迁移方案、退款政策 |
| 免费用户 | 我的数据/内容怎么办？ | 导出工具、数据删除通知 |
| 内部团队 | 我的工作怎么办？技能是否可迁移？ | 1:1沟通、转岗安排、技能培训 |
| 合作伙伴/渠道 | 分成怎么办？合同怎么处理？ | 合同审核、过渡期安排 |
| 客服/运营 | 如何应对用户咨询？ | FAQ文档、统一话术、escalation流程 |
| 媒体/公众 | 品牌形象影响？ | PR策略、创始人/CEO公开信 |
| 投资人/董事会 | 财务影响？战略调整？ | 财务测算、战略Narrative |

### Phase 3：迁移方案设计——用户怎么走？

**迁移方案的设计原则**：
1. **数据可导出**：提供标准格式（CSV/JSON）的数据导出工具
2. **替代品引导**：推荐公司内部或外部替代产品，提供对比指南
3. **过渡期支持**：在迁移期间保持原有功能可用
4. **经济补偿**：对付费用户按比例退款或赠送替代产品权益

**迁移路径示例**：
```
通知期（3个月）→ 过渡期（2个月）→ 功能冻结（1个月）→ 数据导出期（1个月）→ 正式关停
```

**关键细节**：
- 导出数据的格式要通用、文档要清晰
- 提供API让有能力的企业用户自行迁移
- 设置专门的迁移支持团队，响应时间<4小时

### Phase 4：时间线规划——什么时候做什么？

**推荐时间线（以6个月为例）**：

| 时间 | 阶段 | 关键动作 |
|------|------|---------|
| T-6月 | 内部决策 | 完成评估、获得CEO批准、组建下线项目组 |
| T-5月 | 内部沟通 | 通知团队、合作伙伴、客服培训 |
| T-4月 | 用户通知 | 发送第一轮通知邮件、公告、FAQ上线 |
| T-3月 | 停止新签 | 不再接受新付费用户、关闭注册 |
| T-2月 | 功能冻结 | 停止新功能开发、只修P0 bug、开放数据导出 |
| T-1月 | 最终通知 | 二次提醒、推送导出、关闭支付 |
| T-0 | 正式关停 | 服务下线、官网更新为关停说明页 |
| T+1月 | 收尾 | 数据归档（合规保留期）、复盘报告、团队安置完成 |

### Phase 5：执行与风险管控——怎么平稳落地？

**风险管控清单**：

| 风险 | 可能性 | 影响 | 应对方案 |
|------|:---:|:---:|---------|
| 用户大规模投诉/舆情 | 高 | 高 | 准备统一回应话术、KOL提前沟通 |
| 法律纠纷（合同违约） | 中 | 高 | 法务提前审核、准备和解方案 |
| 数据丢失纠纷 | 中 | 极高 | 多重备份、延长导出期、法律免责条款 |
| 团队离职潮 | 中 | 中 | 提前安排转岗、保持透明沟通 |
| 竞品恶意挖用户 | 高 | 中 | 准备竞品对比材料、主动推荐替代品 |

**舆情管理**：
- **不要说**："因为没人用所以关了"
- **应该说**："为了更好地聚焦核心产品，为用户提供更优质的服务，我们决定在X月停止运营Y产品"
- **基调**：感恩、负责、面向未来

### Phase 6：复盘与知识沉淀——我们能学到什么？

产品下线后的复盘是组织学习的重要资产：

**复盘维度**：
1. **产品层面**：为什么这款产品最终没有成功？是市场需求判断错误、执行问题、还是外部环境变化？
2. **过程层面**：下线决策是否及时？是否拖了太久？
3. **管理层面**：团队安置是否妥当？用户反响如何？
4. **资产层面**：可复用的技术资产、用户洞察、运营经验有哪些？

**产出物**：
- 产品死因分析报告
- 用户迁移效果报告
- 团队安置结果
- 可复用资产清单
- 类似项目的"避坑指南"

## 标准回答框架

面对"如何下线产品"的面试题，按以下结构组织：

```
1. 先判断该不该下线（60秒）
   → 展示评估框架：商业价值、用户价值、战略对齐、机会成本
   → 体现你不会轻易下线，但也不回避必要决策

2. 利益相关者分析（60秒）
   → 系统梳理各方利益和沟通策略
   → 特别关注付费用户和内部团队

3. 迁移方案与时间线（90秒）
   → 六阶段模型的展开
   → 具体到时间节点和关键动作

4. 风险管控（30秒）
   → 舆情、法律、数据、团队四大风险
   → 每个风险的应对方案

5. 价值观表达（30秒）
   → 好的下线是对用户的最后一次负责
   → 产品经理要有"善终"的能力
```

## 面试常见追问

**Q1：如果CEO要求立即下线，但用户需要6个月迁移期，你怎么平衡？**

A：核心是**分阶段下线**而非一刀切：
- 立即停止新用户注册和付费
- 对存量用户提供合理过渡期（至少90天）
- 向CEO展示立即关停的舆论风险和法律风险
- 提出折中方案：保留核心服务运行（最小成本），但停止所有新功能开发

**Q2：如何判断是"下线"还是"重构转型"？**

A：用三个测试：
- **用户价值测试**：用户还需要这个核心功能吗？如果需要，考虑重构；如果不需要，下线
- **经济测试**：重构的投入产出比是否合理？如果重构成本≈重新做一个产品，不如下线重来
- **团队能力测试**：团队是否有能力做大幅重构？如果核心人员已经流失，重构风险极高

**Q3：产品下线时，用户数据如何处理？**

A：四个原则——
- **合规优先**：GDPR、个保法等要求的数据保留/删除义务必须遵守
- **用户可控**：提供数据导出和删除选项，让用户自行决定
- **过渡保留**：服务关闭后，数据至少保留30-90天供用户导出
- **安全销毁**：过渡期结束后，按安全标准彻底销毁数据，留存销毁证明

**Q4：产品下线对团队士气打击很大，怎么管理？**

A：三点——
- **透明沟通**：尽早、诚实地和团队沟通决策逻辑和后续安排
- **体面告别**：举办"产品告别仪式"，肯定团队贡献，记录产品的高光时刻
- **积极安置**：提前规划成员的去向，优先内部转岗到核心产品线

## 实战案例

### 案例：Google Inbox 的日落

Google在2018年宣布关停Inbox（一款创新的邮件客户端），关键做法：
1. **提前4个月通知**，给用户充分的过渡时间
2. **明确替代方案**：将Inbox的创新功能逐步迁移到Gmail
3. **功能迁移指南**：详细对比Inbox功能在Gmail中的对应位置
4. **保持Gmail同步**：Inbox底层就是Gmail，所以数据无感知迁移
5. **渐进式关停**：先停止新功能开发 → 停止支持 → 正式关闭

**启示**：好的日落策略不是"突然死亡"，而是"尊严退休"——用户感到被尊重，产品价值被延续。

## 关键收获

1. **产品下线是PM的高阶能力**：它同时考验商业判断、用户同理心、风险管理和团队领导力
2. **六阶段模型**：评估决策 → 利益相关者分析 → 迁移方案 → 时间线规划 → 风险管控 → 复盘沉淀
3. **核心原则：用户体面优先**：即使是关停产品，也要给用户一次"被尊重的告别"
4. **提前规划是关键**：下线决策拖得越久，沉没成本越高，转型空间越小
5. **复盘是最有价值的遗产**：一个产品的失败教训，可能比成功经验对组织更有价值
