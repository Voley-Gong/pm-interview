## 面试场景

> 面试官：「你所在的公司是一个在线教育平台，目前的视频播放器使用的是第三方SDK。但最近出现了两个问题：第一，第三方SDK的定制能力无法满足业务需求（如互动字幕、知识点标记）；第二，每年许可证费用80万。CTO倾向于自研播放器，你作为PM怎么评估这个决策？」

这道题考察的是高级PM的**战略决策能力**——能否跳出「能不能做」的技术视角，从核心竞争力、资源配置、机会成本、风险评估等多维度系统分析 Build-Buy-Partner 决策。

---

## 一、核心概念：Build-Buy-Partner 三角框架

### 1.1 三种策略的本质

| 策略 | 本质 | 适用场景 | 核心权衡 |
|-----|------|---------|---------|
| **Build（自建）** | 投入资源自己开发 | 该能力是核心竞争力 / 市场上无成熟方案 | 高前期投入，长期可控，机会成本大 |
| **Buy（采购）** | 付费购买现成产品/服务 | 市场有成熟方案 / 不是核心竞争力 | 低启动成本，快速上线，长期依赖 |
| **Partner（合作）** | 与他方共建共享 | 需要互补能力 / 分担风险 | 中等投入，利益分配复杂 |

### 1.2 决策的底层逻辑

```
自建的价值 = 能力差异化带来的竞争优势
自建的成本 = 开发成本 + 维护成本 + 机会成本 + 时间成本

当 自建价值 >> 自建成本 → Build
当 自建价值 ≤ 自建成本 → Buy or Partner
```

**最常见的错误**：只算开发成本，不算机会成本。一支10人的工程团队花6个月自研播放器，意味着这10人6个月不能做其他事情——这些「其他事情」的潜在价值就是机会成本。

## 二、决策框架：六维评估模型

### 维度1：核心竞争力对齐度

**关键问题**：这个能力是否直接构成你的竞争壁垒？

```
核心竞争力测试：
- 这个能力能直接影响用户选择我们而不是竞品吗？
- 如果去掉这个能力，我们的产品护城河会变弱吗？
- 这个能力是否难以被模仿？

高对齐度 → 倾向自建
低对齐度 → 倾向采购/合作
```

**在线教育案例**：
- 互动字幕、知识点标记 → **高对齐度**（这是教学体验差异化的关键）
- 视频编解码、CDN分发 → **低对齐度**（这些是基础设施能力）

### 维度2：市场上方案的成熟度

| 方案成熟度 | 代表场景 | 建议 |
|-----------|---------|------|
| 多个成熟竞品 | 支付网关、IMSDK | Buy，比价选型 |
| 少数方案，各有优劣 | 专业播放器、推荐引擎 | Buy + 定制层 |
| 市场空白或极度垂直 | 特定行业的审核引擎 | Build |
| 有能力但不开放 | 大厂内部工具 | Partner |

### 维度3：TCO（总拥有成本）对比

**3年TCO计算示例（视频播放器）：**

| 成本项 | 自建方案 | 第三方SDK |
|-------|---------|----------|
| 人力开发（10人×6月） | 120万 | 0 |
| 年度许可证 | 0 | 80万/年 |
| 维护人力（2人持续） | 48万/年 | 12万/年（对接） |
| 基础设施 | 20万/年 | 包含在许可费中 |
| **3年总成本** | **120 + 48×3 + 20×3 = 324万** | **80×3 + 12×3 = 276万** |
| 隐性成本 | 技术风险、学习曲线 | 定制受限、迁移成本 |

**注意**：自建第3年后边际成本下降，但如果考虑机会成本（这10人去做核心业务功能能带来多少收入），结果可能完全不同。

### 维度4：时间紧迫度

```
               紧迫                    不紧迫
核心竞争力  ┌──────────────────┬──────────────────┐
  高        │ 紧急自建+快速MVP   │ 按计划自建        │
            ├──────────────────┼──────────────────┤
  低        │ 采购快速上线       │ 评估长期ROI       │
            └──────────────────┴──────────────────┘
```

### 维度5：风险与依赖度

**采购/合作的风险：**
- 供应商价格上涨或停服（厂商锁定风险）
- 数据安全与合规风险
- 服务等级（SLA）不受控
- 技术栈绑定

**自建的风险：**
- 技术方案选型错误
- 团队能力不足导致延期
- 长期维护负担
- 技术栈老化需重写

### 维度6：组织战略一致性

- 公司是否有「平台化/自研化」的战略方向？
- CTO/CEO的技术偏好与战略愿景？
- 是否有自研能力的团队基础？
- 投资人对「研发投入占比」的期望？

**这不是政治正确问题——这是现实问题。** PM需要把战略意图纳入决策，而不是假装「纯理性分析」。

## 三、实战决策：在线教育视频播放器

### Step 1：能力解耦——不要把播放器当成一个整体决策

```
视频播放器能力拆解：
├── 核心播放（编解码、播放控制）→ 非核心竞争力 → Buy/Use开源
├── 基础UI（进度条、全屏、音量）→ 非核心竞争力 → 用开源框架
├── 互动层（字幕、知识点标记、答题弹窗）→ 核心竞争力！→ Build
├── 数据采集（播放行为埋点）→ 需要定制化 → Build
└── 分发与加速（CDN）→ 非核心竞争力 → Buy
```

### Step 2：混合方案设计

**推荐方案：开源底座 + 自建互动层**

| 层级 | 方案 | 成本 |
|-----|------|-----|
| 播放引擎 | 开源（Video.js / PLYR / Shaka Player） | 0（社区维护） |
| 互动能力 | 自研插件层（3人×3月） | 18万 |
| 数据采集 | 自研埋点（1人×1月） | 3万 |
| CDN分发 | 阿里云/腾讯云 | 按量付费 |
| **总投入** | **4人×3月 + 基础设施** | **约30万** |

**对比原方案**：节省80万/年许可证费，3年省210万+，同时获得完全的定制能力。

### Step 3：风险缓解

| 风险 | 缓解措施 |
|-----|---------|
| 开源框架维护中断 | 选活跃社区项目（commit频率 > 每周） |
| 互动层开发延期 | 先用CSS overlay做MVP验证 |
| 团队能力不足 | 招1个有播放器经验的前端 |
| 业务需求变化 | 插件化架构保证可替换性 |

### Step 4：分阶段执行计划

```
Phase 1（第1-4周）：MVP验证
- 基于Video.js搭建基础播放器
- 实现互动字幕POC
- 与现有第三方SDK并行运行对比

Phase 2（第5-8周）：核心功能开发
- 知识点标记交互
- 播放行为埋点
- 灰度迁移10%课程

Phase 3（第9-12周）：全量迁移
- 全量替换第三方SDK
- 下线旧SDK集成
- 复盘与数据对比
```

## 四、决策工具箱

### 4.1 Build-Buy-Partner 决策清单

回答以下8个问题，帮助形成判断：

1. 这个能力是否直接决定了用户为什么选我们？
2. 市场上是否有 ≥2 个成熟可用的方案？
3. 自建需要多久？这段时间不做这个功能的业务损失是多少？
4. 自建团队后续怎么安排？维护需要多少人？
5. 采购方案的年费占相关业务收入的百分比是多少？
6. 如果供应商明天涨价50%或停服，我们的Plan B是什么？
7. CEO/CTO的战略倾向是什么？为什么？
8. 这个决策的可逆性如何？（做错了能回退吗？）

### 4.2 评分矩阵模板

| 维度 | 权重 | Build得分(1-5) | Buy得分(1-5) | Partner得分(1-5) |
|-----|------|--------------|-------------|-----------------|
| 核心竞争力对齐 | 25% | 5 | 2 | 3 |
| 方案成熟度 | 15% | 2 | 5 | 3 |
| 3年TCO | 20% | 3 | 3 | 4 |
| 时间效率 | 15% | 2 | 5 | 3 |
| 风险可控性 | 15% | 4 | 2 | 3 |
| 组织战略 | 10% | 4 | 2 | 3 |
| **加权总分** | 100% | **3.45** | **3.05** | **3.15** |

## 五、面试常见追问

### 追问1：「如果CTO坚持自建但你觉得应该采购，怎么处理？」

1. **先理解CTO的理由**——技术负责人可能有你看不到的信息（团队成长需求、技术债考量、长期技术战略）
2. **用数据说话**——TCO对比、机会成本分析、市场方案对比
3. **提出混合方案**——往往不是非此即彼
4. **设定期望验证点**——如果决定自建，什么时候用什么指标判断是否成功
5. **最终尊重决策权**——PM是决策参与者不是决策者，如果CTO/CEO拍板了，做好执行

### 追问2：「开源方案算Build还是Buy？」

既不是传统Build也不是Buy——是一种独特的**Leverage**策略：

| 维度 | 纯自建 | 开源+定制 | 纯采购 |
|-----|-------|----------|-------|
| 初始成本 | 高 | 中 | 低 |
| 定制能力 | 最高 | 高 | 低-中 |
| 维护责任 | 全部 | 自己负责定制部分 | 供应商负责 |
| 社区支持 | 无 | 有 | 不适用 |
| 技术风险 | 全自担 | 部分共担 | 供应商风险 |

**开源+定制往往是中间最优解**，但要评估社区活跃度和维护能力。

### 追问3：「已经自建的系统，什么时候应该切换为采购？」

切换信号：
1. 自建系统的维护成本逐年上升，新增需求性价比下降
2. 市场上出现了成熟的标准化方案
3. 该能力从核心竞争力变为基础能力（如登录、支付）
4. 自建系统的技术债已影响业务发展速度
5. 团队维护意愿下降，人员流失风险高

## 六、标准回答框架

```
1. 结构化拆解（1分钟）
   - 不要一上来给答案
   - 将问题拆成多个能力维度分别评估

2. 核心竞争力判断（1分钟）
   - 这是我们的差异化壁垒吗？
   - 用户因为这个选择我们吗？

3. 全成本分析（1.5分钟）
   - TCO（含隐性成本）
   - 机会成本
   - 时间成本

4. 提出方案（1.5分钟）
   - 通常最优解是混合策略
   - 说明假设和前提条件
   - 分阶段执行计划

5. 风险与缓解（1分钟）
   - 关键风险识别
   - 缓解措施
   - 可逆性分析
```

## 七、关键收获

1. **「能不能做」不是正确的问题**——「值不值得做」「做了之后不做的事情值多少钱」才是
2. **能力解耦是核心技能**——不要把一个复杂系统当成整体来做Build/Buy决策，不同层级可以不同策略
3. **开源+定制是被低估的第三选项**——它兼具定制能力和成本效率
4. **机会成本是最常被忽略的成本**——10个工程师6个月的人力，可以做好几个核心功能
5. **决策的可逆性很重要**——可逆的决策可以快速试错，不可逆的决策要慎重评估
6. **PM在技术战略决策中的角色**——不是做技术判断，而是把用户价值、商业价值、成本和风险翻译成可对比的决策语言
