# 如何构建产品愿景与路线图？

## 面试场景

> 面试官：「假设你被任命负责一款增长乏力的成熟产品，你需要从零构建产品愿景和 12 个月路线图，向团队和高管说明"我们要去哪里"以及"怎么到"。你会怎么做？」

这道题考察的是 PM 的战略思维能力——能否从更高 abstraction 层面定义产品方向，并将其拆解为可执行的阶段性计划。这是中级到高级 PM 的核心分水岭。

---

## 一、核心概念：愿景不是口号，路线图不是功能清单

### 常见误区

| 误区 | 现象 | 后果 |
|------|------|------|
| 空话愿景 | "成为行业领先的智能平台" | 团队无法理解、无法执行 |
| 功能驱动路线图 | 路线图就是"Q1做X功能，Q2做Y功能" | 缺乏战略逻辑，沦为需求排队 |
| 静态规划 | 路线图制定后不更新 | 与市场变化脱节 |
| 自上而下独断 | PM一个人写愿景，团队被动执行 | 缺乏认同感，执行走形 |
| 愿景与资源不匹配 | 愿景宏大但团队只有5个人 | 战略焦虑，士气低落 |

### 产品愿景与路线图的关系

```
产品愿景（Why — 我们最终要创造什么未来）
    ↓
产品战略（What — 聚焦哪些领域、服务谁、如何赢）
    ↓
产品路线图（When — 分阶段实现路径）
    ↓
版本规划（How — 每个版本做什么）
    ↓
需求/功能（Detail — 具体功能的定义和优先级）
```

**关键洞察：** 路线图不是功能列表，而是**战略意图的时间映射**。每个阶段回答的不是"做什么功能"，而是"在这个阶段，我们要验证/达成什么"。

## 二、框架：VISION → STRATEGY → ROADMAP

### Step 1: 构建产品愿景

#### 愿景的三要素

一个好的产品愿景需要包含：

1. **目标用户**（Who）：我们在为谁创造价值？
2. **独特价值**（What）：我们提供什么不可替代的价值？
3. **未来状态**（Where）：3-5年后，用户的世界因为我们的产品变成了什么样？

#### 愿景构建工具：VMOST 分析

```
V - Vision（愿景）：3-5年的目标画面
M - Mission（使命）：我们每天在做什么来推进愿景
O - Objectives（目标）：可衡量的阶段性目标
S - Strategies（策略）：我们用什么方式赢
T - Tactics（战术）：近期具体行动
```

#### 愿景构建流程

```
1. 输入收集（2周）
   ├─ 用户深度访谈（现有用户 + 流失用户 + 潜在用户）
   ├─ 竞品分析与市场趋势
   ├─ 内部 stakeholder 访谈（销售、客服、技术、高管）
   └─ 数据回顾（核心指标趋势、用户行为变化）

2. 草案共创（1周）
   ├─ 核心团队 workshop（PM + 设计 + 技术 lead）
   ├─ 用 "Mad Libs" 格式初稿：
   │   "为 [目标用户]，我们提供 [独特价值]，
   │    让他们能够 [理想结果]，
   │    最终实现 [未来状态]。"
   └─ 收集反馈，迭代 2-3 轮

3. 验证与定稿（1周）
   ├─ 与高管对齐：愿景是否支持公司战略？
   ├─ 与团队验证：团队是否理解且认同？
   └─ 用 "电梯测试"：能否在30秒内说清楚？
```

#### 好愿景 vs 坏愿景

| ❌ 坏愿景 | ✅ 好愿景 |
|-----------|----------|
| 成为最好的项目管理工具 | 让每个团队都能像顶级团队一样高效协作 |
| 打造AI驱动的智能平台 | 让3亿中小商家用对话就能搭建自己的数字化经营系统 |
| 成为行业领导者 | 让等待救援的人均响应时间从小时级降到分钟级 |

### Step 2: 制定产品战略

愿景确定方向后，战略确定"怎么赢"。

#### 战略选择框架：Where to Play × How to Win

```
Where to Play（在哪竞争）：
├─ 目标用户群体（谁）
├─ 使用场景（什么时机）
├─ 产品形态（什么载体）
└─ 市场区域（什么范围）

How to Win（如何赢）：
├─ 差异化价值（为什么选我们）
├─ 竞争壁垒（为什么别人难复制）
├─ 资源配置（钱和人投在哪）
└─ 合作与生态（谁能帮我们）
```

#### 战略取舍：选择不做什么

好的战略一定包含明确的 **"不做清单"**：

> 例：我们的协作工具
> - ✅ 做：深度项目协作场景
> - ❌ 不做：通用即时通讯（打不过微信/钉钉）
> - ❌ 不做：免费个人用户市场（变现路径不清晰）

### Step 3: 构建产品路线图

#### 路线图的四种类型

```
1. Now-Next-Later 路线图（最推荐）
   - Now：未来 2-4 周，明确具体交付物
   - Next：未来 1-3 个月，明确方向和目标，灵活调整
   - Later：未来 3-12 个月，只标大方向和探索领域

2. 基于主题的路线图（Theme-based）
   - 不列功能，列主题（如"降低新用户上手成本"）
   - 每个主题配目标和成功指标
   - 具体功能由团队在每个迭代中决定

3. 基于目标的路线图（OKR-based）
   - 每个季度有明确的 Objective 和 Key Results
   - 功能服务于目标，而非目标服务于功能

4. 基于选项的路线图（Option-based）
   - 列出多个可能的探索方向
   - 标注每个选项的假设、成本、预期价值
   - 随时间推移逐步收敛
```

#### 路线图模板

```
┌─────────────────────────────────────────────────────────┐
│ 产品愿景：让每个团队都能像顶级团队一样高效协作              │
├─────────┬──────────┬──────────┬──────────┬───────────────┤
│  阶段   │ Q3 2026  │ Q4 2026  │ Q1 2027  │ Q2-Q3 2027   │
├─────────┼──────────┼──────────┼──────────┼───────────────┤
│ 主题    │ 降低上手  │ 深度协作  │ 智能化    │ 生态扩展     │
│         │ 成本      │ 能力      │ 升级     │              │
├─────────┼──────────┼──────────┼──────────┼───────────────┤
│ 目标    │ 新用户7日 │ 团队日活 │ AI辅助   │ 开放API      │
│         │ 激活率40%│ 提升30%  │ 任务分配 │ 第三方集成   │
│         │          │         │ 准确率85%│              │
```

#### 路线图沟通策略

不同受众需要不同版本：

| 受众 | 关注点 | 沟通方式 |
|------|--------|----------|
| 高管/投资人 | 商业价值、市场定位、ROI | 1页愿景+战略摘要 |
| 销售/市场 | 可承诺的功能和时间点 | 季度功能预览（但标注不确定性） |
| 工程团队 | 技术方向、架构演进 | 技术路线图 + 架构讨论 |
| 客户成功 | 哪些问题即将被解决 | 按客户痛点组织的优先级 |

## 三、实战案例：Notion 的愿景与路线图演进

### 早期愿景（2015-2017）

> "让每个人都能构建自己的工具，而不需要写代码。"

**Where to Play：** 早期采用者、技术爱好者
**How to Win：** 极致的灵活性（all-in-one workspace）

**路线图策略：** 快速迭代核心编辑器，建立技术壁垒。先做个人用户，再向团队扩展。

### 成长期调整（2018-2021）

愿景微调为：

> "让每个人都能像顶级团队一样，构建和使用自己需要的工具。"

关键变化：
- **Where 扩展：** 从个人到团队，从小团队到企业
- **How 升级：** 增加 **模板生态** 和 **集成能力**，形成网络效应
- **路线图调整：** 增加 **企业版功能**（SSO、权限管理、审计日志）

### 成熟期（2022-至今）

> "让任何人在任何地方，都能以自己的方式组织信息、协作和创造。"

路线图引入 **AI 能力**（Notion AI），将愿景从"工具构建"升级为"能力增强"。

**启示：** 愿景不是一成不变的。它会随着产品成熟度、市场变化和技术演进而迭代。但愿景迭代 ≠ 频繁转向，核心方向应保持稳定。

## 四、面试常见追问

### Q1: 如果高管要求你在路线图里加入你不认同的功能怎么办？

> 我会分三步处理：
> 1. **理解背后的真实需求**：高管要这个功能，本质想解决什么问题？
> 2. **提供替代方案**：用一个更好的方式解决同样的需求，并说明为什么更好
> 3. **如果无法说服，做 "小规模实验"**：用最小成本验证该功能的效果，用数据说话
> 核心原则是：**对目标负责，不对功能列表负责**。

### Q2: 路线图应该公开给用户吗？

> 取决于产品阶段和竞争环境：
> - **B2D/开发者产品**：公开路线图有利于建立信任和吸引贡献（如 Supabase、Cal.com）
> - **竞争激烈的市场**：过度公开会让对手提前应对
> - **折中方案**：公开主题方向，不公开具体功能和时间
> 关键是：**让用户感受到你在倾听和进步，但不给出你做不到的承诺**。

### Q3: 怎么平衡愿景的"宏大"和执行的"落地"？

> 我的方法是 **"双线并行"**：
> - **愿景线**：3-5年的方向，需要足够大，让人有使命感
> - **执行线**：每个季度都要有可衡量的产出，让团队感受到进展
> - **连接二者的是路线图**：每个季度的主题和目标，都应该能追溯到愿景
> 如果团队觉得愿景"太虚"，问题不在愿景本身，而在路线图没有把愿景翻译成近期的、可感知的变化。

### Q4: 如何衡量一个愿景是否"好"？

> 用 **"VRIN"** 框架检验：
> - **V**aluable：为目标用户创造显著价值
> - **R**are：在市场中是独特的，不是所有竞品都在做
> - **I**nimitable：不容易被复制（技术壁垒、网络效应、品牌等）
> - **N**on-substitutable：没有明显的替代方案
>
> 除此之外，还要加一个组织层面的检验：**团队能否在30秒内复述愿景并理解它**。如果说不清，愿景就需要简化。

## 五、标准回答框架

```
1. 愿景先行（60秒）
   "我会先回答三个问题：为谁创造什么价值？
   未来3-5年的画面是什么？
   我们凭什么赢？"
   → 产出一句愿景 + 不做清单

2. 路线图拆解（60秒）
   "我用 Now-Next-Later 格式：
   近期明确交付物和指标，
   中期标主题和假设，
   远期只留方向和探索。"
   → 产出分季度的主题、目标、关键交付

3. 共创与对齐（30秒）
   "愿景不是我一个人写的，
   需要核心团队共创、高管对齐、
   团队验证理解和认同。"

4. 动态调整（30秒）
   "路线图每季度 review，
   每半年大调整。
   用指标和用户反馈验证方向，
   但愿景不能频繁动摇。"
```

## 六、关键收获

| 要点 | 说明 |
|------|------|
| 愿景先于路线图 | 方向不明，规划无意义 |
| 路线图是主题不是功能列表 | 用"要达成什么"而非"做什么功能"来组织 |
| 选择不做什么很重要 | 战略的核心是取舍 |
| 不同受众不同版本 | 高管看价值，工程看方向，销售看承诺 |
| 愿景稳定，路线图动态 | 愿景半年到一年迭代，路线图季度迭代 |
| 共创比独断好 | 团队认同的愿景才有执行力 |

> **核心心法：** 产品愿景是北极星——你永远到不了它，但它指引你每一刻的方向。路线图是你向北极星行进的地图——它需要根据地形和天气不断调整，但方向始终不变。
