## 面试场景

> 面试官：你负责一个跨部门项目，涉及产品、设计、开发、测试、市场五个团队。项目启动两周后，你发现：
> - 设计稿延迟，因为设计师以为产品经理要先确认交互方案
> - 开发说需求不清晰，但PM认为PRD已经写完了
> - 市场团队完全不知道项目进度，却在抱怨没有人提前告知他们
>
> 你会如何使用RACI矩阵来解决这个问题？

这道题考察的不是你是否知道RACI这个概念，而是能否在真实的混乱场景中，用RACI作为工具建立秩序。

---

## 核心概念解析

### 什么是RACI矩阵？

RACI 是一种责任分配矩阵（Responsibility Assignment Matrix），用于明确项目中每项任务的角色和职责：

| 字母 | 角色 | 含义 | 关键特征 |
|------|------|------|----------|
| **R** (Responsible) | 执行者 | 实际完成任务的人 | 可以有多个，但越少越好 |
| **A** (Accountable) | 负责人 | 对结果最终负责的人 | **每项任务只能有一个A** |
| **C** (Consulted) | 咨询者 | 需要征求其意见的人 | 双向沟通 |
| **I** (Informed) | 知会者 | 需要被告知结果的人 | 单向通知 |

### 为什么跨部门项目容易在责任上出问题？

1. **角色模糊（Role Ambiguity）**：每个人对"谁该做什么"理解不同
2. **责任扩散（Diffusion of Responsibility）**：人多反而没人负责
3. **信息孤岛（Information Silos）**：各部门只关注自己的部分
4. **隐性期望（Implicit Expectations）**：没有明说但以为对方知道

## 框架：RACI落地五步法

### 第一步：拆解任务（WBS）

在分配责任之前，先把项目拆成可管理的任务单元：

```
项目：新功能上线
├── 1. 需求定义
│   ├── 1.1 用户需求调研
│   ├── 1.2 需求文档撰写（PRD）
│   └── 1.3 需求评审
├── 2. 设计
│   ├── 2.1 交互方案确认
│   ├── 2.2 UI视觉设计
│   └── 2.3 设计评审
├── 3. 开发
│   ├── 3.1 技术方案评审
│   ├── 3.2 前端开发
│   ├── 3.3 后端开发
│   └── 3.4 联调
├── 4. 测试
│   ├── 4.1 测试用例编写
│   ├── 4.2 功能测试
│   └── 4.3 回归测试
└── 5. 发布
    ├── 5.1 上线检查
    ├── 5.2 市场物料准备
    └── 5.3 上线发布
```

### 第二步：识别角色

列出所有相关方，包括个人和团队：

- PM（你）
- 设计师（小A）
- 前端开发（小B）
- 后端开发（小C）
- QA（小D）
- 市场负责人（小E）
- 技术负责人（大F）

### 第三步：填写RACI矩阵

| 任务 | PM | 设计师 | 前端 | 后端 | QA | 市场 | 技术负责人 |
|------|-----|--------|------|------|-----|------|------------|
| 1.1 用户调研 | **A/R** | C | I | I | I | I | I |
| 1.2 PRD撰写 | **A/R** | C | C | C | C | I | I |
| 1.3 需求评审 | **A** | R | C | C | C | I | C |
| 2.1 交互确认 | C | **A/R** | C | I | I | I | I |
| 2.2 UI设计 | I | **A/R** | C | I | I | C | I |
| 3.1 技术评审 | C | I | R | R | C | I | **A** |
| 3.2 前端开发 | C | I | **A/R** | C | I | I | C |
| 3.3 后端开发 | C | I | C | **A/R** | I | I | C |
| 4.1 测试用例 | C | I | C | C | **A/R** | I | I |
| 5.2 市场物料 | C | C | I | I | I | **A/R** | I |
| 5.3 上线发布 | **A** | I | R | R | R | C | C |

### 第四步：验证RACI原则

检查矩阵是否满足以下原则：

1. **每个任务只有一个A**：如果出现多个A，必须拆分任务或重新指定
2. **每个任务至少有一个R**：没有R的任务无人执行
3. **避免太多C**：每项任务的C不超过3人，否则变成"全员会议"
4. **每个角色都有参与**：如果某人在所有任务都是I，考虑是否真的需要他

### 第五步：沟通与迭代

- **项目启动会**上公布RACI矩阵，让所有人确认
- 将RACI嵌入项目管理工具（Jira、飞书、Notion）
- 每个Sprint回顾时检查RACI是否需要调整

## 实战案例：上面面试题的解法

回到面试场景中的三个问题：

### 问题1：设计稿延迟

**根因**：任务2.1（交互方案确认）的A不明确——设计师以为PM要先确认，PM以为设计师自己会定。

**解法**：
- 明确2.1的A是设计师（如果PM已给方向），或A是PM（如果需要PM拍板）
- 在RACI中标注依赖关系：2.1 depends on 1.3
- 加入前置条件checklist：PM在1.3评审时必须输出"交互方向文档"

### 问题2：开发说需求不清晰

**根因**：1.3（需求评审）的R只有PM，开发是C但没充分参与。

**解法**：
- 把1.3拆成 1.3a（技术预审，开发为R）+ 1.3b（正式评审，PM为A）
- 在RACI中增加一个"Definition of Ready"标准：PRD必须通过技术预审才能进入开发

### 问题3：市场团队不知情

**根因**：所有任务中市场都是I，但没有人执行"知会"这个动作。RACI写了不等于做了。

**解法**：
- 增加一个任务"5.0 利益相关方同步"，A是PM，R是市场负责人
- 设定固定节奏：每周五发送项目简报给所有I角色
- 在RACI旁加一列"沟通渠道"：邮件/群聊/会议

## 面试常见追问

### 追问1：如果两个部门对谁是A有争议怎么办？

**回答思路**：用"决策升级（Escalation）"机制。如果两个候选人都认为自己是A，说明任务定义不够清晰，需要：
1. 回到目标层面：这个任务的核心交付物是什么？
2. 用"谁承担后果"来判断A：如果出问题，谁会被追责？
3. 如果仍无法确定，升级到共同的上级做决策

### 追问2：RACI会不会让团队变得僵化？

**回答思路**：RACI是工具不是法律。好的PM会：
- 在项目初期用RACI建立秩序
- 团队磨合后简化为"责任人"制度
- 保留RACI作为新成员的onboarding文档
- RACI应随项目阶段动态调整，不是一成不变的

### 追问3：小型团队（5人以下）需要RACI吗？

**回答思路**：团队越小，隐性沟通越多，但恰恰是最容易出现"以为对方知道"的情况。小型团队可以用简化版：
- 只标注A和R，省略C和I
- 或者用"主人制"：每个任务指定一个Owner（A=R），其他人默认支持

### 追问4：如何避免RACI变成"甩锅工具"？

**回答思路**：这是RACI最常见的问题。解决方法：
1. 强调A不等于"出事背锅"，而是"确保事情做对"
2. A可以是"赋能者"而非"执行者"——A确保R有资源、有方向
3. 在团队文化中强调"共同责任"——每个人都是项目的一部分
4. RACI的目的是提高效率而非划定边界，定期回顾"我们是否在过度使用RACI"

## 标准回答框架

面试回答RACI相关问题时，建议用以下结构：

```
1. 诊断问题（30秒）
   → 识别当前的协作混乱属于哪种类型（角色模糊/责任扩散/信息孤岛）

2. 提出方案（60秒）
   → 介绍RACI框架
   → 说明如何针对当前问题应用
   → 给出具体的矩阵示例

3. 落地执行（30秒）
   → 如何让团队接受并使用RACI
   → 工具和节奏

4. 反思迭代（30秒）
   → RACI的局限性
   → 如何避免过度流程化
```

## RACI的常见陷阱

| 陷阱 | 表现 | 解法 |
|------|------|------|
| 全员都是A | "大家都要负责" | 拆分任务，每个子任务一个A |
| C的通货膨胀 | 每个人都要被Consulted | 区分"必须咨询"和"知会即可" |
| RACI变成借口 | "这不在我RACI里" | 定期回顾，强调协作优先 |
| 只做不更新 | 项目中期角色变化但RACI没改 | 每个Sprint回顾时更新 |
| 忽略外部依赖 | 忘了标注外部团队的C/I | 加入"外部角色"行 |

---

## 关键收获

1. **RACI的核心价值不是分配任务，而是消除模糊性**——当每个人都清楚自己的角色时，协作自然顺畅
2. **A是RACI的灵魂**——每项任务只有一个A，是避免责任扩散的关键
3. **RACI需要配套节奏**——没有沟通节奏的RACI只是一张死表格
4. **工具是手段不是目的**——当团队成熟到不需要RACI时，才是真正的成功
5. **面试中展示"场景化应用能力"**——不要背诵RACI定义，而是展示你如何在具体混乱中用它建立秩序
