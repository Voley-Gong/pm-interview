# PM 如何做好一次产品评审演讲？

## 面试场景

> 面试官："你刚加入新团队，负责一个全新的功能模块。下周你需要向开发、设计、测试团队以及你的 leader 做产品评审（PRD Review）。你会如何准备和执行这次评审？请详细描述你的方法。"

这类题目考察的是：**你能否清晰、有结构地传达产品方案，并在评审中有效推动共识**。这是 PM 日常最高频的高 stakes 沟通场景。

---

## 一、产品评审的核心目标

### 评审不是什么

| 误解 | 现实 |
|------|------|
| 评审是告诉大家方案 | 评审是让大家**认同**方案 |
| 评审是走流程 | 评审是争取**资源和承诺** |
| PPT 写得详细就够了 | **逻辑和叙事**比信息量更重要 |
| 评审的关键是答辩 | 评审的关键是**建立共识** |

### 评审的三个核心目标

1. **对齐目标**：所有人对"为什么做"达成一致
2. **理解方案**：所有人对"做什么"和"怎么做"有共同认知
3. **获取承诺**：各方明确自己的投入和时间线

---

## 二、评审准备的五要素模型

### 1. 明确受众（Audience）

不同的听众关心不同的问题：

| 角色 | 最关心的问题 | 你需要准备的内容 |
|------|-------------|-----------------|
| 开发 | 技术可行性？工作量？边界case？ | 详细交互逻辑、数据结构、异常处理 |
| 设计 | 用户场景？设计约束？品牌一致？ | 用户故事、参考设计、设计规范 |
| 测试 | 验收标准？异常场景？ | 边界条件、异常流程、兼容性要求 |
| Leader | ROI？风险？里程碑？ | 数据依据、资源评估、上线计划 |
| 运营 | 上线后怎么推？需要什么素材？ | 运营策略、物料需求、数据埋点 |

> ⚠️ 关键技巧：**在评审前 1-2 天，提前和关键干系人 1on1 预沟通**。尤其是技术负责人和你的 Leader。评审会上不应该是第一次看到反对意见。

### 2. 结构化叙事（Storyline）

**推荐使用 SCQA 框架构建你的评审叙事：**

```
S (Situation)    情境：目前的状态是什么？
C (Complication) 冲突：遇到了什么问题/机会？
Q (Question)     问题：我们该如何应对？
A (Answer)       方案：我的解法是什么？
```

**示例（在线问诊 App 加药提醒功能）：**

> **S：** 目前我们 App 的复诊购药率只有 15%，行业平均是 30%。
> **C：** 用户调研发现，70% 的用户看完诊后忘了买药或忘了吃药时间。
> **Q：** 如何在问诊后有效地推动用户完成购药和用药？
> **A：** 我们设计了"用药提醒"功能模块，通过系统自动生成用药计划……

### 3. 内容编排（Content Structure）

**15 分钟评审的时间分配建议：**

```
时间轴：

0-3 min   │ 背景与问题（为什么要做）← 不要超过 3 分钟
3-8 min   │ 方案详解（做什么 + 怎么做）
8-11 min  │ 数据依据与预期收益（为什么值得做）
11-13 min │ 资源需求与时间线（需要什么支持）
13-15 min │ 风险与讨论（开放给团队）
```

**内容清单（按顺序准备）：**

1. 背景说明（1页）：数据支撑的问题描述
2. 用户场景（1页）：典型用户的一天/一个流程
3. 方案概览（1页）：一图看懂整体方案
4. 详细设计（3-5页）：关键流程图、交互说明、异常处理
5. 数据依据（1页）：调研结论、AB测试预期、竞品参考
6. 里程碑（1页）：关键节点和负责人
7. 风险清单（1页）：已识别的风险和应对方案

### 4. 预判反对意见（Objection Handling）

**在准备阶段问自己：**
- 开发会问"这个能不能用现有架构实现？"
- Leader 会问"竞品做得怎么样？我们为什么不是抄他们？"
- 设计会问"这个交互跟现有体验一致吗？"
- 测试会问"极端情况下会怎样？"

**为每个可能的反对意见准备一张"备答卡"**（不一定放在 PPT 里，但要在脑子里）。

### 5. 演练（Rehearsal）

- **至少演练 2 遍**：一遍自查逻辑连贯性，一遍控制时间
- **找人扮演"挑刺者"**：让同事模拟评审中的挑战
- **准备 2 个版本的深度内容**：如果被追问，可以快速深入

## 三、评审执行：四个关键技巧

### 技巧一：开场 30 秒抓住注意力

**好的开场：**
> "上个月，我们因为问诊后购药率低，损失了约 80 万 GMV。今天我要介绍一个方案，预计能把这个数字提升到行业平均水平。整个方案只需要 3 周开发。"

**差的开场：**
> "今天我来给大家讲一下用药提醒功能的 PRD……"（没人在乎你的 PRD）

### 技巧二：用"金字塔原理"组织表达

```
          ┌─────────────┐
          │  核心结论先说  │  ← 第一句话就给出结论
          ├──────┬──────┤
          │ 论据1 │ 论据2 │  ← 然后展开支撑
          ├──┬──┼──┬──┤
          │..│..│..│..│   ← 最后是细节
          └──┴──┴──┴──┘
```

**应用示例：**
- ❌ "用户调研发现……数据表明……竞品分析显示……所以我认为我们应该做这个功能"
- ✅ "我们应该做用药提醒功能。原因有三个：第一，用户需求强烈；第二，预期 ROI 高；第三，技术实现成本低。下面我展开说明。"

### 技巧三：处理现场质疑

**当被质疑时，用"接受-理解-回应"模型：**

1. **接受**："这是一个很好的问题"（先肯定对方）
2. **理解**："你的担心是 XXX 对吗？"（复述确认）
3. **回应**：给出你的方案或承诺后续研究

**千万不要：**
- 当场争辩（评审不是辩论赛）
- 说"这个不重要"（对别人重要的你说了不算）
- 含糊带过（不专业的表现）

### 技巧四：控制节奏

| 信号 | 含义 | 应对 |
|------|------|------|
| 有人看手机 | 内容太长/太浅 | 加速，跳到核心方案 |
| 有人开始讨论细节 | 前面没讲清楚 | 暂停，拉回主线 |
| 沉默 | 可能在消化/不认同 | 主动提问："大家觉得这个方向有什么问题吗？" |
| 多人同时发言 | 有分歧 | 让每人轮流说，你做记录 |

## 四、评审后的跟进

### 24 小时内必做

1. **发送评审纪要**：决策结论、待办事项、负责人、截止日期
2. **更新文档**：根据评审反馈修改 PRD
3. **跟进关键异议方**：1on1 沟通未解决的问题

### 评审纪要模板

```
【产品评审纪要】用药提醒功能

时间：2026-06-18
参与人：PM、开发(3人)、设计(1人)、测试(1人)、Leader

✅ 已达成共识：
- 功能方向确认
- 一期范围确认（用药提醒+购药入口）

⏳ 待确认事项：
- 推送通道选择（APNs vs 自建Push）→ 张三负责，6/20前给出方案
- 用药频次的输入交互 → 李四负责，6/19前出设计稿

❌ 未通过/搁置：
- 社交分享功能（二期再议）

📌 里程碑：
- 设计定稿：6/22
- 开发开始：6/25
- 提测：7/05
- 上线：7/12
```

## 五、面试常见追问

### Q1："如果评审中开发说做不了，怎么办？"

**回答框架：**
1. 先理解原因——是技术不可能？还是工作量太大？还是时间不够？
2. 如果是技术不可能：讨论替代方案
3. 如果是工作量太大：讨论分期——一期做核心，二期做完整
4. 如果是时间不够：和你 Leader 一起重新排优先级
5. **永远不要在评审会上和技术争论技术可行性**——会后单独深入讨论

### Q2："如果你的 Leader 在评审中否定了你的方案呢？"

**应对策略：**
1. 当场不要硬刚，但也不要立刻妥协
2. 问清楚否定原因："您主要担心的是 ROI 还是方向？"
3. 如果是信息不足：承诺补充数据后再 review
4. 如果是方向分歧：请求单独 1on1 深入讨论
5. 评审会后主动找 Leader 沟通，展示你接受反馈的能力

### Q3："你怎么判断一次评审是否成功？"

**指标：**
- 参与度：大家有没有在讨论具体怎么做，而不是在问"为什么要做"
- 承诺度：各方有没有明确说"我会在 XXX 之前完成 YYY"
- 异议度：未解决的异议是否已经安排了 follow-up
- 会后行动：24小时内是否有人开始行动

## 六、标准回答框架

```
准备阶段：
1. 分析受众 → 每个角色关心什么
2. 预沟通 → 提前和关键人 1on1
3. 构建叙事 → SCQA 框架
4. 准备内容 → 15分钟时间分配
5. 预判异议 → 准备备答卡
6. 演练 → 至少2遍

执行阶段：
7. 开场抓注意力 → 数据/痛点/价值
8. 金字塔表达 → 结论先行
9. 管理讨论 → 接受-理解-回应
10. 控制节奏 → 关注信号，及时调整

跟进阶段：
11. 发送纪要 → 24小时内
12. 更新文档 → 闭环反馈
13. 追踪行动 → 确保落地
```

## 七、关键收获

1. **评审的成功 80% 取决于准备**——预沟通比正式评审更重要
2. **叙事结构 > 信息量**——15分钟讲不清的东西，30分钟也讲不清
3. **金字塔原理是 PM 的基本内功**——先说结论，再展开论据
4. **评审的目标是共识，不是说服**——赢了一场辩论，输了一次协作
5. **技术转 PM 的天然优势**：你能用开发听得懂的语言交流，更容易建立信任——用好这个优势

> 记住：**好的 PM 评审，听众走的时候心里想的是"我知道该做什么了"，而不是"他讲了好多"。** 表达的核心不是展示你自己，而是让其他人行动起来。
