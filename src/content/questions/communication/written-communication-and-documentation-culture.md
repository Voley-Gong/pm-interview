# 面试题：你入职后发现团队决策经常在群里口头讨论，没有记录，导致反复扯皮。你如何建立有效的文档文化？

## 面试场景

面试官给你一个组织场景：

> "你新加入一个 15 人的产品团队。你发现一个现象：每次开会长达 2 小时，讨论很热烈，但会后没人记录结论。下一次开会，同样的问题重新争论。关键决策散落在各种微信群聊里，新来的人完全不知道'为什么当初做了这个决定'。你作为 PM，怎么改变这个状况？"

这道题看起来是问"你怎么写文档"，实际上考察的是：**你能否用书面沟通工具来驱动组织效率和决策质量**。初级PM把文档当作业交，高级PM把文档当杠杆用。

---

## 核心概念：为什么书面沟通如此重要？

### 文字的三个独特优势

| 维度 | 口头沟通 | 书面沟通 |
|------|---------|---------|
| **时间跨度** | 只存在于此刻，过后即消失 | 可以跨越天、周、月、年 |
| **可追溯性** | "我记得当时说的是…" | 白纸黑字，精确可查 |
| **可异步性** | 需要所有人同时在场 | 各自按自己的节奏阅读和回复 |

### 亚马逊的"六页备忘录"文化

杰夫·贝索斯在 2004 年禁止了 Amazon 内部使用 PPT 汇报。取而代之的是**结构化备忘录（Narrative Memo）**：

- 每个提案必须写成 4-6 页的结构化文档
- 会议前 20 分钟所有人安静阅读
- 阅读后再开始讨论

**为什么？** 贝索斯发现：
- PPT 让人"隐藏逻辑漏洞"，文字让人"暴露思考缺陷"
- PPT 是"表演"，文字是"思考"
- PPT 信息密度低，文字信息密度高

### 对PM的特殊意义

PM 是"信息枢纽"——你需要把来自用户、数据、设计、技术、商业的信息整合并传递出去。你的书面沟通质量，直接决定了团队对齐的效率。

## 框架：PM 核心文档体系

### 四层文档金字塔

```
           ┌─────────────┐
           │   战略文档   │  ← 方向（季度/年度）
           │  Vision/Roadmap│
           ├─────────────┤
           │   决策文档   │  ← 为什么做这个决定
           │     ADR      │
           ├─────────────┤
           │   执行文档   │  ← 具体怎么做
           │  PRD/Spec    │
           ├─────────────┤
           │   记录文档   │  ← 发生了什么
           │  Meeting/Ops │
           └─────────────┘
```

### 每一层的关键实践

#### Layer 1：战略文档 — 让所有人看到同一个方向

**核心文档：** 产品愿景文档、季度 Roadmap

**PM 新人常犯的错：** 觉得"方向是老板的事，我不需要写"。
**正确认知：** 即使你不是方向制定者，你也需要把方向**翻译成可执行的语言**写下来。

**一个有效的 Roadmap 文档应该包含：**
- 我们今年要解决的 3 个核心问题是什么？
- 每个问题成功的标志是什么？
- 优先级排序是什么？为什么？
- 我们不做什么？为什么不做？

#### Layer 2：决策文档 — 最被忽视但最有价值的文档

**ADR（Architecture Decision Record）的产品版：PDR（Product Decision Record）**

模板：
```markdown
# 决策：[决策标题]
日期：YYYY-MM-DD
状态：[提案 / 已接受 / 已废弃 / 已推翻]

## 背景
为什么要做这个决策？（上下文、痛点、触发事件）

## 决策内容
我们决定做什么？（具体、清晰、不含糊）

## 考虑过的替代方案
还有哪些选项？为什么不选它们？

## 后果
这个决策带来什么好处？什么风险？什么约束？
```

**价值：**
- 6 个月后没人记得"为什么做了这个决定"，PDR 是唯一可靠的答案
- 新人入职时，PDR 是最好的产品历史教科书
- 写 PDR 的过程本身就在检验决策质量——如果你写不清楚"为什么"，说明你还没想清楚

#### Layer 3：执行文档 — PRD 不只是需求清单

**常见误区：** PRD = 功能清单 + 原型图
**正确理解：** PRD = 决策依据 + 用户故事 + 验收标准 + 不做什么

**好 PRD 的自检清单：**
- [ ] 读者能在 5 分钟内理解"为什么做这个"
- [ ] 每个功能点都有对应的用户故事
- [ ] 明确写了"不做什么"及原因
- [ ] 验收标准具体到可测试
- [ ] 没有模糊词语（"体验更好""性能提升"不算需求）

#### Layer 4：记录文档 — 降低组织记忆流失

**会议记录的极简模式（3W）：**
- **What**：讨论了什么
- **Why**：关键分歧和论点
- **What's Next**：谁、做什么、什么时候

**原则：不超过一页。如果会议记录超过一页，说明会议本身没有效率。**

## 实战：如何在团队中推动文档文化

### 策略：从小处切入，用效果说话

**不要一上来就搞"全员文档化运动"。** 这种自上而下的推法大概率会变成形式主义。

#### 阶段一：自己先做好（第 1-2 周）

作为 PM，先以身作则：
- 每次会议后发一页纸的会议记录
- 每个重要决策写 PDR
- PRD 写得清晰、结构化

**目标：** 让团队感受到"有记录 vs 没记录"的差别。

#### 阶段二：影响核心协作人（第 3-4 周）

- 和技术负责人约定：PRD 评审基于文档，不再口头补充
- 和设计师约定：设计决策也写进文档
- 开始在关键会议中引入"会后 5 分钟整理记录"的习惯

**目标：** 让协作效率提升说话。

#### 阶段三：建立团队规范（第 5-8 周）

- 推动建立文档模板和存放位置
- 和 Leader 对齐：将"文档质量"纳入项目复盘
- 新人入职时引导他们阅读 PDR 理解历史

**目标：** 让文档成为团队习惯，而不是 PM 一个人的事。

### 常见阻力及应对

| 阻力 | 应对 |
|------|------|
| "写文档太花时间" | "你花 30 分钟写文档，省掉 3 次重复讨论（每次 30 分钟 × 5 人 = 150 分钟）" |
| "情况一直在变，文档写了也白写" | "文档不是刻碑，是记录此刻的最佳判断。变了就更新，顺便记录为什么变" |
| "口头沟通更快" | "口头沟通当下更快，但 3 个月后没人记得。文档是给未来的自己和新人写的" |
| "我不是PM，不需要写文档" | "决策记录不是PM专属，是所有思考者的工具。技术有ADR，设计有设计文档" |

## 面试常见追问

### 追问 1："文档写了没人看怎么办？"

**回答思路：**
- 先问自己：文档是否写得太长、太模糊、结构混乱？
- 好的文档应该让人"3 分钟内找到自己需要的信息"
- 推动会议中"先读文档再讨论"的规则——如果不读，不开会
- 把关键信息前置（BLUF 原则：Bottom Line Up Front）

### 追问 2："远程团队和面对面团队，文档文化的区别？"

**回答思路：**
- 远程团队必须比面对面团队更重视文档——因为没有"走到工位问一句"的兜底
- 远程团队应该建立"默认异步"的文化：默认通过文档沟通，只有必要时才开会
- 面对面团队的陷阱是"说了就等于定了"——需要把口头讨论的关键结论补充记录

### 追问 3："你怎么平衡'写文档'和'做事情'的时间？"

**回答思路：**
- 文档不是额外的工作，是工作的"副产品"——你在思考时自然产出的结构化内容
- 用模板降低写文档的门槛——模板让你关注内容，而非格式
- 投资回报率：花在文档上的每 1 分钟，会节省未来 3-5 分钟的沟通成本
- 劣势：如果文档质量不高（模糊、啰嗦、过时），它不仅不省时间，还会制造混乱

## 标准回答框架

### 1. 问题诊断（1 分钟）
- 指出团队的核心问题不是"不写文档"，而是"信息没有沉淀"
- 分析这对团队效率的具体影响（重复讨论、新人上手慢、决策不可追溯）

### 2. 解决方案（2 分钟）
- 提出四层文档体系的简要版本
- 重点强调 PDR（决策记录）的价值——这是最容易被忽视但 ROI 最高的
- 展示一个 PDR 的简单模板

### 3. 推动策略（1.5 分钟）
- 不要自上而下推改革，而是"自己先做好，用效果带动"
- 三阶段渐进推动：自己做好 → 影响核心协作人 → 建立团队规范
- 针对常见阻力准备好论据

### 4. 个人实践（1 分钟）
- 举例说明你曾经写过的一个文档如何解决了具体的对齐问题
- 如果你还没有 PM 文档经验，用技术文档（wiki、README、ADR）的经验替代

### 5. 升华（30 秒）
- 文档文化的终极目标不是"写文档"，而是"让团队的每个决策都经得起时间的检验"
- 好的文档文化是一个组织的"知识复利"

---

## 关键收获

1. **书面沟通是 PM 最被低估的核心能力**：说话能影响在场的人，文字能影响不在场、不同时区、甚至未来的人
2. **四层文档体系的核心是 PDR（决策记录）**：它解决了"为什么当初做了这个决定"这个组织中最常见的知识流失问题
3. **好的文档不是写出来的，是"思考出来的"**：写作是思考的工具，不是思考的记录——如果你写不清楚，说明你还没想清楚
4. **推动文档文化的策略是"以身作则 + 用效果说话"**：不要用权力推改革，用价值说服人
5. **对于技术转PM 的人，这是天然优势**：写 ADR、README、技术方案的能力，可以无缝迁移到 PDR、PRD、产品决策文档
