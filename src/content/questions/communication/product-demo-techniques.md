# 产品演示与Demo技巧：如何让听众在15分钟内爱上你的产品

## 面试场景

> 面试官："假设你负责一款B2B SaaS产品，下周要给一个重要客户做产品Demo。对方CEO和技术总监都会参加，你只有20分钟。你会怎么准备？"

> 追问："如果Demo进行中突然出现了Bug，功能用不了，你怎么办？"

这道题考察的不只是"会不会做PPT"，而是你对产品价值的理解深度、对听众心理的把控能力、以及临场应变能力。

## 核心概念解析

### 产品Demo的本质是什么？

很多人以为Demo就是"把产品功能展示一遍"——**大错特错。**

Demo的本质是：**用最短的时间，让听众相信你的产品能解决他们的问题。**

三个关键要素：
1. **场景驱动**（不是功能驱动）——不是"看，我们有这个功能"，而是"看，这个功能怎么帮你解决问题"
2. **故事叙事**（不是流程罗列）——带着听众从"痛点"走到"解决方案"再到"价值实现"
3. **互动参与**（不是单向展示）——让听众觉得"这是为我定制的产品"

### 产品Demo的三大类型

| 类型 | 目标听众 | 核心目标 | 成功标准 |
|------|----------|----------|----------|
| **销售型Demo** | 客户决策者 | 推进成单 | 客户主动询价/要求POC |
| **路演型Demo** | 投资人/行业 | 展示愿景和吸引力 | 产生合作兴趣 |
| **内部型Demo** | 领导/跨部门 | 推动资源/对齐方向 | 获得资源或决策支持 |

## Demo设计的黄金框架：SCENE五步法

### S — Segment（定位听众）

**在做Demo之前，你需要回答三个问题：**

1. **听众是谁？** 角色不同，关注点完全不同
   - CEO：关心ROI、竞争壁垒、战略价值
   - 技术总监：关心架构、安全性、集成能力
   - 一线员工：关心好不好用、能否减轻工作量

2. **他们当前的痛点是什么？**
   - 用什么方案在解决？为什么不满意？
   - 如果不解决会损失什么？

3. **他们需要听到什么才会行动？**
   - 什么信息会让他们说"这个我要"
   - 什么疑虑会阻止他们决策

**一个Demo不可能打动所有人——聚焦核心决策者，兼顾其他人。**

### C — Context（设定场景）

**用真实业务场景代替功能罗列**

❌ 错误做法：
> "我们的产品有仪表盘功能，可以看数据。还有告警功能，可以设置阈值。还有报表功能……"

✅ 正确做法：
> "想象你是运维负责人，凌晨3点服务器负载突然飙升。以前你需要登录5个系统才能找到问题——现在，你只需要打开这一个仪表盘，所有关联数据一目了然，系统自动定位异常原因并推送告警。"

**场景设计的三个要素：**
- **角色代入**："想象你是……"
- **痛点共鸣**："以前你需要……（描述低效现状）"
- **解决方案**："现在，你只需要……（展示产品价值）"

### E — Engage（建立互动）

**互动不是问"大家有没有问题"——而是在Demo过程中制造参与感：**

1. **预设问题**："你们现在的数据处理大概需要多长时间？"（让听众说出痛苦）
2. **选择分支**："你们更关心成本控制还是效率提升？我重点展示对应的模块。"
3. **数据个性化**：用客户自己的数据或行业数据做Demo，而不是通用测试数据
4. **肢体互动**：请听众上台操作某个关键功能，或在关键节点暂停，问"你觉得接下来会发生什么？"

### N — Narrative（构建故事线）

**经典Demo故事结构：**

```
开场（2分钟）
├── 痛点共鸣：用一个真实案例或数据开场
├── 价值承诺："今天我会展示如何用15分钟解决过去需要2天的问题"
└── 议程预告：简述你会讲什么（不超过30秒）

主体（12-15分钟）
├── 场景1：核心痛点场景 + 产品解决方案
│   └── 展示关键功能（最多3个）
├── 场景2：延伸场景 + 进阶价值
│   └── 展示差异化能力（1-2个）
└── 场景3：未来扩展 + 生态价值
    └── 展示路线图和想象力

结尾（3分钟）
├── 价值回顾：用数据对比"过去vs现在"
├── 下一步：明确行动建议（试用/POC/商务对接）
└── 互动问答：预留答疑时间
```

**关键原则：**
- **1个Demo最多展示5个功能点**——多了听众记不住
- **每个功能点必须有对应的业务价值**——功能不是价值
- **用对比强化记忆**：Before/After对比是最有力的说服

### E — Execute（完美执行）

**Demo前的准备清单：**

- [ ] **录制备用视频**：如果现场Demo出问题，立即切换到录屏视频
- [ ] **准备Demo环境**：独立环境，不依赖网络，预装好数据
- [ ] **练习3次以上**：精确到每分钟讲什么
- [ ] **检查设备**：投影/HDMI转接头/分辨率适配
- [ ] **准备FAQ**：预判听众会问的10个问题，准备好答案
- [ ] **安排助手**：复杂Demo需要有人在旁边配合（比如操作另一个角色的账号）

**Demo中的注意事项：**
- **速度要慢**：你熟悉每个操作，但听众是第一次看。关键操作暂停2秒。
- **鼠标要稳**：不要随意晃动鼠标，移动时慢且明确
- **页面要净**：关闭无关的Tab、通知、弹窗
- **字号要大**：确保最后一排的人也能看清屏幕文字
- **节奏要控**：每3-4分钟一个"高潮点"（展示关键价值），中间是过渡

## 实战案例：B2B SaaS产品Demo

### 场景
某数据分析SaaS公司，给一家零售连锁企业CEO和技术总监做Demo。

### Demo结构设计

**听众分析：**
- CEO：关心ROI、怎么帮门店提升业绩
- 技术总监：关心数据安全、集成难度、维护成本

**故事线设计：**

```
开场（2min）：
"王总，您提到目前300家门店的数据要等到月底才能汇总。
这意味着一个问题：如果某个门店在月初开始下滑，您要到下个月才发现——
这中间30天的损失是多少？按照平均门店月营收50万，下滑10%就是5万，
30天就是150万的损失。今天我会展示如何把这个发现时间从30天缩短到1天。"

场景1 - CEO视角（6min）：
→ 展示实时仪表盘：各门店业绩热力图
→ 演示异常自动预警：某门店连续3天下滑→自动推送通知
→ 展示根因分析：一键钻取到具体下滑原因（品类/时段/客流）

场景2 - 技术总监视角（5min）：
→ 展示数据集成方式：API对接现有POS系统，2周上线
→ 展示权限管理和审计日志
→ 简要展示技术架构（不必深入，点到即可）

场景3 - 未来扩展（3min）：
→ AI预测能力：不仅发现异常，还能预测下月趋势
→ 门店对比智能分析：自动找到"最佳实践门店"

结尾（2min）：
价值总结："发现时间30天→1天，潜在挽回损失每月百万级"
下一步："建议先选3家门店做2周POC验证"
```

## Demo中Bug处理：应急处置方案

### 方案1：录屏备份（推荐）
> "为了让大家更清楚地看到操作细节，我准备了一段录屏视频。"
→ 平滑切换到备用视频，听众甚至不知道出了Bug

### 方案2：幽默化解
> "看来我们的产品太激动了。这个问题我已经记录了，可以稍后单独聊。我们先看下一个亮点。"
→ 不要纠结于修复Bug，继续推进有价值的内容

### 方案3：转为讨论
> "这个功能在正常环境下运行是很流畅的。我想先听听大家对这个场景的期待是什么？"
→ 把展示转为对话，既化解了尴尬又获得了需求洞察

### 绝对不能做的事：
- ❌ 当场调试Bug（浪费所有人的时间）
- ❌ 反复刷新页面等待恢复（显得不专业）
- ❌ 假装没出问题（听众又不瞎）
- ❌ 把责任甩给技术团队（PM要承担责任）

## 面试常见追问

### Q1: 如果听众中途打断问你问题，怎么办？
**判断类型：**
- 澄清型（"这个数据是实时的吗？"）→ 30秒内简短回答，继续Demo
- 深挖型（"这个功能支持自定义规则吗？"）→ "好问题，我在第3部分会讲到"或"我们后面单独聊"
- 跑题型（"你们公司融资了吗？"）→ "今天我们聚焦在产品价值上，商务问题会后可以详聊"
- 挑战型（"你们的方案比XX贵很多吧？"）→ "价格取决于价值，我们先看看能带来多少价值"

### Q2: Demo时间被临时缩短怎么办？
**准备三个版本：**
- 完整版（20min）：3个场景 + 完整故事
- 精简版（10min）：1个核心场景 + 价值总结
- 极简版（3min）：一句话价值 + 一个最核心的功能截图

### Q3: 如何衡量Demo的效果？
- **即时指标**：听众是否主动提问/要求试用
- **后续指标**：Demo后1周内的客户跟进响应率
- **成单指标**：从Demo到签约的转化率和周期
- **反馈指标**：会后向听众收集满意度评分

## 标准回答框架

```
1. 听众分析：谁在场，各自关心什么
2. 场景设计：选什么业务场景最能打动他们
3. 故事结构：开场→主体→结尾的内容安排
4. 功能取舍：选哪3-5个功能展示，为什么
5. 风险预案：Demo出问题怎么办
6. 后续行动：Demo结束后推动什么下一步
```

## 关键收获

1. **Demo的目标不是"展示功能"而是"激发渴望"**——听众看完应该想"我需要这个"
2. **场景永远大于功能**——用业务故事包装产品能力，而不是用功能清单考验听众耐心
3. **准备永远是超额的**——台上的从容来自台下的100小时准备
4. **Bug不是终点，应变能力才是考点**——面试官想看你如何处理不确定性
5. **最好的Demo让听众忘记你在推销**——他们觉得你在帮他们解决问题
