# 如何做一场让人记住的产品演讲？

## 面试场景

> 面试官："假设你负责的产品即将在公司年度大会上做 15 分钟的产品发布演讲，台下有 CEO、销售团队、核心客户。你会怎么准备和呈现这场演讲？"

这道题考察的不是"你敢不敢上台"，而是你是否理解：**产品演讲本质上是一场有目的的故事讲述**。技术转 PM 的同学尤其容易陷入"堆功能"的陷阱。

---

## 核心概念：产品演讲的三层结构

一场好的产品演讲，需要在三个层面同时发力：

### 第一层：信息层（What）
听众走的时候能记住的一句话说什么？

### 第二层：情感层（How）
你用了什么故事、比喻、冲突来让这句话"粘住"？

### 第三层：行动层（So What）
听众听完后应该做什么？试用？采购？支持？

**常见错误：** 90% 的技术背景演讲者只做到了第一层——把 PPT 塞满功能列表。

---

## 框架：SCOR 演讲模型

这是我自己总结的产品演讲框架，面试中可以直接用：

### S — Setup（铺垫场景）
用一个真实的用户场景开场，而不是"今天我来介绍 XX 功能"。

> ❌ "今天我来介绍我们的新数据分析平台 v3.0"
> ✅ "上周三凌晨两点，某电商团队的运营小李还在用 Excel 拼接 7 个数据源的手动报表——这不是个例，这是我们调研的 200 个团队中 73% 的日常"

### C — Conflict（制造冲突）
抛出矛盾、痛点或意外发现，让听众产生"然后呢？"的好奇心。

> "有趣的是，当我们深入访谈这 73% 的团队时，发现他们不缺工具——他们有 Excel、有 BI 平台、有数据工程师。他们缺的是'把数据变成决策'的最后一公里。"

### O — Offer（给出方案）
这时候才引出你的产品方案，而且要聚焦在"解决了什么冲突"，而不是"有哪些功能"。

> "所以我们做了三件事来补上这最后一公里：智能归因、一键洞察、决策看板。不是更多功能，而是更少决策。"

### R — Resonance（收尾共鸣）
用一个画面、一句金句或一个数字来收尾，让演讲有一个"记忆锚点"。

> "我们的愿景是：让每个团队的数据分析师，不再凌晨两点还对着 Excel。"

## 实战案例：三段式产品发布演讲

假设你要发布一款"开发者效率度量平台"，以下是完整的 15 分钟演讲设计：

| 时间 | 环节 | 内容 | 目的 |
|------|------|------|------|
| 0-2 min | 开场故事 | 一个开发团队因为无法量化技术债务，导致项目延期 3 倍的真实案例 | 建立共情 |
| 2-4 min | 问题拆解 | 行业数据：70% 的技术团队没有可量化的效率指标 | 制造紧迫感 |
| 4-5 min | 转折点 | "效率不可度量，就不可管理"（金句） | 记忆锚点 |
| 5-10 min | 产品方案 | 三个核心能力：自动采集 → 可视化看板 → AI 诊断建议 | 逻辑展开 |
| 10-12 min | Demo | 现场演示一个"从接入到看到洞察"的 90 秒流程 | 眼见为实 |
| 12-14 min | 客户证言 | 某团队使用 3 个月后，平均需求交付周期缩短 40% | 信任背书 |
| 14-15 min | Call to Action | "今天就注册，前 100 个团队免费试用 90 天" | 明确行动 |

## 面试常见追问

### 追问 1："如果台下听众背景差异很大（技术 vs 业务），你怎么平衡？"

**回答思路：** 使用"三明治结构"——
- 面向所有人：用故事和场景开场（业务听得懂，技术有共鸣）
- 面向特定群体：深入技术细节时给一句翻译（"简单来说就是……"）
- 面向所有人：用价值总结收尾（所有人关心的是"so what"）

### 追问 2："演讲前你害怕忘词怎么办？"

**回答思路：** 不要背逐字稿。记住三个锚点：
1. **开场第一句话**（必须背到肌肉记忆）
2. **三个核心要点的关键词**（不是完整句子）
3. **结尾金句**（必须背到肌肉记忆）

中间的内容用"逻辑链"自然推导——如果你真的理解你要讲的东西，你不会忘。

### 追问 3："如果演讲中途出了技术故障（PPT 放不出来），你怎么办？"

**回答思路：**
> "这正是检验你是否真的理解产品的时刻。如果你依赖 PPT 才能讲清楚，说明你还没真正想清楚。我会用白板画出产品的核心流程，用纯口头讲述继续。实际上，有时候'裸讲'反而更真诚、更有感染力。"

### 追问 4："你怎么衡量演讲的效果？"

**回答思路：** 三个层次——
- **即时层：** 现场反应（提问质量、眼神聚焦度、会后主动交流人数）
- **行为层：** 演讲后 48 小时内的注册量 / 试用申请 / 内部支持邮件
- **记忆层：** 一周后随机问听众"还记得讲了什么"，能复述核心观点的比例

## 标准回答框架

面试时遇到"如何做产品演讲"类题目，按这个结构回答：

```
1. 演讲本质定义（30秒）
   "产品演讲不是功能汇报，而是用故事驱动行动。"

2. 准备阶段（60秒）
   → 研究听众画像（决策者关心什么？执行者关心什么？）
   → 确定一个核心信息（如果听众只记一句话，是什么？）
   → 设计故事线（SCOR 模型）

3. 结构设计（60秒）
   → 开场（场景 + 冲突）
   → 主体（3 个核心价值点，不是功能点）
   → 收尾（金句 + 明确行动呼吁）

4. 排练与优化（30秒）
   → 至少完整排练 3 次
   → 录像回看，砍掉一切不服务于核心信息的内容
   → 准备 5 个可能被问到的问题

5. 衡量效果（30秒）
   → 即时反馈 + 行为转化 + 记忆留存
```

## 技术转 PM 的特殊注意事项

作为技术背景的 PM 候选人，面试官会特别关注你是否能克服以下倾向：

| 技术人员的习惯 | 产品演讲的要求 |
|----------------|----------------|
| 从架构讲起 | 从用户场景讲起 |
| 炫耀技术深度 | 展示用户价值 |
| 功能列表式介绍 | 故事化叙事 |
| 追求"全面" | 追求"记忆点" |
| 假设听众懂术语 | 每个术语都有翻译 |

**一句话总结：** 技术人做演讲最大的优势是"你真的懂这个产品"，最大的风险是"你以为别人也懂"。翻译能力是关键。

---

## 关键收获

1. **产品演讲 ≠ 功能介绍。** 它是一场有明确目的的故事讲述——目的是让听众记住一个信息并采取行动。

2. **SCOR 模型**（Setup → Conflict → Offer → Resonance）是一个实用的演讲设计框架，比常见的"总-分-总"更能抓住注意力。

3. **准备演讲时，先写一句话**：如果听众只能记住一句话，你希望是什么？整个演讲围绕这一句话展开。

4. **技术转 PM 的核心转变：** 从"我能做出这个功能"到"我能让别人相信这个功能值得用"。

5. **演讲效果的可衡量性**也是 PM 思维的体现——不要说"效果不错"，要用注册量、转化率、记忆留存率来量化。
