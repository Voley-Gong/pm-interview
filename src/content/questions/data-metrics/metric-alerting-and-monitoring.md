## 面试场景

> 面试官：「你的产品日活突然下降了20%，但你是3天后才知道的。你会怎么设计一套实时指标监控与告警体系，确保类似事件能在10分钟内发现、1小时内响应？」

## 核心概念解析

### 监控告警体系的三个层次

| 层次 | 目标 | 时间窗口 | 典型指标 |
|------|------|---------|---------|
| **L1 实时告警** | 发现"出血点"，立刻响应 | 秒级~分钟级 | 错误率、支付成功率、核心链路QPS |
| **L2 日常监控** | 发现趋势异常，及时介入 | 小时级~天级 | DAU、转化率、留存率、功能渗透率 |
| **L3 长期追踪** | 发现结构性变化，战略调整 | 周级~月级 | 北极星指标、LTV/CAC、市场份额 |

面试中常考的是L1和L2的设计能力，因为这是PM日常协作最密切的部分。

### SLA / SLO / SLI 概念框架

- **SLI（Service Level Indicator）**：你测量的具体指标，如"支付成功率"
- **SLO（Service Level Objective）**：你对SLI设定的目标，如"支付成功率 > 99.95%"
- **SLA（Service Level Agreement）**：对外的承诺，通常比SLO更宽松，如"支付成功率 > 99.9%，否则赔偿"

PM需要参与SLO的制定——它直接影响用户体验和团队的工作节奏。

## 设计监控告警体系的六步法

### Step 1：确定关键用户旅程（Critical User Journey）

不是所有指标都值得实时监控。先识别产品的核心路径：

```
示例：电商App的关键用户旅程
注册/登录 → 搜索/浏览 → 加购物车 → 下单 → 支付 → 履约 → 评价
```

每个环节定义一个**成功指标**和**失败指标**：
- 支付环节：成功指标=支付成功率，失败指标=支付失败笔数
- 搜索环节：成功指标=搜索结果点击率，失败指标=零结果率

### Step 2：选择监控指标并设定基线

对每个关键环节，确定：

| 指标类型 | 说明 | 例子 |
|---------|------|------|
| **流量指标** | 反映规模 | QPS、UV、PV |
| **成功率指标** | 反映可用性 | 支付成功率、API成功率 |
| **延迟指标** | 反映性能 | P50/P95/P99响应时间 |
| **饱和度指标** | 反映容量 | CPU使用率、队列长度 |
| **错误指标** | 反映质量 | 错误率、Crash率 |

**USE原则**（Utilization-Saturation-Errors）：系统指标至少覆盖这三个维度。

### Step 3：设定告警阈值

这是最容易出错的一步。两种方法论：

**方法A：固定阈值法**
- 简单直接：成功率 < 99.9% → 告警
- 适用场景：有明确SLA承诺的核心指标

**方法B：动态阈值法（Anomaly Detection）**
- 基于历史数据建模：当前值偏离过去7天同时段均值 > 3σ → 告警
- 适用场景：有周期性波动的指标（如DAU、转化率）
- 工具支持：环比(yesterday vs today)、同比(weekday vs weekday)、季节性分解

**阈值设定原则**：
```
误报代价 < 漏报代价 → 阈值宽松些（宁可多报警）
误报代价 > 漏报代价 → 阈值严格些（避免告警疲劳）
```

### Step 4：告警分级与路由

| 级别 | 定义 | 响应时间 | 通知方式 | 示例 |
|------|------|---------|---------|------|
| **P0** | 核心功能不可用 | 立即 | 电话+短信+IM | 支付全线崩溃 |
| **P1** | 核心功能受损 | 15分钟 | 短信+IM | 支付成功率下降5% |
| **P2** | 次要功能异常 | 1小时 | IM | 某非核心API错误率升高 |
| **P3** | 需关注趋势 | 工作日 | 日报 | 留存率连续3天下降 |

**关键原则：告警必须可操作。** 如果一条告警不需要任何人采取行动，它就不应该发出。

### Step 5：建立Runbook（应急手册）

每条告警必须配有Runbook，说明：

```markdown
## 告警：支付成功率低于99%

### 可能原因
1. 第三方支付通道故障 → 检查通道健康面板
2. 风控规则误杀 → 检查风控拦截日志
3. 数据库慢查询 → 检查DB监控面板
4. 客户端版本bug → 检查崩溃率是否同步上升

### 处理步骤
1. 查看告警面板中的细分维度（渠道/版本/地区）
2. 如果单一渠道问题 → 切换备用通道
3. 如果风控误杀 → 联系风控团队临时调整规则
4. 如果DB问题 → 通知DBA团队
5. 如果客户端bug → 评估是否需要热修

### 升级机制
- 15分钟未恢复 → 通知Tech Lead
- 30分钟未恢复 → 通知CTO + PM Head
- 1小时未恢复 → 启动P0应急流程
```

### Step 6：告警体系健康度评估

定期review告警体系本身：

- **告警信噪比**：真正有效的告警 / 总告警数。低于30%说明阈值需要调优
- **平均响应时间（MTTA）**：从告警触发到有人开始处理的时间
- **平均恢复时间（MTTR）**：从告警触发到恢复的时间
- **告警疲劳度**：同一时段告警数量 > 10条时，人的注意力急剧下降

## 实战案例：打车App高峰期订单骤降

**场景**：周五晚8点，某打车App的核心城区订单量在10分钟内下降35%。

**如果没有告警体系**：
- 运营在2小时后看日报才发现
- 司机端已大量空驶→司机退回
- 乘客端排队过长→差评涌入
- 次日DAU下降8%

**有告警体系的10分钟响应**：

```
T+0min：动态阈值触发P1告警
        → 自动通知值班PM + 值班工程师

T+2min：值班PM查看分维度面板
        → 发现仅iOS端下降，Android正常
        → 排除天气/政策因素

T+5min：确认是iOS新版本发布后Crash率飙升
        → 触发P0升级

T+8min：决策——启动版本回滚
        → 通知发布工程师执行灰度回滚

T+15min：回滚完成，Crash率恢复
         → 订单量逐步回升

T+30min：事态平稳，启动Root Cause分析

T+24h：发布事后复盘报告
       → 补充iOS灰度发布到Crash监控的自动告警链路
```

## 面试常见追问

### 追问1：「告警太多导致团队麻木了怎么办？」

**参考回答**：
"告警疲劳是监控系统最危险的退化模式。我会做三件事：
1. **告警审计**：拉出过去30天的所有告警，统计每条的实际行动率。行动率低于10%的告警要么提升阈值，要么降级为日报
2. **聚合降噪**：同一根因触发的多条告警应该聚合为一条。比如DB故障导致10个服务报错，应该只收到一条根因告警
3. **分层路由**：P0电话轰炸，P3只进日报。确保工程师手机不会被低优先级告警淹没。"

### 追问2：「业务指标和技术指标怎么统一管理？」

**参考回答**：
"我倾向于建立一个**指标分层树（Metric Tree）**：

- L0 北极星指标（如GMV）
  - L1 业务指标（如订单量 × 客单价）
    - L2 漏斗指标（如浏览→下单→支付的转化率）
      - L3 技术指标（如支付接口延迟、错误率）

当L3的技术指标异常时，可以通过指标树快速追溯到对L0北极星指标的影响程度，从而决定告警级别。这样既不会让业务方看不懂技术告警，也不会让工程师忽略业务影响。"

### 追问3：「PM在监控体系中的角色边界是什么？」

**参考回答**：
"PM不是值班SRE，但PM承担三个独特角色：
1. **业务影响评估者**：工程师发现异常后，PM判断'这对用户和业务的影响有多大'，决定是否升级
2. **用户沟通决策者**：是否需要公告？是否需要补偿？这是PM的职责
3. **复盘改进推动者**：事后复盘不能只讨论技术修复，PM要推动产品层面的改进——比如增加降级方案、优化用户提示
PM的视角确保监控系统服务于'用户体验保障'，而不仅仅是'系统可用性'。"

## 标准回答框架

```
1. 建立层次认知
"监控体系分三层：实时告警、日常监控、长期追踪。
面试题描述的'3天后才知道'说明实时告警层缺失。"

2. 识别关键旅程
"首先确定产品的关键用户旅程，
对每个环节定义成功率、延迟、错误率三类指标。"

3. 设计阈值策略
"核心指标用固定阈值（SLA约束），
波动指标用动态阈值（同比/环比异常检测）。
关键是平衡误报和漏报的代价。"

4. 建立响应机制
"告警分级（P0-P3）+ 路由到正确的人 + 
每条告警配有Runbook + 升级机制。"

5. 持续迭代
"定期审计告警信噪比，
推动指标树建设，
确保告警体系越用越精准。"
```

## 关键收获

1. **告警的本质是决策触发器**：一条好告警必须让人知道"出了什么问题"和"我需要做什么"
2. **层次思维很关键**：不是所有指标都需要实时监控，过度监控比监控不足更危险
3. **动态阈值是进阶能力**：同比环比、季节性分解能大幅减少无效告警
4. **Runbook是活文档**：每次事故后更新Runbook，是组织知识沉淀的核心
5. **PM的独特价值**：在技术告警和业务影响之间架起桥梁，确保系统服务于用户体验
