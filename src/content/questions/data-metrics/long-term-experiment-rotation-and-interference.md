## 面试场景

> 面试官：你们的 App 同时在跑 8 个 A/B 测试——首页改版、推荐算法升级、定价策略调整、注册流程优化等等。产品经理跑来说"注册流程实验显示提升了 15%"，但你发现推荐算法实验组恰好也改了新用户引导逻辑。你怎么判断这个 15% 是否可信？如果不可信，你如何设计实验体系来避免这类问题？

这道题考察的是你对**实验干扰（Experiment Interference）**的理解——当多个实验同时运行时，一个实验的处理可能污染另一个实验的结论。这是中高级 PM 和数据驱动型组织的核心挑战，也是面试中区分"懂实验"和"真懂实验"的分水岭。

## 核心概念：实验干扰的类型

### 1. 直接干扰（Direct Interference）

一个实验的处理直接影响另一个实验的指标。

**例子**：实验 A 测试"免费试用期从 7 天延长到 14 天"，实验 B 测试"注册页减少一步"。实验 B 的转化率提升可能部分来自实验 A 带来的"14 天试用"更有吸引力——用户看到更长的试用期所以更愿意注册。

### 2. 间接干扰（Indirect Interference）

两个实验不直接影响同一指标，但通过共享用户池产生关联。

**例子**：实验 A 改变推荐算法（影响用户停留时长），实验 B 改变付费墙触发时机（影响付费转化）。如果实验 A 让用户停留更久，用户可能更容易触发付费墙——实验 B 的效果就被放大了。

### 3. 运营干扰（Operational Interference）

实验本身不干扰，但运营活动（促销、推送、内容更新）与实验重叠，导致归因混乱。

**例子**：你在跑定价 A/B 测试的同一周，市场部做了大型促销推送——促销可能抵消或放大定价实验的效果。

### 4. 交互效应（Interaction Effect）

实验 A 和实验 B 的组合效果 ≠ 各自效果之和。

数学表达：
- A 的主效应 = +5%
- B 的主效应 = +3%
- A + B 同时上 = +12%（协同效应）或 +2%（抵消效应）

这就是**交互效应**——两个实验同时存在时，效果不是简单叠加。

## 方法论：实验隔离与检测框架

### 方法一：正交分层（Orthogonal Layering）

将实验分配分成多个"层"（Layer），每层独立分配流量：

```
层1（UI 层）：首页改版实验     → 50% 控制 / 50% 实验
层2（算法层）：推荐算法实验    → 50% 控制 / 50% 实验
层3（商业化层）：定价策略实验  → 50% 控制 / 50% 实验
```

**关键原理**：同一用户在不同层被独立随机分配。层1的分配结果不影响层2的分配结果——这就是"正交"。

**优点**：每层的实验可以独立分析，互不干扰。
**前提**：实验之间确实不存在交互效应。如果存在交互，正交分层无法检测到。

### 方法二：因子设计（Factorial Design）

当怀疑两个实验可能存在交互时，故意同时运行并测量组合效果：

```
2×2 因子设计：
         B=控制     B=实验
A=控制     基线      仅B
A=实验     仅A      A+B

分析四个组的指标，可以分解出：
- A 的主效应
- B 的主效应
- A×B 交互效应
```

**适用场景**：当你有理由相信两个实验可能存在交互（如都影响新用户的核心路径）。

**成本**：需要 4 个分流组，实验周期更长。

### 方法三：序贯实验（Sequential Experiments）

一次只跑一个实验，跑完再跑下一个。

**优点**：完全消除实验间干扰。
**缺点**：实验吞吐量低，不适合高速迭代的产品。

**适用场景**：重大改动（如核心商业模式变更）、样本量充足、可以等待。

### 方法四：互斥分组（Mutually Exclusive Groups）

将两个可能有交互的实验放在互斥的流量分组中：

```
分组 A（50%流量）：跑实验1（首页改版）
分组 B（50%流量）：跑实验2（推荐算法）
→ 两个实验的用户完全不重叠
```

**与正交分层的区别**：互斥分组是"同一批用户只参与一个实验"，正交分层是"同一用户在不同层独立分配"。

## 实战案例

### 案例：诊断"15% 提升"是否可信

回到面试场景——注册流程实验 + 推荐算法实验可能存在干扰。

**诊断步骤：**

#### Step 1：检查流量分配

```
推荐算法实验是否覆盖了注册流程实验的用户？
- 如果两个实验在同一层（非正交），可能存在干扰
- 如果在不同层（正交分配），理论上独立
```

#### Step 2：检查时间重叠

```
两个实验的时间窗口是否重叠？
- 推荐算法实验何时开始？结束？
- 注册流程实验何时开始？结束？
- 如果只有部分时间重叠，对比重叠和非重叠期的效果差异
```

#### Step 3：检查用户路径交叉

```
推荐算法实验是否影响了新用户引导逻辑？
- 如果推荐算法的变体改变了新用户首次体验
  （比如新用户看到更好的推荐，提高了"完成注册"的动力）
  → 这就是交互效应
```

#### Step 4：设计验证实验

```
将用户分成四组：
A组：注册控制 + 推荐控制（基线）
B组：注册实验 + 推荐控制（仅注册改动）
C组：注册控制 + 推荐实验（仅推荐改动）
D组：注册实验 + 推荐实验（两者同时改动）

对比四组数据：
- B vs A = 注册改动的主效应
- C vs A = 推荐改动的主效应
- D vs A = 组合效应
- (D-A) vs (B-A)+(C-A) = 交互效应
  → 如果相等：无交互
  → 如果大于：协同效应
  → 如果小于：抵消效应
```

### 案例结论

如果发现交互效应显著：
1. **不要单独采纳注册实验的 15% 结论**——这个数字可能在推荐实验的全量上线后缩水
2. **重新评估上线顺序**——哪个先上、哪个后上会影响最终效果
3. **考虑合并上线**——如果交互是正向的，两个改动一起上效果更好

如果交互效应不显著：
1. **15% 可信**——可以独立决策
2. **后续按正交分层继续跑**——不需要担心干扰

## 实验轮换策略

### 为什么需要轮换

长期运行的实验面临两个问题：
1. **疲劳效应**：用户长期处于实验状态，行为可能从"认真对待"变为"忽略"
2. **沾染效应**：实验组的用户可能将新功能分享给对照组用户（口碑传播）

### 轮换方案设计

**方案一：固定面板轮换**
- 将 100% 流量分成 4 个面板
- 每个面板轮流承担实验任务
- 每 4 周轮换一次

**方案二：新鲜用户注入**
- 持续引入新用户进入实验
- 对比新用户和老用户的实验效果差异
- 如果效果差异大，说明存在疲劳/沾染效应

**方案三：留出"休息期"**
- 实验跑 2 周，停 1 周，再跑 2 周
- 对比"休息期"前后的指标变化

## 实验平台治理建议

作为 PM，你应该推动团队建立以下实验规范：

1. **实验注册制度**：每个实验必须注册 hypothesis、指标、预期效果、所在层
2. **交互效应预评估**：新实验上线前，评估与正在运行的实验是否存在潜在交互
3. **层分配策略**：按功能域分层（UI层、算法层、商业化层），同层实验互斥
4. **定期审计**：每月检查是否有"僵尸实验"（跑了太久但没人看结果）
5. **结果归档**：实验结束后，记录最终结论和是否有交互效应，为未来提供参考

## 面试常见追问

### 追问1："你怎么知道两个实验有交互效应？"

检测方法：
- **统计检验**：在因子设计中，对交互项做 t 检验，看 p 值
- **差异对比**：如果实验 A 在"实验 B 的实验组"和"实验 B 的对照组"中效果不同，说明存在交互
- **业务判断**：如果两个实验影响同一用户旅程的关键节点，大概率有交互

### 追问2："如果实验平台不支持正交分层怎么办？"

实际操作：
- 用哈希函数将用户ID分组，确保不同实验的分组独立
- 如果连哈希分流都做不到，退而求其次用互斥分组
- 最差情况下序贯运行——牺牲吞吐量换准确性

### 追问3："交互效应的大小有阈值吗？"

经验法则：
- 交互效应 < 主效应的 10%：可以忽略，独立分析
- 交互效应 10%-30%：需要注意，在汇报时注明
- 交互效应 > 30%：必须重新设计实验，不能独立下结论

### 追问4："上线时如何处理已知有交互效应的实验？"

决策框架：
1. 如果是协同效应（1+1>2）：优先同时上线
2. 如果是抵消效应（1+1<2）：需要选择其一，或找新的组合方案
3. 如果不确定长期效果：先全量一个，稳定后再测试另一个

## 标准回答框架（面试时）

```
1. 识别问题本质（30秒）
   "这个场景的核心风险是实验干扰——具体来说可能是交互效应"

2. 诊断方法（1分钟）
   "我会分四步诊断……"
   → 检查流量分配、时间重叠、路径交叉、设计验证实验

3. 解决方案（1分钟）
   "如果确认存在干扰，我的方案是……"
   → 因子设计、正交分层、或序贯实验

4. 预防体系（30秒）
   "长期来看，我会推动建立……"
   → 实验平台治理、层分配策略、交互预评估

5. 沟通策略（30秒）
   "我会怎么跟 PM 沟通这个 15%……"
   → 不是否定，而是"让我们验证一下这个结论的稳健性"
```

## 技术转 PM 特别提示

技术背景的 PM 在实验设计上有天然优势（理解统计原理），但容易陷入以下误区：

1. **过度关注统计严谨性，忽略决策效率**：不是所有决策都需要完美实验——有时候快速上线比精确测量更重要。
2. **忽视业务逻辑层面的交互**：统计上正交不代表业务上独立——两个实验可能通过用户心理产生关联。
3. **不善于用简单语言解释复杂概念**：你需要把"交互效应"解释为"两个改动撞在一起的效果不是简单叠加"，让非技术背景的决策者理解。

## 关键收获

1. **多实验并行是现代产品团队的常态**——实验干扰不是例外，而是必然，关键在于你是否检测到它
2. **正交分层是基础设施，因子设计是武器**——前者防止大部分干扰，后者用来诊断剩余的交互
3. **实验轮换解决时间维度的污染**——长期实验需要新鲜血液和休息期
4. **实验治理是组织能力**——不是单个 PM 的事，需要团队建立规范和流程
5. **诚实面对不确定性**——当结论可能被污染时，说出来，而不是假装数字是确定的
