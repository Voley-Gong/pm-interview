# 如何构建指标树并进行归因拆解

## 面试场景

> 面试官问："你负责的一款SaaS产品，本月MRR（月经常性收入）环比下降了15%。请描述你会如何拆解这个问题，找到根本原因。"

这道题考察的不是"知道答案"，而是你是否有系统化的指标拆解能力。很多候选人会东猜一个原因（"可能是流失增加了？"），而优秀的PM会展示出一套结构化的归因方法。

---

## 一、核心概念：什么是指标树

### 定义

指标树（Metric Tree）是一种将核心业务指标按业务逻辑逐层拆解的可视化工具：

```
                    北极星指标
                   /          \
            一级指标            一级指标
           /    |    \          /    \
       二级指标 二级指标 二级指标  二级指标 二级指标
```

### 指标树的三个层级

| 层级 | 含义 | 特征 | 示例（电商） |
|------|------|------|-------------|
| L0 - 北极星 | 公司级核心目标 | 唯一、稳定 | GMV |
| L1 - 业务线 | 按业务公式拆解 | 互不重叠 | 流量 × 转化率 × 客单价 |
| L2 - 可执行 | 团队能直接影响的指标 | 可归因、可干预 | 各渠道流量、各步骤转化率 |

### 为什么需要指标树

1. **诊断异常**：当北极星指标波动时，能快速定位到哪个环节出了问题
2. **对齐目标**：不同团队清楚自己的指标如何影响全局
3. **避免局部优化**：防止某团队优化自己的指标但损害了全局

## 二、构建指标树的框架

### 框架一：公式拆解法（自上而下）

适用于有明确数学关系的业务模型。

**电商GMV拆解示例：**

```
GMV
 = 活跃用户数 × 下单转化率 × 客单价 × 复购频次
 
活跃用户数
 = 新用户 + 老用户留存 + 唤醒用户
 = (各渠道获客) + (各生命周期留存) + (召回活动)

下单转化率
 = 浏览→加购率 × 加购→下单率
 = (搜索转化 + 推荐转化 + 活动转化)

客单价
 = 品类结构 × 单品价格 × 满减策略影响
```

### 框架二：用户旅程拆解法

适用于流程型产品（SaaS、工具类）。

```
MRR
 = 客户数 × ARPU

客户数
 = 新签 + 续费 - 流失

新签路径：
  市场线索 → MQL → SQL → 试用 → 成交
  (每步都有转化率)

续费路径：
  到期客户数 × 续费率
  续费率受：产品使用深度 + 客户成功介入 + 竞品替代 影响

流失分析：
  主动流失（竞品切换）vs 被动流失（预算缩减、倒闭）
```

### 框架三：维度拆解法（交叉验证）

当公式拆解定位到某层后，用维度切分进一步定位：

| 维度 | 切分方式 | 诊断问题 |
|------|---------|---------|
| 时间 | 按日/周/月对比 | 是否某个时间点突变？ |
| 渠道 | 按获客渠道拆 | 是否某个渠道质量下降？ |
| 用户群 | 按新老/付费层级 | 是新用户问题还是老用户？ |
| 功能 | 按功能模块使用 | 是否某功能改动影响？ |
| 平台 | 按iOS/Android/Web | 是否某端有bug？ |

## 三、实战案例：MRR下降15%的归因拆解

### Step 1：确认数据准确性（5分钟内）

- 数据口径是否一致？（是否有定义变更）
- 数据管道是否完整？（是否丢数）
- 是否有季节性因素？（对比去年同期）

### Step 2：公式拆解定位（30分钟）

```
MRR = 客户数 × ARPU

→ 客户数环比下降 8%
→ ARPU 环比下降 7%
→ 双因素下降，需要分别归因
```

### Step 3：客户数下降归因

```
客户数 = 新签 + 续费 - 流失

→ 新签：环比持平 ✓
→ 续费率：从 85% 降到 72% ⚠️
→ 流失：环比增加 40% 🔴
```

**进一步拆解流失：**
- 按客户分层：中小企业流失率从 12% 升到 22%
- 按使用时长：使用<3个月的客户流失率最高
- 按功能使用：未使用"自动化报表"功能的客户流失率是使用者的3倍

### Step 4：ARPU下降归因

```
ARPU = 套餐分布 × 各套餐价格

→ 高级套餐客户占比从 30% 降到 22%
→ 中端套餐客户数持平
→ 低端套餐客户增加（新签集中在低端）
```

### Step 5：交叉验证找到根因

结合客户数和ARPU的分析：
- **根因假设**：上月调整了定价策略，高级套餐涨价20%，导致一批中小企业客户降级或流失
- **验证方式**：对比涨价前后的客户行为数据，确认因果关系

### Step 6：输出诊断报告

```
诊断结论：定价策略调整导致中小企业客户降级/流失
影响范围：贡献了MRR下降的 11%/15%
紧急行动：
  1. 对未续费的高价值客户启动一对一挽回
  2. 评估是否推出中间档套餐填补价格断层
  3. 暂停涨价策略A/B测试，评估长期影响
```

## 四、面试常见追问

### 追问1："如果拆解后发现多个指标都下降了，你怎么判断哪个是主因？"

**回答框架：贡献度分析**

- 计算每个子指标变化对北极星指标的贡献比例
- 公式：贡献度 = (子指标变化量 × 该子指标的权重) / 北极星总变化量
- 示例：流量下降贡献了30%，转化率下降贡献了50%，客单价下降贡献了20%→主因是转化率

### 追问2："指标树和仪表盘有什么区别？"

- **仪表盘**是数据的展示工具，告诉你"发生了什么"
- **指标树**是数据的逻辑结构，告诉你"为什么发生"
- 仪表盘可以有几百个指标，指标树通常只有3-4层、15-20个核心指标
- 好的仪表盘应该按照指标树的结构来组织

### 追问3："如果两个子指标之间有相关性，指标树怎么处理？"

- 理想情况下，同一层级的指标应该互斥（MECE原则）
- 实际中完全互斥很难做到，可以接受弱相关
- 如果强相关，考虑合并为一个综合指标，或者用加权方式拆分
- 标注清楚指标间的关系，避免团队重复claim功劳

### 追问4："你怎么确保指标树是最新的？"

- 指标树应该随业务模式变化而更新，通常每季度review一次
- 业务模式变化信号：新增了收入线、目标用户群扩展、商业模式调整
- 维护方式：指定一个PM owner，在季度业务review时同步更新

## 五、标准回答框架

面试时遇到指标拆解类问题，按以下步骤回答：

1. **确认问题边界**（30秒）："先确认一下，这个下降是从什么时候开始的？有没有同时做过什么产品或运营调整？"

2. **展示拆解框架**（1分钟）："我会用一个三层指标树来拆解。第一层按收入公式，第二层按用户旅程，第三层按维度交叉验证。"

3. **执行拆解过程**（2-3分钟）：边拆边讲逻辑，每一层说出拆解依据

4. **定位根因**（1分钟）："综合来看，最可能的根因是XX，因为它同时解释了多个指标的异常。"

5. **提出行动建议**（1分钟）：区分短期止血和长期改进

## 六、关键收获

1. **指标树是PM的基础功**：没有指标树，所有数据分析都是"拍脑袋猜原因"
2. **公式拆解 > 维度拆解**：先按业务公式找到数学关系，再用维度交叉验证
3. **MECE原则**：同层指标尽量不重叠、不遗漏
4. **归因不是终点，行动才是**：拆解出根因后，必须有对应的行动方案
5. **数据先验证再分析**：花5分钟确认数据准确性，避免在错误数据上浪费一小时
