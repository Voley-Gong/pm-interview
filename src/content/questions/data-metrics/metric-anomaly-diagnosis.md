# 核心指标突然下降20%，你如何排查？

## 面试场景

> 面试官："周一早上你打开数据看板，发现你们产品的日活跃用户数相比上周五下降了20%。请告诉我你会怎么排查这个问题。"

这是PM面试中**最高频的数据题之一**。它考察的不是你会不会写SQL，而是你有没有一套**系统化的归因思维框架**——能否在压力下有条不紊地定位问题根因。

---

## 核心概念：指标异动诊断

指标异动诊断（Metric Anomaly Diagnosis）是指当核心业务指标发生显著偏离预期值的变化时，通过系统化的分析方法锁定根因的过程。

关键在于：**不是所有下降都是"问题"，有些是正常波动，有些是外部因素导致的不可控变化。** PM的职责是快速区分这些情况。

---

## 排查框架：五层下钻法

### 第一层：确认数据真实性（Data Integrity Check）

**先别慌，先验真。**

| 检查项 | 具体操作 |
|--------|----------|
| 数据管道是否正常 | 检查数据上报是否中断、ETL任务是否失败 |
| 统计口径是否变更 | 近期是否有埋点修改、口径调整 |
| 对照基线是否合理 | 上周五的数据本身是否异常偏高（如节假日效应） |
| 前端展示是否出错 | 看板本身是否有bug |

> 💡 **面试加分点**：很多候选人直接跳到分析用户行为，但资深PM会先排除"数据本身就有问题"的可能性。大约30%的"指标异动"最终发现是数据管道问题。

### 第二层：拆解维度，定位范围（Dimensional Breakdown）

确认数据真实后，用**维度下钻**锁定异动的范围：

**常用拆解维度：**

1. **时间维度**：是某一天断崖式下跌，还是逐步下降趋势的延续？
2. **平台维度**：iOS / Android / Web / 小程序——是否只在某一端下降？
3. **渠道维度**：自然流量 / 付费投放 / 社交分享——哪个渠道变化最大？
4. **地域维度**：是否只在某些地区/城市出现？
5. **用户分层**：新用户 vs 老用户 / 免费用户 vs 付费用户
6. **功能模块**：哪个功能页的访问/使用下降最多？

**实操方法**：对每个维度计算"贡献度"（Contribution to Change），找出**贡献了大部分下降的子维度**。

```
示例：
DAU总下降 = iOS端下降(-5%) + Android端下降(-15%)
其中Android端：新用户下降(-12%)，老用户下降(-3%)
→ 根因大概率在Android端新用户获取环节
```

### 第三层：区分内外因（Internal vs External）

| 内部因素 | 外部因素 |
|----------|----------|
| 新版本发布引入bug | 竞品发布重大更新/促销 |
| 服务端宕机/接口异常 | 节假日/特殊日期效应 |
| 运营活动结束（如补贴到期） | 行业政策变化 |
| 推荐算法调整 | 天气/社会事件 |
| 支付通道故障 | 应用商店排名变化 |

**排查技巧**：建立时间线——把内部产品变更和外部事件按时间排列，与指标变化做关联。

### 第四层：假设验证（Hypothesis Testing）

根据前三层的发现，形成具体假设，用数据验证：

- **假设1**："Android v3.2.1版本存在启动崩溃" → 查该版本崩溃率
- **假设2**："周末效应导致自然流量下降" → 对比历史周末数据
- **假设3**："竞品大促活动抢占了用户" → 查竞品活动时间线与自身用户重叠

**验证原则**：一个假设必须能用数据证伪。如果无法证伪，说明假设不够具体。

### 第五层：形成结论与行动计划

输出结构化结论：

```
【根因】Android v3.2.1版本在小米机型上存在内存泄漏，导致启动崩溃率从0.5%飙升至8%
【影响范围】影响Android新用户约12万，占DAU下降的85%
【紧急行动】1. 热修复内存泄漏 2. 对小米机型回滚到v3.2.0
【后续预防】1. 增加低端机型兼容性测试 2. 设置崩溃率自动告警阈值
```

## 实战案例

### 案例：某电商App订单转化率突降

**背景**：某电商App在某周三开始，订单转化率从3.2%下降到2.5%，持续3天。

**排查过程**：

1. **数据验真** ✅ 数据管道正常，埋点无变更
2. **维度拆解**：发现iOS端转化率正常(3.1%)，Android端从3.3%降至2.1%
3. **进一步下钻**：Android端中，华为机型转化率从3.5%暴降至1.2%
4. **内外因排查**：发现周二发布了Android热更新，修改了支付页面的WebView组件
5. **假设验证**：华为机型的WebView存在兼容性问题，支付页面白屏，用户无法完成下单
6. **结论**：热更新引入的WebView兼容性问题导致华为用户支付失败

**解决**：紧急回滚热更新，华为机型转化率恢复。

## 面试常见追问

### 追问1："如果维度拆解后发现是整体均匀下降，没有明显单一维度异常，怎么办？"

**回答思路**：这说明可能是系统性因素。需要：
- 检查是否有全局性的产品变更（如首页改版、核心链路调整）
- 检查服务端整体性能（API响应时间、错误率）
- 排查第三方依赖（CDN、推送服务、支付网关）
- 考虑宏观因素（如竞品集中投放期、季节性波动）

### 追问2："如何建立指标异动的自动预警机制？"

**回答思路**：
- 设置合理的阈值（绝对值 + 相对变化率双维度）
- 区分"预期异动"（如运营活动结束）和"非预期异动"
- 建立分级告警（P0/P1/P2），不同级别触发不同响应流程
- 结合同环比、移动平均值来判断是否为真异动

### 追问3："如果排查了一周还是找不到原因怎么办？"

**回答思路**（体现成熟度）：
- 评估影响程度：如果业务影响可控，不要无限投入排查资源
- 建立监控持续观察：设置告警，一旦有新线索立即介入
- 并行推进止损措施：即使根因未明，也可以做防御性优化
- 组织复盘：把已知信息整理成文档，邀请更多团队brainstorm
- 诚实面对不确定性："目前我们排除了A/B/C因素，锁定在D/E范围内，但尚未确认根因"

## 标准回答框架（面试用）

面试时按以下结构回答，展示系统性思维：

```
1. 先验真（30秒）
   "首先确认数据本身是否准确——检查数据管道、埋点、口径是否有变更。"

2. 分层拆解（60秒）
   "确认数据真实后，我会从平台、渠道、用户类型、时间等维度做下钻分析，
    找到贡献最大变化的子维度。"

3. 内外归因（30秒）
   "同时排查内部因素（版本发布、服务异常）和外部因素（竞品、节假日、政策）。"

4. 假设验证（30秒）
   "基于以上分析形成具体假设，用数据验证或证伪。"

5. 行动闭环（30秒）
   "确认根因后，制定短期修复和长期预防方案，并做事后复盘。"
```

## 常见误区

| 误区 | 正确做法 |
|------|----------|
| 上来就猜原因 | 先验真、再拆解、最后假设 |
| 只看单一维度 | 多维度交叉下钻 |
| 忽略外部因素 | 同时考虑内外因 |
| 找到原因就结束 | 还要制定预防方案，形成闭环 |
| 追求100%确定性 | 评估ROI，合理分配排查资源 |

---

## 关键收获

1. **"先验真再分析"是分水岭**——30%的异动是数据本身的问题
2. **维度下钻是核心技能**——用贡献度排序，优先看影响最大的子维度
3. **假设必须可证伪**——"可能是用户不喜欢了"不是有效假设
4. **闭环思维**——从发现→排查→确认→修复→预防→复盘，缺一不可
5. **沟通能力同样重要**——排查过程中需要及时同步关键干系人，管理预期
