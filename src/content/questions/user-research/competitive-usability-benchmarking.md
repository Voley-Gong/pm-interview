## 面试场景

> 面试官：「你负责一款B2B SaaS项目管理工具，市场上已有3个主要竞品。老板要求你在两周内产出一份数据驱动的竞品可用性对比报告，作为下一季度产品优化的参考。你会如何设计这个评估？」

---

## 核心概念解析

### 什么是竞品可用性基准评估？

竞品可用性基准评估（Competitive Usability Benchmarking）是通过**统一的标准、任务和度量**，对自家产品和竞品进行系统化的可用性测量，从而获得可比较的量化数据。

与简单的「竞品功能对比表」不同，可用性基准评估关注的是**用户在真实任务场景下的表现和体验**，而非功能的有无。

### 关键区别

| 维度 | 功能对比表 | 可用性基准评估 |
|------|-----------|---------------|
| 关注点 | 有没有 | 好不好用 |
| 数据类型 | 功能清单 | 任务完成率、时间、错误数、满意度 |
| 主观性 | 高（ checklist 谁都能列） | 低（标准化任务+量化指标） |
| 可操作性 | 低（知道缺什么，不知道怎么改） | 高（精确定位体验瓶颈） |

## 评估框架：CUBA 模型（Competitive Usability Benchmarking Assessment）

### Step 1: 定义评估维度

从 Nielsen 可用性启发式出发，选取与产品类型最相关的维度：

```
                ┌─────────────────────┐
                │   评估维度选取       │
                └──────────┬──────────┘
                           │
        ┌──────────────────┼──────────────────┐
        │                  │                  │
   ┌────▼─────┐     ┌─────▼────┐     ┌──────▼─────┐
   │ 效率      │     │ 有效性    │     │  满意度     │
   │ Efficiency│     │Effectiveness│   │Satisfaction│
   └────┬─────┘     └─────┬────┘     └──────┬─────┘
        │                 │                  │
   · 任务完成时间    · 任务完成率       · SUS 问卷
   · 操作步数        · 错误率           · NASA-TLX
   · 学习曲线        · 辅助请求次数     · 主观净推荐
```

### Step 2: 设计标准化任务

任务设计是评估质量的关键。需要遵循 **SMART-T 原则**：

- **S**pecific：任务目标明确，无歧义
- **M**easurable：成功标准可量化
- **A**ctionable：用户知道从哪里开始
- **R**ealistic：贴近真实使用场景
- **T**imed：有合理的时间预期
- **T**ransferable：在所有竞品上均可执行

**示例任务集（B2B项目管理工具）：**

| 序号 | 任务描述 | 成功标准 | 难度 |
|------|---------|---------|------|
| T1 | 创建一个新项目并邀请3名成员 | 项目创建成功，成员收到邀请 | 低 |
| T2 | 在项目中创建看板并添加5个任务卡片 | 看板+5卡片存在 | 中 |
| T3 | 设置任务的依赖关系并切换到甘特图视图 | 依赖关系正确显示在甘特图 | 高 |
| T4 | 导出项目进度报告为PDF | 成功下载PDF文件 | 中 |
| T5 | 配置自动化规则：任务完成时通知负责人 | 规则创建并触发一次 | 高 |

### Step 3: 选取评估方法

**方法A：专家启发式评估（Heuristic Evaluation）**

适用场景：预算有限、时间紧张

- 3-5名UX专家独立评估
- 使用 Nielsen 10 启发式或自定义维度
- 每个问题打分：严重程度 0-4
- 合并去重，计算每个竞品的平均严重度

**方法B：用户可用性测试（Moderated Usability Testing）**

适用场景：需要真实用户数据支撑

- 每个竞品招募 5-8 名目标用户（总 15-32 人）
- 平衡顺序效应（Latin Square 设计）
- 每个任务记录：完成率、时间、错误数
- 测试后填写 SUS（System Usability Scale）问卷

**方法C：自动化分析工具辅助**

- 使用 Hotjar/FullStory 录屏分析（仅限自家产品）
- AI 辅助启发式评估（如 Galileo AI 等工具）
- 趋势分析：SimilarWeb / Sensor Tower 的用户行为数据

### Step 4: 数据收集与分析

#### 数据矩阵模板

```
           │ 竞品A │ 竞品B │ 竞品C │ 自家产品
───────────┼──────┼──────┼──────┼─────────
T1完成率    │ 95%  │ 88%  │ 100% │  82%
T1平均时间  │ 45s  │ 72s  │ 38s  │  95s
T1错误数    │ 0.2  │ 0.5  │ 0.1  │  1.1
───────────┼──────┼──────┼──────┼─────────
T2完成率    │ ...  │ ...  │ ...  │  ...
...
───────────┼──────┼──────┼──────┼─────────
SUS 总分    │ 78   │ 65   │ 82   │  59
```

#### 分析方法

1. **雷达图对比**：将各维度归一化后绘制雷达图，直观展示优劣势
2. **差距热力图**：计算自家与最佳竞品的差距百分比，标红关键落后项
3. **体验加权评分**：根据业务优先级加权（如效率 40%、有效性 35%、满意度 25%）

### Step 5: 输出与行动

评估报告应包含：

```
┌─────────────────────────────────────────┐
│           竞品可用性基准报告             │
├─────────────────────────────────────────┤
│ 1. Executive Summary（一页纸结论）      │
│ 2. 评估方法说明                          │
│ 3. 关键发现：                            │
│    - 🟢 领先项（比竞品强的地方）         │
│    - 🟡 持平项（与竞品相当）             │
│    - 🔴 落后项（急需改进）               │
│ 4. 优先级排序的改进建议                  │
│ 5. 附录：完整数据矩阵                    │
└─────────────────────────────────────────┘
```

## 实战案例

### 某协同文档产品的竞品可用性评估

**背景**：国内某协同文档产品（类似飞书文档）面临 Notion 和 语雀的竞争压力。

**评估发现**：
- 🔴 **多光标协作延迟**：自家产品在 3 人以上同时编辑时延迟达 800ms，竞品 A 仅 200ms
- 🔴 **首次使用引导**：新用户完成第一个文档创建平均需要 4.2 分钟，竞品 B 仅需 1.8 分钟
- 🟢 **评论体验**：行内评论和 @ 提及的流畅度优于两个竞品
- 🟡 **模板库丰富度**：与竞品 A 持平，但模板分类体验略差

**行动转化**：
1. Q1 优先解决协作延迟（技术优化）
2. Q2 重新设计新手引导流程（Onboarding 2.0）
3. Q3 优化模板分类体系

## 面试常见追问

### 追问1：「只有两周时间和有限预算，你怎么选方法？」

**回答策略**：
优先采用 **专家启发式评估 + 轻量用户测试** 的组合：

1. 第 1-3 天：3 名内部 UX 专家对所有竞品执行启发式评估
2. 第 4-7 天：用内部员工（非产品团队）作为代理用户，每竞品测 5 人
3. 第 8-10 天：数据分析 + 报告撰写
4. 第 11-14 天：内部评审 + 修订

**关键认知**：内部员工不是理想的目标用户，但足以发现主要的可用性问题（Nielsen 的「5 用户法则」）。

### 追问2：「竞品不断更新，评估结果多久有效？」

**回答框架**：
- 可用性基准是**时间快照**，建议每 6 个月做一次全面评估
- 关键竞品的大版本更新后，做增量评估（只测变更部分）
- 建立**可复用的评估体系**（任务清单、评分标准），降低重复成本
- 在报告中标注评估时间，避免使用过期数据做决策

### 追问3：「老板只想要一个排名，你怎么回应？」

**回答策略**：
1. **理解诉求**：老板要的可能不是学术级报告，而是决策依据
2. **提供分层信息**：一页纸 Executive Summary 给出加权总分排名 + 关键差距
3. **避免过度简化**：说明每个维度的相对差异比单一排名更有价值
4. **引导行动**：「排名第三不代表全面落后，我们在X维度其实领先，只是Y维度拉低了总分」

### 追问4：「如何处理不同竞品用户群体的差异？」

**回答框架**：
- 理想情况下，所有测试用户应是同一目标人群
- 如果竞品面向不同用户群，需要分层分析：
  - 群体A 在竞品X vs 自家产品上的表现
  - 群体B 在竞品Y vs 自家产品上的表现
- 不要用竞品X的资深用户和竞品Y的新用户做直接对比

## 标准回答框架

```
1. 澄清目标（30秒）
   "这份报告是给谁看？解决什么决策问题？"

2. 方法选择（30秒）
   "基于时间和预算，我建议用 [方法] 组合"

3. 评估设计（1分钟）
   "我会定义 N 个标准化任务，覆盖 [效率/有效性/满意度] 三个维度"

4. 执行计划（30秒）
   "两周时间分配：设计3天 + 执行5天 + 分析4天 + 报告2天"

5. 预期产出（30秒）
   "交付物包括：差距热力图、优先级改进清单、一页纸摘要"
```

## 关键收获

1. **竞品可用性评估 ≠ 功能对比表**：核心是用户在标准化任务上的真实表现
2. **任务设计决定评估质量**：SMART-T 原则确保任务可执行、可比较
3. **专家评估 + 用户测试的组合最实用**：兼顾效率和可信度
4. **数据要转化为行动**：报告的价值不在于发现差距，而在于指导改进优先级
5. **评估体系要可复用**：建立标准化任务库和评分体系，降低后续评估成本
6. **对老板要用「分层沟通」**：一页纸摘要 + 详细附录，满足不同读者的需求
