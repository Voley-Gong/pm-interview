## 什么是可用性测试

可用性测试（Usability Testing）是**让真实用户在观察下完成特定任务**，从而发现产品中的操作障碍、认知摩擦和体验断点。

它不是"问用户喜欢什么"——用户说的和做的往往不一样。它是**观察用户实际操作**，看哪里卡住了、哪里犹豫了、哪里放弃了。

面试官通过这道题考察：
- 你是否能**设计有效的测试任务**（而不是引导性提问）
- 你是否理解**样本量的科学依据**（为什么 5-8 个人就够了）
- 你能否**区分真实问题和个例**（什么情况下需要追加测试）
- 你知道**何时用哪种测试形式**（实验室、远程、 hallway）

## 可用性测试的类型

| 类型 | 适用场景 | 样本量 | 成本 | 深度 |
|------|----------|--------|------|------|
| **实验室 moderated** | 复杂产品、早期原型、需深挖原因 | 5-8 人 | 高 | ★★★★★ |
| **远程 moderated** | 跨地域用户、疫情后主流 | 5-8 人 | 中 | ★★★★ |
| **远程 unmoderated** | 验证已上线功能、大样本快速反馈 | 20-50 人 | 低 | ★★ |
| **Hallway（走廊测试）** | 快速验证、内部同事代替 | 3-5 人 | 极低 | ★★ |
| **First-click test** | 评估信息架构和导航 | 20+ 人 | 低 | ★★ |

**选择原则：**
- 探索性研究（为什么有问题）→ **moderated，少量深度**
- 验证性研究（问题修好了没）→ **unmoderated，大量快速**

## 核心方法：Nielsen 的"5 用户法则"

Jakob Nielsen 的经典研究结论：

> **5 个用户就能发现 85% 的可用性问题。**

数学原理（Nielson-Landauer 公式）：

```
问题发现率 = 1 - (1 - L)^N

其中：
- L = 单个用户发现的问题比例（通常约 31%）
- N = 测试用户数
```

| 用户数 | 发现问题比例 | 边际收益 |
|--------|-------------|---------|
| 1 | 31% | — |
| 3 | 67% | +36% |
| **5** | **85%** | **+18%** |
| 8 | 93% | +8% |
| 10 | 97% | +4% |

**结论：5 人之后的边际收益急剧递减。** 不如先测 5 人，修复问题后再测 5 人（迭代测试），而不是一次性测 20 人。

**例外情况需要更多样本：**
- 用户群体差异大（如同时有老年人和年轻人）→ 每类至少 3-4 人
- 产品极其复杂（如 ERP 系统）→ 分模块测试，每模块 5 人

## 完整流程：六步实操指南

### 第一步：确定测试目标

**不要试图在一次测试中验证所有东西。**

好的目标定义：
- ✅ "评估新用户能否在 3 分钟内完成首次发布商品的流程"
- ✅ "找出购物车结算环节的流失原因"
- ❌ "看看用户觉得我们的 App 怎么样"（太泛，无法设计任务）

### 第二步：招募用户

**招募原则：测试用户应代表你的目标用户群体。**

| 维度 | 考虑因素 |
|------|----------|
| 人口属性 | 年龄、性别、职业、城市 |
| 产品经验 | 新用户 vs 老用户（通常分别测） |
| 技术水平 | tech-savvy vs 普通用户 |
| 设备 | iOS / Android / Web |

**常见偏差：**
- 用内部同事测试 → 同质化严重，对产品太熟悉
- 只招朋友和家人 → 不是真实用户
- 招募太贵 → 可以用社区、用户群、甚至路上拦截

### 第三步：设计测试任务

**任务是可用性测试的核心。好的任务让用户"做"，而不是"想"。**

#### 任务设计原则

| 原则 | 说明 | 示例 |
|------|------|------|
| **具体而非抽象** | 给明确目标，不给操作步骤 | ✅ "找到一件 200 元以下的生日礼物送给你朋友" ❌ "浏览一下商城" |
| **不引导** | 不暗示正确路径 | ✅ "你想给这篇文章点赞就点" ❌ "请点击右下角的点赞按钮" |
| **有完成标准** | 能判断任务是否完成 | ✅ "成功下单并看到订单确认页" |
| **贴近真实** | 模拟真实使用动机 | ✅ "你听说这个 App 可以学英语，试着用一下" |

#### 经典任务模板

```
【场景导入】
"想象你最近想 [做 X 事情]，你听朋友说 [我们的产品] 可以帮你。"

【任务描述】
"请尝试用这个产品 [完成 Y 操作]。"

【放权声明】
"这不是考试，没有对错。出问题一定是产品的问题，不是你的问题。
你可以在任何地方停下来，也可以随时放弃。"

【思考法（Think-Aloud Protocol）】
"请在操作过程中把你心里的想法说出来——你在找什么、你期望看到什么、
你困惑什么。这能帮我们理解你的操作过程。"
```

### 第四步：执行测试

**主持人（Moderator）的核心技巧：**

| 该做 | 不该做 |
|------|--------|
| 保持沉默，让用户自己探索 | "你觉得这个按钮应该点吗？" |
| 观察非语言信号（犹豫、皱眉） | 只记录用户说的话 |
| 问"你在想什么？" | 问"你为什么点这里？"（会引发防御） |
| 记录"在哪里卡住了多久" | 只记录"最终完成了没有" |
| 鼓励 think-aloud | 在用户操作时解说"这是搜索框…" |

**关键观察指标：**

| 指标 | 说明 | 记录方式 |
|------|------|----------|
| **任务完成率** | 成功 / 失败 / 部分完成 | ✅❌⚠️ |
| **任务耗时** | 从开始到完成的总时间 | 秒表/录屏 |
| **错误数** | 走错路径、点错按钮 | 划线计数 |
| **求助次数** | 用户主动问"这个怎么弄" | 计数 |
| **主观满意度** | 任务后评分（SUS 量表等） | 问卷 |
| **行为信号** | 犹豫、回退、叹气、皱眉 | 录屏/笔记 |

### 第五步：分析数据

**问题分级矩阵：**

| 严重程度 | 定义 | 示例 | 优先级 |
|----------|------|------|--------|
| **P0 - 致命** | 用户完全无法完成任务 | 注册流程报错，无法进入 | 立即修复 |
| **P1 - 严重** | 大部分用户卡住，需要多次尝试 | 找不到关键按钮，导航混乱 | 本版本修复 |
| **P2 - 一般** | 部分用户困惑，但最终能完成 | 文案不清晰，需要思考几秒 | 下版本优化 |
| **P3 - 轻微** | 个别用户提到，影响不大 | 某个 icon 不够直观 | 有空再改 |

**关键原则：关注高频问题，而非个例。** 如果 5 个人中有 3 个在同一个地方卡住，那就是设计问题，不是用户问题。

### 第六步：输出报告与跟进

**可用性测试报告结构：**

```
1. 测试概述（目标、用户画像、方法）
2. 关键发现（3-5 个最重要的问题）
3. 问题详情（严重程度 + 录屏截图 + 用户原话）
4. 优化建议（针对每个问题的改进方案）
5. 优先级排序（按 严重程度 × 用户影响面）
```

**跟进机制：**
- 测试报告 48 小时内同步给设计/开发
- 每个问题分配 owner 和 deadline
- 修复后安排一轮验证测试（3-5 人即可）

## 实战案例：电商 App 结账流程优化

### 背景

某电商 App 发现购物车到支付环节的转化率只有 45%（行业平均 60%+），需要找出原因。

### 测试设计

- **用户：** 8 人（4 个新用户 + 4 个老用户），覆盖 iOS 和 Android
- **任务：** "你选好了一件 199 元的衣服，请完成下单"
- **方法：** 远程 moderated + 屏幕共享 + think-aloud

### 关键发现

| 问题 | 严重程度 | 发现频率 | 根因 |
|------|----------|----------|------|
| 优惠券输入框不明显 | P1 | 6/8 用户忽略 | 视觉层级弱，被折叠在底部 |
| 运费说明在最后一步才出现 | P1 | 5/8 用户感到不信任 | 费用透明度不够 |
| 支付方式默认选中"银行卡" | P2 | 4/8 用户想用微信但没注意默认 | 默认值设置不合理 |
| 地址确认页无法返回修改 | P0 | 2/8 用户直接退出 | 流程设计死胡同 |

### 优化结果

修复以上问题后，结账转化率从 45% 提升到 62%。

## 面试常见追问

### Q1：用户测试和 A/B 测试有什么区别？

| 维度 | 可用性测试 | A/B 测试 |
|------|-----------|----------|
| **回答的问题** | "为什么用户操作困难？" | "哪个方案数据更好？" |
| **样本量** | 5-8 人 | 数万到数十万 |
| **数据类型** | 定性（观察、访谈） | 定量（转化率、点击率） |
| **适用阶段** | 设计/原型阶段 | 上线后验证 |
| **速度** | 1-2 周 | 2-4 周 |

**最佳实践：先可用性测试（发现 Why），再 A/B 测试（验证 What）。**

### Q2：如果用户在测试中什么问题都没遇到怎么办？

这本身就是发现：
1. 可能你的测试任务太简单 → 重新设计更有挑战的任务
2. 可能你的产品确实可用 → 那就测更深层的任务（如高级功能）
3. 可能用户太熟悉产品了 → 招募新用户
4. 可能用户在"表演顺利" → 强调"这不是考试，出问题不是你的错"

### Q3：预算只够测 3 个人怎么办？

3 个人也能发现约 67% 的问题。关键是：
- 选对用户（目标用户，非内部人员）
- 任务设计精准（聚焦核心流程）
- 主持人有经验（能深挖操作背后的心理）

**3 个用户的测试，远好于 0 个用户的测试。**

### Q4：如何在敏捷开发中融入可用性测试？

| Sprint 节奏 | 测试活动 |
|-------------|----------|
| Sprint 规划（Day 1） | 确定本 Sprint 要测试的功能 |
| Sprint 中期（Day 3-4） | 原型/可交互稿就绪时安排 3-5 人快速测试 |
| Sprint 评审（Day 9-10） | 分享测试发现，纳入下 Sprint 优化 |
| 每月 | 安排一次较大规模（8-10 人）的系统测试 |

## 标准回答框架

面试回答时，建议用以下结构：

```
1. 定义（30秒）：什么是可用性测试，解决什么问题
2. 方法选择（30秒）：moderated vs unmoderated，实验室 vs 远程
3. 流程（90秒）：目标 → 招募 → 任务设计 → 执行 → 分析 → 报告
4. 关键原则（30秒）：5人法则、不引导、think-aloud
5. 案例（60秒）：用实际项目说明你怎么做的、发现了什么、怎么改进的
```

## 常见误区

| 误区 | 纠正 |
|------|------|
| "问用户喜欢哪个方案" | 可用性测试是**观察行为**，不是收集偏好 |
| "样本越多越准确" | 5-8 人就能发现 85% 的问题，边际收益递减 |
| "测试是为了验证设计" | 测试是为了**发现问题**，不是确认已有判断 |
| "用户说好就是好" | 用户说的 ≠ 用户做的。看行为，不看表态 |
| "内部同事可以代替用户" | 同事对产品太熟悉，会错过真实用户的困惑点 |

## 关键收获

1. **可用性测试的核心是"观察行为"，不是"收集意见"** — 用户做的比说的更真实
2. **5 人法则**是成本和收益的最优平衡点 — 先测 5 人，修问题，再测 5 人
3. **任务设计决定测试质量** — 具体目标、不引导、贴近真实场景
4. **主持人最大的敌人是自己** — 保持沉默，忍住不帮，忍住不解释
5. **测试不是一次性活动** — 嵌入迭代节奏中，持续发现问题
