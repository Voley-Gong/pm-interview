# 如何用卡片分类法优化产品信息架构？

## 面试场景

> 面试官：我们的一款内容类App用户反馈"找不到想要的内容"，导航结构被产品团队反复修改但效果不好。你会如何系统性地重新设计信息架构？请描述你的研究方法和决策过程。

这道题考察的是候选人是否掌握**结构化的用户研究方法**来解决信息架构（Information Architecture, IA）问题，而不是凭直觉或个人偏好来设计导航。

---

## 核心概念解析

### 什么是卡片分类法？

卡片分类法（Card Sorting）是一种用户研究方法：让参与者将一组"卡片"（每张卡片代表一个内容项或功能）按照他们认为合理的方式进行分组，并为每组命名。

**核心目的**：发现用户的**心智模型**（Mental Model）——即用户如何理解和组织信息，而不是产品团队怎么认为。

### 三种卡片分类形式

| 类型 | 做法 | 适用场景 |
|------|------|----------|
| **开放式（Open）** | 用户自由分组并命名 | 早期探索，从零构建IA |
| **封闭式（Closed）** | 用户将卡片放入预设类别 | 验证已有分类的合理性 |
| **混合式（Hybrid）** | 预设类别 + 可自建 | 既有框架又留探索空间 |

### 为什么不是"开会讨论"？

产品团队开会讨论导航结构是最常见的做法，但存在致命问题：
- **知识的诅咒**：团队比用户更了解产品，无法回到用户视角
- **强势嗓门效应**：谁的声音大谁的决定大，而非数据驱动
- **妥协产物**：各方博弈产生的分类往往逻辑混乱

卡片分类法通过结构化用户输入，将"主观争论"转化为"数据驱动的共识"。

## 完整方法论框架

### 第一步：确定研究范围与卡片集

**选卡原则**：
- 数量控制在 **30-50张**（太少无法聚类，太多用户疲劳）
- 覆盖产品的**核心内容和功能**，不能有遗漏
- 每张卡片描述要清晰，避免歧义（如"个人中心"太模糊，应为"修改头像""查看订单"等具体项）

**实操技巧**：先做一次内容审计（Content Audit），列出所有内容和功能清单，然后抽样选取代表性卡片。

### 第二步：选择分类形式

| 情况 | 推荐形式 |
|------|----------|
| 新产品，从零设计 | 开放式 |
| 现有产品改版 | 混合式（保留核心结构，允许新增） |
| 验证特定方案 | 封闭式 |

### 第三步：招募参与者

- **人数**：开放式建议 **15-20人**，封闭式 **10-15人**（Nielsen Norman Group 研究）
- **画像匹配**：参与者应代表真实用户群体，而非随机样本
- **避免内部员工**：员工的分类会被专业知识污染

### 第四步：执行研究

**线上工具**：OptimalSort、Maze、Useberry
**线下执行**：实体卡片 + 拍照记录（适合小团队快速验证）

**标准化指令示例**：
> "这里有40张卡片，每张代表App中的一个内容或功能。请你按照你觉得最合理的方式将它们分组，并为每组取一个名字。没有正确答案，按照你的直觉来。"

### 第五步：数据分析

#### 1. 亲和图法（定性分析）

将所有参与者的分组结果铺开，人工寻找**共性聚类**：
- 哪些卡片经常被放在一起？（高共识）
- 哪些卡片总是"漂移"？（高争议 → 可能是边界内容）

#### 2. 相似度矩阵（定量分析）

对每对卡片计算"被分在同一组的频率"，生成 N×N 的相似度矩阵。

#### 3. 层次聚类分析（Dendrogram）

基于相似度矩阵做层次聚类，生成**树状图**，直观展示哪些卡片应该归为一类。

**工具**：R 的 `hclust`、Python 的 `scipy.cluster.hierarchy`、或在线工具自动生成。

#### 4. 最一致分类 vs 最优分类

- **实际一致率**：用户分类的共识程度
- **理论最优**：基于算法生成的"最合理"分类
- 两者差距越大，说明当前IA越需要优化

## 实战案例：电商App导航重构

### 背景
某垂直电商App，原有导航：首页 → 分类（男装/女装/童装/家居/美妆）。用户反馈找货难，跳出率高。

### 研究过程
1. **内容审计**：梳理出 48 个内容/功能卡片（如"夏季T恤""美妆教程""订单退款"等）
2. **开放式卡片分类**：18 位目标用户参与
3. **关键发现**：
   - 用户不按"性别"分类，而是按"**使用场景**"（通勤 / 约会 / 居家 / 运动）
   - "美妆教程"和"美妆商品"被大量用户放在一起 → 想要"种草+购买"一体
   - "订单退款"被反复放入"我的"而非分散在各分类
4. **方案重构**：
   - 一级导航从"品类"改为"场景+品类"混合
   - 内容与商品融合展示
   - 售后功能全部归入个人中心
5. **验证**：用封闭式卡片分类验证新方案，一致率从 43% 提升到 78%

## 面试常见追问

### 追问1：如果用户之间的分类差异很大怎么办？

**标准回答**：差异大本身就是一个重要发现。需要进一步分析：
- 是否因为**用户群体本身存在多种心智模型**？如果是，可能需要支持多种导航路径（如支持"按品类"和"按场景"双入口）
- 是否因为**卡片描述不够清晰**导致理解分歧？复盘卡片选取
- 是否存在**极端值用户**？检查是否有参与者的理解完全偏离（可能是筛选问题）

**关键认知**：不存在"唯一正确"的IA，目标是找到**多数用户认可 + 少数用户也能找到**的最佳平衡。

### 追问2：卡片分类法的局限性是什么？

| 局限 | 说明 | 对策 |
|------|------|------|
| **不反映实际使用场景** | 用户在桌面上分类 ≠ 在手机上操作 | 结合任务测试 |
| **忽略了搜索行为** | 很多用户直接搜索，不浏览导航 | 配合搜索日志分析 |
| **静态快照** | 用户认知会随使用深入而变化 | 分阶段做多次 |
| **只解决结构问题** | 不回答"标签叫什么好" | 结合A/B测试标签命名 |

### 追问3：如果资源有限只做5个用户，够吗？

5个用户能发现**主要的分类模式**（通常覆盖70-80%的共识），但会遗漏边缘情况。建议：
- 先做5个快速验证 → 形成假设
- 再做10-15个 → 统计验证
- 如果实在没资源，5个 + 内部专家审查也比纯拍脑袋强

## 标准回答框架

```
1. 问题定义（15%）
   → 明确是IA问题还是搜索/推荐问题
   → 确认现有导航的痛点（数据 + 用户反馈）

2. 方法选择（15%）
   → 为什么选卡片分类法（vs 树测试、首次点击测试）
   → 选择开放/封闭/混合

3. 执行计划（25%）
   → 卡片选取逻辑（30-50张，覆盖核心内容）
   → 参与者招募（匹配用户画像）
   → 线上/线下选择

4. 分析方法（25%）
   → 定性：亲和图、共性聚类
   → 定量：相似度矩阵、层次聚类
   → 差异分析：高争议卡片单独处理

5. 验证与落地（20%）
   → 封闭式验证新方案
   → 配合树测试（Tree Testing）验证可发现性
   → 上线后持续监控导航点击数据
```

## 关键收获

1. **信息架构是骨架，不是表面装饰** —— 导航混乱的根因通常是分类逻辑与用户心智模型不匹配
2. **卡片分类法是工具，不是万能药** —— 需要配合树测试、搜索分析、任务测试才能形成完整闭环
3. **用户的心智模型 > 产品团队的专业判断** —— 当用户"找不到"时，问题不在用户，在结构
4. **定量分析 > 拍脑袋** —— 相似度矩阵和聚类分析能将主观争论转化为客观数据
5. **对于技术转PM**：信息架构设计和技术架构设计有共通之处——都在追求"高内聚低耦合"的组织方式，但判断标准从"技术可维护"变成了"用户可理解"
