# 如何绘制和运用用户旅程图？

## 面试场景

> 面试官："假设你负责一款在线问诊 App 的产品优化。用户反馈'体验不好'但说不出具体哪里有问题。你会如何用用户旅程图的方法来系统性地诊断问题？请描述你的完整操作流程。"

这类题目考察的是：**你是否能从用户视角（而非功能视角）审视产品体验的全貌，并从中发现隐藏的问题和机会**。

---

## 一、什么是用户旅程图？

### 定义

用户旅程图（User Journey Map）是一种**可视化工具**，它按照时间顺序描绘用户为达成某个目标而经历的**完整体验过程**，包括：

- 用户的**行为**（做了什么）
- 用户的**情绪**（感受如何）
- 用户的**思考**（在想什么）
- 与产品的**触点**（交互在哪里发生）
- 体验中的**痛点**和**机会点**

### 它不是什么

| 常见误解 | 实际含义 |
|----------|----------|
| 它是流程图 | ❌ 流程图是系统视角，旅程图是**用户视角** |
| 它是用户画像 | ❌ 画像描述"用户是谁"，旅程图描述"用户经历了什么" |
| 它是线框图 | ❌ 线框图展示界面结构，旅程图展示**情绪起伏** |
| 它只需要画一次 | ❌ 旅程图需要随产品迭代持续更新 |

### 为什么 PM 需要旅程图？

1. **建立团队共情**：让工程师、设计、运营都能"看见"用户在经历什么
2. **发现断点**：功能层面没问题，但串联起来的体验可能断裂
3. **识别优先级**：情绪最低谷的环节通常就是最该优化的地方
4. **对齐内部视角**：不同部门对"用户体验"的理解可能完全不同

## 二、绘制用户旅程图的六步法

### Step 1：明确旅程的边界

**关键决策：**
- **用户角色（Persona）**：是谁的旅程？（如：首次使用的老年患者）
- **目标场景**：用户要完成什么任务？（如：完成一次在线问诊）
- **起点和终点**：从哪里开始，到哪里结束？

> ⚠️ 常见错误：边界太宽（"用户的一天"）或太窄（"点击预约按钮"）。好的边界是一个**完整的任务闭环**。

**示例边界定义：**
- 角色：35岁的职场妈妈，孩子突发皮疹
- 目标：通过 App 找到皮肤科医生，完成在线问诊，拿到处方
- 起点：打开 App 搜索医生
- 终点：收到处方并在附近药店买到药

### Step 2：梳理用户行为阶段

将旅程拆分为 5-8 个**阶段**，每个阶段包含用户的具体行为。

**在线问诊 App 的阶段拆分：**

| 阶段 | 用户行为 |
|------|----------|
| 1. 发现 | 打开 App，搜索皮肤科 |
| 2. 选择 | 浏览医生列表，对比评价和价格 |
| 3. 预约 | 选择时段，填写病情描述 |
| 4. 等待 | 等待接诊通知 |
| 5. 问诊 | 与医生视频沟通 |
| 6. 结束 | 查看处方，了解用药说明 |
| 7. 后续 | 评价医生，去药店买药 |

### Step 3：描绘情绪曲线

对每个阶段，评估用户的**情绪状态**（从 -2 到 +2）：

```
情绪 +2 │         ●●                    ●●
     +1 │       ●●  ●●                ●●  ●●
      0 │─────●●      ●●──────────●●●
     -1 │                ●●    ●●●
     -2 │                  ●●●
        └──────────────────────────────────
        发现  选择  预约  等待  问诊  结束  后续
```

**关键洞察：**
- **情绪低谷** = 痛点集中的地方
- **情绪高峰** = 体验亮点，需要保持
- **情绪跳变** = 体验不一致的地方

### Step 4：标注触点与渠道

在每个阶段标注用户与产品的**交互触点**：

| 阶段 | 触点 | 渠道 |
|------|------|------|
| 发现 | App首页搜索框 | App |
| 选择 | 医生卡片、评价系统 | App |
| 预约 | 预约表单、日历组件 | App |
| 等待 | 推送通知、排队进度 | 推送+App |
| 问诊 | 视频通话界面 | App |
| 结束 | 处方页面、用药提醒 | App+短信 |
| 后续 | 评价入口、附近药店地图 | App+小程序 |

### Step 5：挖掘痛点和机会点

**在每个阶段问三个问题：**
1. 用户在这里可能**卡住**吗？
2. 用户在这里可能**流失**吗？
3. 这个阶段有没有**超出预期**的可能？

**示例分析：**

| 阶段 | 痛点 | 机会点 |
|------|------|--------|
| 选择 | 医生太多，无法快速决策 | 增加"智能推荐"和快速筛选 |
| 预约 | 病情描述不知怎么写，表单太长 | 提供模板+拍照识别症状 |
| 等待 | 不知道还要等多久，焦虑 | 实时排队进度+预计时间 |
| 问诊 | 视频卡顿，沟通不流畅 | 弱网优化+图文备用方案 |
| 结束 | 处方术语看不懂 | 通俗化解释+用药提醒设置 |

### Step 6：输出行动清单

将发现转化为**具体的 Product Action**：

```
优先级矩阵（情绪低谷 × 影响用户数）：

  高影响 │  ● 预约表单优化     ● 等待进度透明化
        │  ● 处方通俗化
        │
  低影响 │     ● 医生推荐算法    ● 评价引导设计
        │
        └────────────────────────────────
          高痛点                低痛点
```

## 三、高阶技巧：场景化旅程图

### 不同的 Persona = 不同的旅程

同一个产品的不同用户群体，旅程图可能完全不同：

| 维度 | 新用户旅程 | 老用户旅程 |
|------|-----------|-----------|
| 心态 | 犹豫、不确定 | 熟练、追求效率 |
| 关键触点 | 引导教程、首单体验 | 快速入口、个性化推荐 |
| 核心痛点 | 信任建立、操作学习 | 效率瓶颈、审美疲劳 |
| 优化方向 | 降低门槛 | 提升速度 |

### 全链路旅程 vs 微旅程

- **全链路旅程**：覆盖从认知到忠诚的完整生命周期（宏观）
- **微旅程**：聚焦某一个关键环节的深度体验（微观）

> 面试加分项：当面试官问到具体体验优化时，先画微旅程定位问题，再回到全链路看上下文。

## 四、面试常见追问

### Q1："用户旅程图和服务蓝图有什么区别？"

| 维度 | 用户旅程图 | 服务蓝图 |
|------|-----------|----------|
| 视角 | 纯用户视角 | 用户+后台视角 |
| 包含 | 行为、情绪、痛点 | 前台、后台、支持流程 |
| 用途 | 发现体验问题 | 设计服务流程 |
| 适用 | 产品调研阶段 | 产品设计/优化阶段 |

### Q2："你画完旅程图之后，怎么确保发现的问题能落地？"

**回答框架：**
1. 将痛点按"情绪低谷 × 影响面"排序
2. 选出 Top 3 痛点转化为具体需求
3. 将需求纳入版本规划，标注优先级
4. 上线后回到旅程图，重新评估该阶段的情绪变化
5. 形成闭环：旅程图 → 痛点 → 需求 → 迭代 → 验证

### Q3："如果没有时间和资源做完整调研，你怎么画旅程图？"

**务实做法：**
- 用团队内部访谈（客服、销售、运营）快速收集用户反馈
- 基于客服工单、App Store 评论提取关键触点
- 先画出"假设版"旅程图，再逐步用数据验证
- 旅程图的价值在于**对齐团队认知**，而非绝对精确

---

## 五、标准回答框架

```
1. 定义边界 → 谁的旅程？目标是什么？起点终点在哪？
2. 拆解阶段 → 5-8 个关键阶段，覆盖完整任务闭环
3. 标注情绪 → 每个阶段的用户情绪高低
4. 识别触点 → 每个阶段用户在哪里与产品交互
5. 挖掘痛点 → 三个关键问题（卡住？流失？超出预期？）
6. 优先排序 → 痛点 × 影响面矩阵
7. 输出行动 → 转化为具体的需求和版本规划
```

---

## 六、关键收获

1. **用户旅程图是共情工具**——它的核心价值不是"画得好看"，而是让团队真正看见用户
2. **情绪曲线是灵魂**——没有情绪标注的旅程图只是流程图
3. **边界定义决定质量**——太宽则空泛，太窄则失去上下文
4. **旅程图要形成闭环**——发现问题 → 转化需求 → 迭代验证 → 更新旅程图
5. **技术转 PM 的优势**：你能更精准地判断哪些痛点可以用技术手段解决（如弱网优化），哪些需要产品方案

> 记住：**用户不会按照你设计的流程走，他们会按照自己的需要走。** 旅程图的意义就是帮你跳出"产品经理视角"，真正站在用户那一边看世界。
