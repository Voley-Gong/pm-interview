# 焦点小组设计与主持人指南

## 面试场景

> 面试官：你负责一款社区类产品，想了解目标用户对"匿名社交"功能的态度。有人建议发问卷，有人建议做一对一访谈。你的leader问：要不要做一场焦点小组？你怎么设计？焦点小组适合什么场景，不适合什么场景？

这道题考察PM对用户研究方法的深度理解——焦点小组是最容易被误解和滥用的研究方法之一。

---

## 一、焦点小组的本质

### 1.1 什么是焦点小组

焦点小组（Focus Group）是一种**结构化的群体定性访谈**，通常由6-8名参与者加一名专业主持人组成，在90-120分钟内围绕特定主题展开深入讨论。

**关键特征：**

| 特征 | 说明 |
|------|------|
| 人数 | 6-8人（少于5人动力不足，多于10人有人会被边缘化） |
| 时长 | 90-120分钟 |
| 形式 | 半结构化讨论（有提纲但不照读） |
| 产出 | 定性洞察、假设生成、观点光谱 |
| 核心价值 | 群体互动激发的"化学反应"——一个人说的引发另一个人的想法 |

### 1.2 焦点小组 vs 一对一访谈

| 维度 | 焦点小组 | 一对一访谈 |
|------|---------|-----------|
| 深度 | 浅但宽 | 深但窄 |
| 群体动力 | ✅ 核心价值 | ❌ 不存在 |
| 敏感话题 | ❌ 社会期望偏差大 | ✅ 更适合 |
| 产出效率 | 高（一次8人） | 低（一次1人） |
| 主持人要求 | 高（需控场+引导） | 中（主要是倾听） |
| 适用阶段 | 探索期（发散） | 验证期（深入） |

**核心判断：如果你需要的洞察来自"人与人之间的碰撞"，选焦点小组。如果来自"个人深层的反思"，选一对一。**

## 二、什么时候该用/不该用焦点小组

### 2.1 适合的场景

1. **探索性研究**：对某个领域还不太了解，想快速收集多元观点
2. **概念验证**：展示2-3个产品概念方向，看用户群体的偏好光谱
3. **需求假设生成**：通过群体讨论生成大量需求假设，后续用定量验证
4. **品牌感知**：了解用户群体对品牌的集体认知
5. **共创工作坊**：让用户一起参与产品方案的共创

### 2.2 不适合的场景

1. **敏感话题**：金融收入、健康问题、职场抱怨——群体环境下无人说真话
2. **可用性测试**：交互体验需要个人独立操作，群体会互相干扰
3. **定量结论**：8个人的观点不能代表整体用户的比例分布
4. **争议性决策**：群体极化效应可能导致极端结论
5. **强势用户在场**：如果参与者地位不对等（如经理和员工），弱势方会沉默

## 三、焦点小组的完整设计流程

### 3.1 第一阶段：研究设计

#### 明确研究目标

```
❌ 差的目标："了解用户对我们产品的看法"
✅ 好的目标："了解Z世代用户在使用匿名社交功能时，
   对隐私安全、社交压力、内容质量三个维度的态度和顾虑"
```

#### 参与者筛选

- **同质性原则**：每组的参与者应在关键维度上尽量同质（同一年龄段、同一用户类型），避免"专家"压制"新手"
- **陌生化原则**：参与者之间最好互不认识，避免熟人社会效应
- **排除标准**：排除极端用户（超级粉丝/超级黑粉）、行业从业者（市场营销/产品经理等）

#### 分组策略

| 策略 | 适用场景 | 示例 |
|------|---------|------|
| 按用户类型分 | 想对比不同群体差异 | 重度用户一组，轻度用户一组 |
| 按人口统计分 | 想了解人口学差异 | 大学生一组，职场人一组 |
| 混合分组 | 想获得多元视角 | 不同类型用户混合（需注意控场） |

**经验法则：至少做2-3组，单组结论不可靠。**

### 3.2 第二阶段：讨论指南设计

一份好的讨论指南是"漏斗型"的：

```
        ┌──────────────────┐
        │   开场暖身（10min）  │  建立信任，设定规则
        ├──────────────────┤
        │  话题引入（15min）   │  从宽泛话题切入，激活思维
        ├──────────────────┤
        │  深度探讨（45min）   │  核心问题逐一展开
        ├──────────────────┤
        │  概念展示（20min）   │  展示原型/概念，收集团队反应
        ├──────────────────┤
        │  总结收尾（10min）   │  回顾关键观点，补充遗漏
        └──────────────────┘
```

#### 讨论指南示例（匿名社交功能焦点小组）

```
【开场暖身】10分钟
- 欢迎与感谢
- 自我介绍（名字+平时用什么社交App）
- 规则说明：没有对错答案、尊重每个人发言、可以不同意

【话题引入】15分钟
- "你们平时在社交平台上，有没有想过用匿名身份说话？"
- "什么情况下你特别希望自己能匿名？"
- （不引导，让用户自然表达）

【深度探讨】45分钟
模块A：隐私安全维度
- "你觉得匿名社交最可能出什么问题？"
- "如果有人通过匿名功能骚扰别人，你觉得平台应该怎么办？"
- "你能接受的匿名程度是什么？（完全匿名？昵称？头像？）"

模块B：社交压力维度
- "你在实名社交平台上，有什么内容是你想说但不敢说的？"
- "匿名之后，你会更愿意分享什么？"
- "你觉得匿名会拉近还是疏远你和朋友的距离？"

模块C：内容质量维度
- "你觉得匿名社区的内容质量会怎样？"
- "你会信任匿名用户发布的信息吗？"
- "如果匿名社区充斥低质内容，你会继续使用吗？"

【概念展示】20分钟
- 展示3种匿名社交的概念方案
- 让每位参与者用一句话描述第一感受
- 群体讨论各方案的优缺点

【总结收尾】10分钟
- "还有什么我没问到但你们想说的？"
- 每人用一句话总结："如果这个功能上线，我会/不会使用，因为..."
```

### 3.3 第三阶段：现场执行

#### 主持人核心技巧

| 技巧 | 说明 | 示例 |
|------|------|------|
| 开放式提问 | 不给选项，让用户自由表达 | "你对这个功能有什么感受？" 而非 "你喜欢这个功能吗？" |
| 沉默等待 | 提问后留3-5秒空白 | 有人会打破沉默说出深层想法 |
| 追问深挖 | 用"为什么"和"能具体说说吗" | 从表面观点挖到根本动机 |
| 圆桌均衡 | 主动邀请沉默者发言 | "小王还没说话，你怎么看？" |
| 控制话霸 | 礼貌但坚定地打断过度发言者 | "这个观点很有意思，我想听听其他人的看法" |
| 群体激荡 | 利用一个观点引发讨论 | "李姐刚才说了一个很有趣的观点，大家怎么看？" |
| 中立立场 | 主持人不表达个人观点 | 不说"这个想法很好"，而说"这个观点很独特" |

#### 常见现场问题及应对

**问题1：冷场**
- 原因：参与者紧张或话题太抽象
- 应对：用具体场景替代抽象问题。"想象你正在用这个功能，第一步你会做什么？"

**问题2：一人独占**
- 原因：有强势参与者或领域专家
- 应对：物理空间调整（让话霸坐在主持人旁边），发言规则约束（"我们每人先说1分钟"）

**问题3：群体极化**
- 原因：群体讨论容易走向极端（更激进或更保守）
- 应对：主动引入反面观点。"有没有人持不同看法？"

**问题4：社会期望偏差**
- 原因：参与者说"正确的话"而非"真实的话"
- 应对：投射提问法。"你觉得你的朋友们会怎么看？"

## 四、焦点小组的分析方法

### 4.1 数据记录

- **录音+录像**（需征得同意）
- **现场笔记**：由记录员记录非语言信息（表情、犹豫、肢体语言）
- **即时摘要**：每组结束后30分钟内，主持人和记录员快速整理关键发现

### 4.2 分析框架

#### 观点光谱法

将参与者的态度放在一个光谱上，而非简单统计"几人支持几人反对"：

```
强烈反对 ←─────────────────────→ 强烈支持
     |          |          |          |
  "绝对不用"  "观望"    "可能会试"  "很期待"
  (2人)     (3人)     (2人)      (1人)
```

**关键是理解每个位置的理由，而非统计人数。**

#### 主题分析法

1. 将所有发言转录
2. 逐句编码（标记主题）
3. 聚合为高层主题
4. 交叉对比不同组别的主题差异

### 4.3 产出物

| 产出物 | 受众 | 格式 |
|--------|------|------|
| 关键发现摘要 | 产品团队 | 1页纸，3-5个核心洞察 |
| 用户语录集 | 设计/市场团队 | 精选10-15条金句 |
| 观点光谱图 | 决策层 | 可视化的态度分布 |
| 需求假设清单 | PM | 转化为待验证的假设列表 |

## 五、面试常见追问

### 追问1："焦点小组和问卷调查，你会先做哪个？"

**标准回答：**
> 通常先做焦点小组（定性探索），再做问卷调查（定量验证）。焦点小组帮我发现"我不知道自己不知道"的维度，问卷帮我量化"这些维度的分布比例"。如果直接发问卷，你可能会遗漏重要维度。

### 追问2："8个人的观点能代表全体用户吗？"

**标准回答：**
> 不能，也不需要。焦点小组的目的不是统计代表性，而是洞察深度。它帮我理解"为什么"和"有哪些维度"，而不是"多少比例"。要了解比例，需要后续的定量研究。这就像你不需要尝完整锅汤才知道咸淡——但你不能把一锅汤的味道推广到所有锅。

### 追问3："如果焦点小组的结论和数据看板矛盾怎么办？"

**标准回答：**
> 这恰恰是最有价值的发现。矛盾意味着某个维度被忽略了。用户说的和他们做的不同——这个"不同"本身就是洞察。比如用户在焦点小组里说"很需要这个功能"，但数据显示他们从不使用。可能这个功能是"说起来重要但用起来不痛"的——这对优先级判断至关重要。

## 六、标准回答框架

```
1. 方法定位（30秒）
   "焦点小组是一种定性群体研究方法，核心价值是群体互动激发的化学反应。
   它适合探索期发散假设，不适合做定量结论。"

2. 适用判断（30秒）
   "对于匿名社交这个话题，焦点小组是合适的，因为：
   - 这是一个探索性话题，我们需要了解用户的多维态度
   - 群体讨论能激发个体想不到的观点
   但需要注意敏感话题的社会期望偏差。"

3. 设计要点（60秒）
   "我会这样设计：
   - 3组，每组6-8人，按用户类型分组
   - 漏斗型讨论指南：暖身→话题引入→深度探讨→概念展示→总结
   - 三个深度模块：隐私安全、社交压力、内容质量"

4. 主持人关键技巧（30秒）
   "核心是：开放式提问、沉默等待、圆桌均衡、中立立场。
   特别注意防止群体极化和话霸控制。"

5. 产出和后续（30秒）
   "产出是定性洞察和需求假设清单，不是统计结论。
   后续会将假设转化为问卷做定量验证。"
```

## 七、关键收获

| 洞察 | 说明 |
|------|------|
| 群体动力学是双刃剑 | 它能激发观点，也能制造偏见 |
| 主持人是成败关键 | 一个好的主持人需要同理心+控场力+中立性 |
| 定性≠定量 | 焦点小组的结论不能直接外推到全体用户 |
| 场景决定方法 | 敏感话题、可用性测试、定量需求都不适合焦点小组 |
| 投射提问是好工具 | "你觉得别人会怎么想"比"你怎么想"更能获得真实答案 |
