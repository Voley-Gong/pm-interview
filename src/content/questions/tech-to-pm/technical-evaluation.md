## 为什么技术评估是 PM 的高阶能力

在 PM 面试中（尤其是技术驱动型公司），面试官会刻意考察你的技术深度：

> "开发说这个需求技术上做不到，你怎么判断？"
> "技术给你了三个方案，你怎么选？"
> "团队想用 3 周重构，你觉得值不值？"

这些问题的核心不是考你写代码——而是考你是否具备**技术判断力**。

**技术判断力 = 理解技术可行性 + 评估技术方案合理性 + 权衡技术投入产出比**

这是技术转 PM 人群的**核心竞争力**，也是你在面试中拉开差距的关键。

## PM 需要掌握的技术评估五层模型

```
Layer 5: 技术趋势判断    （这技术方向值不值得跟）
Layer 4: 架构决策评审    （这个架构设计合理吗）
Layer 3: 方案对比与选型   （三个方案选哪个）
Layer 2: 工作量评估合理性 （3周还是3天）
Layer 1: 可行性判断      （能不能做）
```

### Layer 1：可行性判断（能做 vs 不能做）

**核心问题：** 这个需求在技术上可行吗？

PM 的判断框架：
| 维度 | 检查点 |
|------|--------|
| 平台限制 | 系统 API 是否支持？（如 iOS 后台运行限制） |
| 性能边界 | 响应时间、并发量、数据量级 |
| 第三方依赖 | 需要的服务/API 是否可用、合规 |
| 数据要求 | 需要的数据是否有、质量够不够 |

**面试场景：**
> 开发说"这个做不到"——你不能直接接受，也不能直接反驳，而是要：

1. **问清原因**：是平台限制、性能限制、还是工作量限制？
2. **追问边界**：完全做不到，还是在某些条件下做不到？
3. **探索替代**：如果不能完美做到，有没有 80% 的替代方案？
4. **判断真伪**：有些"做不到"其实是"不想做"或"没做过"

### Layer 2：工作量评估合理性

**核心问题：** 开发给的工作量评估是否合理？

**经验估算参考表（移动端常见功能）：**

| 功能类型 | 初级评估 | 合理范围 |
|----------|----------|----------|
| 简单 UI 页面（展示型） | 1-2 天 | 0.5-3 天 |
| 表单 + 提交逻辑 | 2-3 天 | 1-5 天 |
| 列表 + 下拉刷新 + 分页 | 2-3 天 | 1-4 天 |
| 复杂交互（拖拽/动画） | 5-7 天 | 3-10 天 |
| 接入第三方 SDK | 2-4 天 | 1-7 天 |
| 支付/登录全流程 | 5-10 天 | 3-15 天 |

**判断方法（不写代码也能判断）：**
1. **类比法**：团队之前做过类似功能花了多久？
2. **拆解法**：让开发拆解到子任务，每个子任务 0.5-2 天才算合理
3. **并行法**：如果可以多人并行，总工期应该短于各人天数之和

**注意：以上是经验值，不是标准答案。** 不同团队、不同代码库基础、不同复杂度差异很大。PM 的价值不在于精确估算，而在于**发现明显异常的评估**。

### Layer 3：方案对比与选型

**核心问题：** 技术给了多个方案，怎么选？

**PM 的技术方案评估矩阵：**

| 维度 | 权重 | 方案 A | 方案 B | 方案 C |
|------|------|--------|--------|--------|
| 开发周期 | 高 | 2 周 | 4 周 | 1 周 |
| 用户体验 | 高 | 一般 | 优秀 | 差 |
| 可维护性 | 中 | 低 | 高 | 低 |
| 扩展性 | 中 | 低 | 高 | 低 |
| 风险 | 高 | 低 | 中 | 高 |
| 技术债 | 中 | 增加 | 减少 | 大幅增加 |

**决策原则：**
- **用户价值优先**：如果方案 A 和 B 对用户感知差异不大，选快的
- **关键路径不妥协**：影响核心体验的地方，选体验最好的
- **技术债意识**：短期快的方案如果积累技术债，要看是否影响后续迭代速度
- **风险底线**：如果有 P0 级风险（数据丢失、安全漏洞），一票否决

### Layer 4：架构决策评审

**核心问题：** 这个架构设计合理吗？

PM 不需要做架构设计，但需要理解架构决策对**产品和业务**的影响。

**PM 应该关注的架构决策：**

| 架构决策 | 对产品的影响 |
|----------|-------------|
| 前后端分离 vs SSR | 首屏加载速度、SEO 效果 |
| 微服务 vs 单体 | 功能上线速度、独立部署能力 |
| 原生 vs 跨平台 | 性能、开发效率、用户体验 |
| 自建 vs SaaS | 成本、灵活性、数据安全 |
| 实时 vs 批处理 | 数据时效性、系统复杂度 |

**案例：**
技术团队提议从单体架构迁移到微服务。作为 PM，你需要评估：
- **业务价值**：是否真的需要独立部署/独立扩展？（如果团队 < 20 人，大概率不需要）
- **机会成本**：迁移的 3 个月里，产品迭代会停多久？
- **风险**：迁移过程中功能是否受影响？回滚方案是什么？
- **ROI**：迁移带来的效率提升，多久能收回投入？

### Layer 5：技术趋势判断

**核心问题：** 这个技术趋势我们要不要跟？

**评估框架（Hype Cycle 简化版）：**

| 阶段 | 特征 | PM 策略 |
|------|------|--------|
| 技术萌芽期 | 概念火但落地少 | 关注但不投入，除非有明确场景 |
| 期望膨胀期 | 所有人都在谈 | 小规模实验，不要 all-in |
| 泡沫破裂期 | 质疑声四起 | 如果有真实场景，这是最佳投入时机 |
| 稳步爬升期 | 开始有真实价值 | 认真评估，制定 adoption 计划 |
| 成熟期 | 行业标准 | 不跟就是落后 |

**面试加分点：** 能说出当前技术趋势（如 AI Agent、RAG、边缘计算）处于哪个阶段，以及你的 PM 视角判断。

## PM 与开发沟通的核心原则

### 原则一：说"为什么"，不说"怎么做"

| ❌ 不要说 | ✅ 应该说 |
|----------|----------|
| "用 React Native 做" | "这个页面需要快速跨平台上线，你觉得 RN 合适吗？" |
| "接口返回太慢了" | "用户在加载页等了 5 秒，有什么办法优化到 2 秒以内？" |
| "加个缓存就好了" | "这个数据更新频率不高，有没有方案避免每次实时请求？" |

### 原则二：理解技术债的利息

技术债就像信用卡——有时借钱是合理的（快速上线验证需求），但利息（维护成本）会累积。

PM 需要做的：
- **有意地**承担技术债（不是意外积累的）
- **记录**技术债（让团队知道哪里欠了债）
- **规划还债**（在版本规划中留出技术优化时间，通常 20%）

### 原则三：学会问技术问题

好的 PM 问的问题：
- "这个方案的瓶颈在哪？"
- "如果我们用户量翻 10 倍，这个方案还撑得住吗？"
- "如果这个模块出问题，影响范围多大？"
- "有没有降级方案？"
- "这个改动会影响哪些现有功能？"

## 面试常见追问

### Q1：开发评估了 3 周，你怎么办？

1. **不要直接砍时间**——这是对抗，不是协作
2. **请开发拆解**："能帮我把这 3 周拆到天级吗？我想理解一下每个模块的工作量"
3. **找优化点**：
   - 有没有现成的库/组件可以用？
   - 有没有非核心路径可以砍？
   - 有没有可以并行的部分？
4. **如果确实需要 3 周**：接受评估，但调整版本规划或业务预期

### Q2：你如何判断要不要做技术重构？

**评估框架：**
| 指标 | 判断标准 |
|------|----------|
| 迭代速度 | 新功能开发速度是否明显下降？ |
| Bug 率 | 每次发布的 bug 是否在增加？ |
| 团队士气 | 开发是否经常抱怨代码质量？ |
| 业务影响 | 是否有用户感知的性能问题？ |
| 即将到来的需求 | 未来 3 个月有没有需要大量新功能的大版本？ |

如果 3 个以上"是"，重构的优先级应该提高。

### Q3：你有没有推动过技术优化但业务不重视的场景？

**回答策略：**
1. 用**业务语言**说技术问题：不说"GC 导致卡顿"，说"每次启动崩溃率增加 2%，导致 5000 用户/天流失"
2. 量化**机会成本**：不优化会导致未来迭代慢 30%，相当于每月少做 2 个功能
3. 找到**业务触发点**：在有大型版本需要高质量交付时，推动技术优化更容易获批

## 实战案例

**场景：** 社交 App 的消息推送延迟问题

**背景：** 用户反馈"收到消息但没有推送通知"，客服投诉量上升。

**技术方案对比：**

| 方案 | 周期 | 效果 | 风险 |
|------|------|------|------|
| A：优化现有推送服务 | 1 周 | 延迟从 30s 降到 10s | 低，改动范围小 |
| B：接入第三方推送（如极光/个推） | 2 周 | 延迟降到 3s 以内 | 中，需要接入 SDK + 数据合规评估 |
| C：自建长连接推送 | 6 周+ | 延迟降到 1s 以内 | 高，维护成本大 |

**PM 的决策过程：**
1. 分析用户影响：30s 延迟导致多少用户错过重要消息？客诉转化率？
2. 业务阶段判断：当前是增长期，消息时效性影响核心体验
3. 技术债考量：方案 A 是补丁，方案 B 是可接受的中长期方案
4. **决策：先做 A 止血（1 周），再做 B 彻底解决（2 周），C 暂不考虑**

这就是技术评估能力的实际体现：不是选技术最好的，而是选**当前阶段最合适的**。

## 标准回答框架

面试时被问技术相关评估题，按以下结构回答：

1. **澄清场景**（30 秒）：确认技术约束、业务目标、时间要求
2. **评估方案**（60 秒）：用评估矩阵对比多维度（周期、体验、可维护性、风险）
3. **权衡决策**（60 秒）：用户价值优先 + 技术债意识 + 风险底线
4. **沟通方式**（30 秒）：如何与开发团队协作推进

## 关键收获

- 技术评估不是让 PM 写代码，而是让 PM 有**技术判断力**
- 五层模型从可行性到趋势判断，是技术转 PM 的**护城河**
- 与开发沟通的核心：**说 Why 不说 How，问问题不下指令**
- 技术债管理是高级 PM 的必修课——有意承担、明确记录、规划偿还
- 技术转 PM 的你，在这道题上应该**比纯业务 PM 更有深度**——这是你的主场，充分展示
